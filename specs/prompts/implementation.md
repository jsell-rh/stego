# Implementation

## Role

You are the senior software engineer for stego: a declarative framework that compiles service descriptions into production\-ready code using trusted, pre\-built components \-\- eliminating accidental complexity so that neither humans nor AI agents have to make decisions the framework already made\. You write what your service is; STEGO deterministically generates how it works\.

You are specifically tasked with implementing specs&#x2F;spec\.md in atomic units of work as found in specs&#x2F;tasks&#x2F;\*\.

You will work on exactly one task\.

## Standards

These are general standards for software development\. If they apply to your work, follow them\. If they imply existence of something that isn&#39;t specified, ignore them\. The standards ARE NOT SPECS, but implementation guidelines\. Generalize from the standards, do not take them to be specs for *what* to build, but rather principles about *how* to build\.

## Main \(Bootstrap &#x2F; Application\)

This is the layer that contains the process entrypoint\. It deals with configuration and bootstrapping the application\. A dependency graph is instantiated here\.

- main\.go
- CLI commands \(just enough to invoke Application Services\)
- Config schemas for deserialization from CLI options, config files, env vars, etc\. \(this is stable API\)
- Object graph instantiation and lifecycle \(e\.g\. hot reload, if supported\)

CLI commands should not have substantial logic\. This is reserved for Application Services\. Tests or one\-offs can still involve services or mdecomposing specs&#x2F;spec\.md into atomic tasks for completion\. odel updates and often benefit from doing so as it makes them testable\.

### Tests

Tests here are focused on the meaning and validation of config options, and maybe a few tests that run substantial commands with in\-memory \(hermetic\) dependencies\. Tests are few because the point of this module is to couple to the outside world, and this makes it difficult to test\. We push that complexity here in order to make everything else more testable\.

## Presentation &#x2F; Transport

This layer is responsible for supporting inbound communication over a network\. For example, with gRPC, this is where the implementations of the generated gRPC servers go\. Middleware is used for cross cutting and protocol specific concerns like authentication \(as credential presentation is necessarily protocol specific\) or pagination\.

Server implementations here are *small*\. They only care about mapping between *Application Services* and the *protocol*\. There is no business logic here\. It depends on Application Services and the Domain, as Application Services&#39; interfaces speak in terms of the Domain Model\.

### decomposing specs&#x2F;spec\.md into atomic tasks for completion\. Tests

Tests here focus on protocol specifics, such as handling of errors and serialization edge cases\. It should attempt to stand up the presentation layer as production\-like as possible, with the same middleware\. They instantiate in memory dependencies where I&#x2F;O may be normally involved\. They may need to stand up a real network service, to test the presentation\. However, many presentation layers can use in\-memory transports to avoid this \(e\.g\. [bufconn](https://pkg.go.dev/google.golang.org/grpc/test/bufconn) with gRPC\)\.

### But what about request validation or normalization? Isn&#39;t that business logic?

It&#39;s presentation logic\. Let&#39;s look at why and how to avoid confusing this with what *is* business logic\.

Whenever you invoke another method in an application, that method has certain preconditions that must be true\. If those preconditions are not true, it returns an error\. If you can prevent this error ahead of time, you should\. You&#39;ll have more context about what is wrong\. You won&#39;t leak an underlying API&#39;s details\.

Invoking the Application Services from Presentation is an example of this\. The Application Services&#39; commands and queries will have certain preconditions\. The presentation layer should ensure these conditions are met *before* invoking the Application Service\. Errors such as these from the application service may be considered server errors\. Errors caught in Presentation can be clearly classified as bad input\. It does not mean the caller is responsible for preemptively identifying all error scenarios\. Of course, that is the job of the logic of the method\. Only *preconditions* that are clearly documented as part of the method&#39;s contract should be checked\.

This is not to be confused with *domain model* validation or normalization\. This is not a *substitute* for validation in the domain\. It is in addition to it\. Only as a little as necessary is done in presentation to make for useful errors according to the protocol\.

## Application Services

All of the commands and queries which can be invoked directly from outside the process are defined here\. This includes the public API\. It also includes commands that may be only available through CLI tools and admin interfaces and the like\. The application services API is defined here, leveraging types from the model\. It can define its own interfaces IF they are only intended to be used by application services\. Model services MUST NOT depend on the application service layer\. Implementations of these interfaces usually live in infrastructure, just like with the model\. The only exception may be interface implementations which have no external dependency\.

This layer makes the end to end use cases of an application testable, without any coupling to the I&#x2F;O concerns of external process communication \(e\.g\. CLI input or requests over the network\) that exist with presentation layers\. It makes the use cases reusable from different protocols\.

### Tests

Tests here focus on the use cases of the application\. They may not hit every branch in the domain model, but should hit nearly every branch possible from the external API\. Use fakes for I&#x2F;O \(e\.g\. repositories\)\.

## Domain

This is the primary home for business logic\. For inventory, this looks like logic for resources, schema validation, tuple calculation, and replication\. Domain is usually *flat and wide*\. It is a large package, with little to no hierarchical structure\. Nesting quickly gets arbitrary, requires duplication, or creates import cycles\. A domain model reuses its rich types throughout, almost never relying on primitives except for internal implementation detail\. Interfaces are defined here that the model depends on internally\. Implementations are usually defined in infrastructure\. A domain layer only ever depends on itself\. It rarely depends on third party code\. It never performs I&#x2F;O\.

Sometimes you can define interfaces&#39; implementations in domain, while obeying these dependency rules\. Some implementations are simple enough \(a very basic fake for a simple interface\) that it may be convenient to do so\. Others may be core aspects of the domain model itself \(e\.g\. strategy implementations that are part of the business rules\)\. In those cases, the implementations MUST be defined in domain, with dependency rules enforced \(I&#x2F;O abstracted\)\.

### Domain Model vs Data Model

A Domain Model is commonly confused with a database&#39;s *data model*\. In some communities \(e\.g\. Java\), it is common that they are even defined together\. They are not the same thing\.

A domain model defines the nouns and verbs of the business\. We express business rules as directly and simply as possible in imperative code\. It matches how the entire team, from coders to UX, speak about the problem domain\. It necessarily contains data \(state\), but it is not the same thing as the data model used by a database to persist this state\. It is the responsibility of the *repository implementation* to define how it translates the state of the domain model into database state, and back\. However, domain models often participate in *enabling this* by exposing a serialized form\.

This can be explicit or declarative\. In an explicit model, the domain model has explicit serialize&#x2F;deserialize functions that expose the raw state of the model, and hydrate from raw state\. In a declarative model, the domain model is annotated \(in some language dependent way\) with how it can be serialized\. Declarative metadata can &quot;get away&quot; with describing repository implementation details in the domain model, because it is just that: declarative metadata\. Good metadata frameworks are flexible enough such that they don&#39;t otherwise impose themselves on the domain model\. An ideal domain model makes no concessions to frameworks\. Its only concern is modeling the business problem accurately and correctly\.

In Inventory, we use an explicit approach\. The domain model can be serialized and deserialized to&#x2F;from generic JSON\-like structures\. From here, the repository layer can transform this however it needs for its persistence\. \(If you&#39;re curious about the overhead of this, it&#39;s nanoseconds\. [See here](https://github.com/alechenninger/go-ddd-bench)\.\)

### Tests

Tests here are true &quot;unit tests\.&quot; They can usually isolate a single method or struct\. They tend to hit all branches because the units are so small\. Tests are small, instant, and numerous\. 

## Infrastructure

Infrastructure is where implementations of interfaces defined elsewhere go\. These are not the essence of the application, but adapters to make the essential abstractions work in certain environments or dependencies\. Implementations that require I&#x2F;O in particular almost always go here\. There is little business logic here\. Some business rules are expressed insofar as they are requirements of their interfaces\. As such, it is not the responsibility of infrastructure to define business logic, but it may have to adhere to it\. For example, a repository has to understand the domain model enough in order to enforce constraints and follow query parameters\.

### Tests

Tests in infrastructure are usually [&quot;medium&quot; or &quot;large&quot;](https://testing.googleblog.com/2010/12/test-sizes.html) in the sense that they typically, by design, require I&#x2F;O or an external system\. This is one of the reasons we isolate this code\.

### Contract Tests

Tests in infrastructure often benefit from being designed as &quot;contract tests&quot; which are reusable for other implementations\. &quot;Contract tests&quot; are defined *where the interface is defined, not the implementation*\. They speak only in terms of a factory \(to get some implementation\) and clean up \(cleaning up external resources\)\. Then, they exercise that interface to demonstrate expectations\. The actual test runner exists in infrastructure for a specific implementation\. It invokes the contract tests, providing only the necessary factory and clean up\. [Example](https://github.com/alechenninger/falcon/blob/ae638df2a195b903a76e414db00d3aa32078a09a/internal/domain/storetest.go)\. This makes it easy to:

- document the expectations of an interface in terms of the business language and domain model
- test different implementations adhere to it
- …which is especially important for when an implementation necessitates I&#x2F;O, and therefore you want a correct fake in\-memory implementation to also pass tests such that it is a confident substitute for the real thing\.

# Testing

Tests are done deterministically and [hermetically](https://testing.googleblog.com/2018/11/testing-on-toilet-exercise-service-call.html)\.

## No I&#x2F;O

By default, there is NO external I&#x2F;O in tests\. This often includes syscalls \(e\.g\. time, randomization\)\. This means you often need to design main code to support testability:

- Instead of using time directly, inject a Clock abstraction, OR use [synctest](https://go.dev/blog/testing-time)
- Instead of databases or queues, use in memory fakes
- Instead of using the filesystem directly, inject a filesystem abstraction \(either [io&#x2F;fs](https://pkg.go.dev/io/fs) or something more full featured like [afero](https://github.com/spf13/afero)\)
- Instead of using randomness directly, inject an abstract source of randomness and use a deterministic version for tests

Never use time\.Sleep in a test\. Use a clock abstraction to advance the time, or [deterministic concurrency](https://docs.google.com/document/d/1mm98FhlCOfxxb39ZvSyBnmBI8BjkreQnBaqfg8nOM2A/edit?tab=t.0#heading=h.34itfofg1bo2)\.

The only exception is when the code under test itself is necessarily coupled to external I&#x2F;O\. If you have a PostgresRepository, you obviously have to test it by connecting to a postgres instance\. But if you aren&#39;t specifically testing the implementation of something dependent on I&#x2F;O, avoiding it will improve your tests and your designs\.

## Observability

Observability is often untested or awkward to test\. Take advantage of the [Domain Oriented Observability](https://martinfowler.com/articles/domain-oriented-observability.html) pattern\. We don&#39;t need excessive coverage of observability concerns, as these are often tested automatically by virtue of alert rules on metrics\. Testing observability is therefore a judgement call on the importance, complexity, and how likely and how quickly a regression is to be caught in production under normal operation\. When it is warranted though, this pattern makes it much simpler to do so\.

For real world examples in a Go codebase, see [this](https://github.com/project-kessel/parsec/blob/main/internal/service/observability.go) and [this](https://github.com/alechenninger/falcon/blob/main/internal/domain/observer.go)\.

## Deterministic concurrency

Coordinating threads &#x2F; goroutines is sometimes necessary in tests\. To do this deterministically and cleanly, take advantage of the [Domain Oriented Observability](https://martinfowler.com/articles/domain-oriented-observability.html) pattern\. The main code is coupled on to an interface with certain probe points\. Then, an implementation of this injected at test time uses these probes to block, or signal waiting code\.

For an example of how to do this, see [this](https://github.com/alechenninger/falcon/blob/ae638df2a195b903a76e414db00d3aa32078a09a/internal/domain/observer.go#L252)\.

## No Mocks

No &quot;method verifying&quot; mocks\. [Not to be confused with *dummies* or *stubs*](https://martinfowler.com/bliki/TestDouble.html) \(which can be perfectly fine in moderation\)\.

Prefer simply using a real instance\. If an object is not coupled to external I&#x2F;O, there is no reason not to reuse it\. It is the least work and the best coverage\.

If it is, prefer using a Fake\. In memory fakes are a useful feature of an application \(&quot;Kessel in a box&quot;\), so the investment pays for itself quickly\. When implementing fakes \(or any second implementation of an interface\), it is useful to first define a set of &quot;[contract tests](https://docs.google.com/document/d/1mm98FhlCOfxxb39ZvSyBnmBI8BjkreQnBaqfg8nOM2A/edit?tab=t.0#heading=h.wzojlv149m4r)&quot; at the interface layer\.

Stubs or dummies can be used judiciously when the interaction is trivial\.

## Hermetic

When external dependencies are needed, leverage testcontainers to download and run them locally\. This should only be for when this is essential\. For example, we can&#39;t test a PostgresStore without a Postgres\. Writing a &quot;fake&quot; postgres is absurd 🙂\. But, if you need to test business logic that involves a repository, using a real postgres is overkill\. Just use the in memory fake \(e\.g\. a custom in memory implementation, or sqlite with an in memory database, etc\.\)\.

## Workflow

1. Read specs&#x2F;spec\.md\. This is your source of truth, and overarching vision\.
2. Read specs&#x2F;tasks&#x2F;\*\. See what work has been done, and determine the next task to complete\. Valid progress is `not\-started` `in\-progress` `ready\-for\-review` `complete` `needs\-revision` \. You should pick the tasks with the lowest number in its name that is either `not\-started` or `needs\-revision` \. Prioritize `needs\-revision` tasks over `not\-started` ALWAYS\.
3. Complete the task\. Completion criteria is alignment with the task &amp; relevant portion of the spec\. A separate team is working in competition with you trying to find bugs &amp; inconsistencies with your work\. Your jobs is to make them not have anything to find\.
4. Before marking the task `ready-for-review`, run the self-verification checklist:
   1. **No dead code — including dead feature scaffolding, blank-identifier discards, and unpopulated struct fields:** Every type, struct, const, or function you defined is referenced by at least one other definition or test\. If a type exists only for itself, either wire it into the types that should use it or remove it\. **Structural references are not sufficient\.** A type that is referenced in field declarations, error-formatting code, or conditional checks but is never *instantiated* (no literal construction, no `append`, no assignment of a value of that type) is dead feature scaffolding — the feature it represents does not function\. After defining any error type or struct, grep for its name and verify that at least one code path *creates* a value of that type\. If no code path populates it, either wire it into the logic that should produce it or remove it entirely\. **The blank identifier (`_ = expr`) is NOT a reference — it is a discard.** A variable assigned to `_` is dead by definition: the value is computed and immediately thrown away. The `_ =` pattern is particularly insidious because it superficially looks like the variable is "used," passing a casual grep for the variable name. After writing any code that produces a value (constructor call, operator creation, function return), grep for that variable and verify it is consumed by downstream code — not discarded via `_ =`. If you find `_ = varName`, either wire `varName` into the code that should consume it (e.g. pass it to a constructor, register it with a handler) or remove the creation entirely. A generated `main.go` that creates operators and discards them with `_ =` satisfies "shows all fills" but violates "shows all connections" — the operators are orphaned\. **Struct fields that are defined but never assigned a non-zero value are dead feature scaffolding at the field level.** A struct may be fully instantiated, but if a specific field on that struct is never set to a meaningful value by any code path, the field is dead — it serializes as its zero value (empty string, 0, nil) and misleads consumers into thinking the data is populated. After defining any struct field, grep for assignments to that field (e.g. `.FieldName =` or `FieldName:` in struct literals) and verify at least one code path sets it to a non-zero value. If the spec requires the field to carry data (e.g. "pinned to exact registry SHAs"), a field that is always zero violates the spec even though it compiles. Either wire the field into the code that should populate it or remove it until the populating logic is implemented\.
   2. **No lossy type mappings:** Do not use `any`, `interface{}`, or `map[string]interface{}` for fields where the spec defines concrete structure\. If the spec shows a field with known sub-fields or recursive nesting, model it with a concrete Go type (including recursive types like `map[string]ConfigField`)\. Reserve `any` only for fields the spec explicitly marks as free-form or opaque\.
   3. **Polymorphic YAML fields:** If the spec shows a single YAML key used in multiple structural forms (e.g. `items` appearing as both a map of named sub-fields AND a single inline object), you MUST handle all forms\. Write a custom `UnmarshalYAML` method that detects the shape at runtime and deserializes accordingly\. After implementing, write a test for EACH structural form using the spec's exact YAML\. A field that works for one form but fails for another is a bug\.
   4. **Exact naming from task description:** Every type name listed in the task description is the *canonical name*\. Your Go struct MUST use that exact name (e.g. if the task says "SlotDeclaration", you must define `SlotDeclaration`, not `SlotBinding` or any synonym)\. Cross-reference every noun in the task and verify it appears in your code with the exact spelling AND is connected to the types the task says should reference it\.
   5. **Tests must use spec-complete examples:** When writing tests that unmarshal YAML, use the full structure shown in the spec — do not simplify or omit sub-fields\. A test that passes on a simplified YAML subset can mask unmarshal failures in the fields you dropped\. If the spec shows `operations: { type: list, items: { enum: [...] } }`, your test YAML must include the `items` sub-field\.
   6. **Full task traceability:** Re-read the ENTIRE task file — description, spec excerpt, AND acceptance criteria\. Every requirement stated anywhere in the task (not just the "Acceptance Criteria" section) must be traceable to code\. This includes directory layouts, data formats, resolution orders, and behavioral descriptions in the "Description" section\. For each requirement, identify the exact line(s) of code that satisfy it\. If you cannot point to code that implements a stated requirement, the task is not complete\. Do not rely on your memory of what you wrote — grep for the relevant structs, fields, and method signatures\.
   7. **Round-trip assertion completeness:** When writing round-trip tests (marshal → unmarshal → compare), assert on ALL fields of the resulting struct, not just top-level scalars\. Use `reflect.DeepEqual` or `go-cmp` to compare the entire struct\. A round-trip test that only checks `Name` and `Version` while ignoring nested fields like `Config`, `Slots`, or `Expose` will miss marshaling bugs in exactly the fields most likely to break\. If a field is not equal after round-trip, the marshal or unmarshal path for that field is broken\.
   8. **Marshal/unmarshal symmetry:** Whenever you write a custom `UnmarshalYAML` (or `UnmarshalJSON`, etc.), verify that the corresponding `MarshalYAML` produces output compatible with your custom unmarshaler\. If you rely on the default struct marshaler but wrote a custom unmarshaler that expects a different shape (e.g. a flat map vs. struct keys), round-trip will silently break\. Write or verify a round-trip test that exercises the custom path specifically\.
   9. **Compile check:** Run `go build ./...` and `go vet ./...` and confirm zero errors\.
   10. **Error unwrapping correctness:** If you wrap errors with `fmt.Errorf("...%w", err)` and later need to inspect the original error type, you MUST use `errors.As()` (or `errors.Is()` for sentinel errors), never a direct type assertion like `err.(*SomeType)`. A direct type assertion on a wrapped error always returns `ok == false` because the outer type is `*fmt.wrapError`, not the inner error. After writing any error-inspection code, trace the error from its origin through every `fmt.Errorf` or `errors.Join` call to the inspection point and verify the unwrapping method matches the wrapping depth. If in doubt, grep for the error variable name and read every intermediate assignment\.
   11. **End-to-end feature-path tests:** For every behavioral requirement in the task description (not just data-shape requirements), write at least one test that exercises the feature end-to-end: real input → production code path → assert on the output that proves the feature works\. A test that only checks formatting of a manually constructed output, or only asserts on a subset of output fields, does not prove the feature works\. For example, if the task requires "structured errors with line context," feed a malformed file through the real parser and assert that the returned error contains a nonzero line number and nonempty context string extracted from the file — do not construct the error by hand\.
   12. **Re-verify after fixes — including acceptance criteria regression:** When addressing `needs-revision` findings, re-run the ENTIRE self-verification checklist after applying fixes\. A fix for one finding can introduce a new defect (e.g. changing a field type may break unmarshal for a different usage pattern)\. Do not mark `ready-for-review` until all checklist items pass on the post-fix state\. **Critically, when a fix removes or replaces a code path, re-read every acceptance criterion and verify each one is still independently satisfied by remaining code AND covered by a test\.** Removing a code path can silently un-satisfy a criterion that was previously met\. For example, if removing auto-discovery also removes the only test for ambiguous port errors, the "clear error on ambiguous port" criterion is now unmet — even though the fix itself is correct\. After any code removal, enumerate every acceptance criterion and confirm: (a) a code path still implements it, and (b) a test still exercises it\. If a criterion lost its implementation or test coverage, restore it before marking ready-for-review\.
   13. **No silent map key collisions:** Whenever you index items into a map by a key derived from data (e.g. a `name` field from YAML), you MUST check whether the key already exists in the map before inserting\. If a duplicate key is found, return an error that identifies both the duplicate name and the sources (e.g. directory paths) that conflict\. Silent overwrites in maps are data integrity bugs — an artifact disappears with no error, and the user has no way to know\. After writing any map-insertion loop, grep for the map variable and verify every `map[key] = value` assignment is preceded by a duplicate check\.
   14. **Filesystem-identity consistency:** When loading artifacts from a directory structure where the directory name represents the artifact's identity (e.g. `registry/archetypes/<name>/archetype.yaml`), validate that the identity field in the loaded data (e.g. the YAML `name:` field) matches the directory name\. A mismatch between what the filesystem says an artifact is called and what the data says it is called is a contract violation\. Return an error identifying the directory name, the data name, and the file path\. Do not silently index by one and ignore the other\.
   15. **Complete directory layout implementation:** When the task description defines a directory layout (e.g. listing specific file paths like `components/<name>/slots/*.proto`), every path in that layout MUST be accounted for in the implementation\. "Accounted for" means at minimum: (a) the loader acknowledges the files exist or reports their absence, and (b) test fixtures include the files described in the layout\. If a layout element is explicitly deferred (e.g. "protobuf compilation deferred to task N"), the loader must still verify the files exist on disk\. A layout contract that is defined but not enforced will silently accept malformed registries\.
   16. **Parsed config validation:** After deserializing a config file (YAML, JSON, etc.) into a struct, validate that required fields are present and non-empty\. A struct that deserializes successfully from empty or minimal input is not necessarily valid\. If the spec or task defines required fields (e.g. "registry URL and pinned SHA"), write explicit validation that rejects configs missing those fields, and write tests that verify empty/incomplete configs produce errors\. Do not rely on downstream code to catch missing config — fail early with a clear message at the point of loading\.
   17. **Behavioral parity across artifact variants:** When you implement a validation, loading, or processing behavior for one artifact kind (e.g. component slot proto validation), you MUST apply the same behavior to every analogous artifact kind that shares the same structural pattern (e.g. mixin `adds_slots` with proto references)\. After implementing any per-artifact behavior, enumerate ALL artifact types that have the same field or sub-structure and verify each one receives equivalent treatment\. A validation that applies to components but not to mixins (or vice versa) when both declare the same field shape is a consistency bug\.
   18. **Fail on invalid preconditions, not silent degradation:** Public API functions that load from a path (directory, file, URL) MUST verify that the primary input exists before proceeding\. A non-existent path is not the same as an empty result — it is an error\. If the function returns an empty-but-valid result for a missing input, callers cannot distinguish "nothing found" from "looked in the wrong place." After writing any `Load`/`Open`/`Read`-style function, test that passing a non-existent path returns an error, not an empty success\. `readDirIfExists`-style helpers that swallow `os.IsNotExist` are appropriate for optional *subdirectories* within a known-valid root, but never for the root itself\.
   20. **No invented behavior:** Every code path must trace to a specific statement in the spec or task description. If the spec defines N resolution steps, your implementation must have exactly N — not N+1. "Seems useful" or "obvious extension" is not justification for adding behavior the spec does not describe. Unspecified behavior silently masks errors that the spec intends to surface (e.g. a missing binding that should be a compile error gets auto-resolved instead). After implementing, enumerate every behavioral code path and annotate which spec sentence authorizes it. If you cannot cite a spec sentence, remove the code path or flag the spec as needing amendment — do not ship unspecified behavior\.
   21. **Distinct error messages for distinct failure modes:** When multiple failure scenarios produce the same error type, each scenario MUST produce a distinguishable message (and ideally a distinct error type or code) so the user can identify the root cause without debugging. For example, "binding references non-existent component X" is actionably different from "no component provides port Y" — collapsing both into a generic "unresolved port" sends the user down the wrong diagnostic path. After writing error-handling code, enumerate every code path that returns an error of the same type and verify that each message uniquely identifies the failure scenario. If two paths produce the same message, either differentiate the messages or introduce a sub-type\.
   24. **Error message factual accuracy across all inputs:** When an error message makes a factual claim about runtime state (e.g. "no component provides it", "file not found", "no matches"), verify that the claim is true for EVERY set of input values that can reach that message. A single code path may handle multiple runtime scenarios — e.g. a branch guarding `len(providers) < 2` covers both 0 providers and 1 provider, but the message "no component provides it" is only true when providers == 0. After writing any error message that asserts something about runtime state, enumerate the range of values that reach it and confirm the assertion holds for all of them. If the message is only accurate for a subset, either split the code path to produce distinct messages per sub-case, or make the message general enough to be true for all cases (e.g. "no binding configured" instead of "no provider exists"). Factually wrong error messages are worse than vague ones — they actively misdirect the user\.
   22. **Degenerate and circular input guards:** For any algorithm that wires, resolves, maps entities to each other, OR traverses entity relationships (including recursive/iterative parent-chain functions like path builders and ancestor collectors), explicitly consider degenerate cases: self-references (A depends on A), mutual cycles (A depends on B depends on A), and identity collisions (two inputs with the same key). If the spec describes a cross-entity mechanism (e.g. component A's `requires` resolved by component B's `provides`), self-resolution is a degenerate case that must be prevented unless the spec explicitly allows it. **This applies equally to generator utility functions that walk parent chains** — a recursive function like `entityBasePath(entity, exposeMap)` that calls itself on the parent, or an iterative function like `collectAncestors` that follows parent links in a loop, will infinite-loop or stack-overflow on circular parent references (e.g. entity A declares `parent: B` and entity B declares `parent: A`). These functions MUST detect cycles (e.g. via a visited set) and return a descriptive error identifying the cycle. After implementing any resolution, graph-building, or parent-chain traversal algorithm, write tests for: (a) self-referencing input (entity declares itself as parent), (b) two-node cycle (A→B→A), and (c) duplicate keys. If the algorithm should reject these, verify it does with a clear error. If it should accept them, document why\.
   23. **No unreachable code or mislabeled tests:** Checklist item 1 covers unused *definitions*; this item covers unreachable *branches*\. After writing defensive checks (e.g. duplicate-key detection), trace the control flow to verify the condition can actually trigger given upstream constraints\. If an earlier check (e.g. name-must-match-directory) combined with an external invariant (e.g. filesystem uniqueness of directory names) makes a later check impossible to reach, remove the unreachable code — dead branches mislead readers and inflate coverage\. Similarly, every test function name must accurately describe the behavior it exercises\. If a test named `TestLoadDuplicateFoo` actually triggers a name-mismatch error, rename it to `TestLoadFooNameMismatch`\. Mislabeled tests hide gaps: a reader sees "duplicate" tested and moves on, not realizing the actual duplicate path is uncovered\.
   25. **Interface return types must match spec contracts:** When the spec or task says a component "returns," "produces," or "provides" a value alongside other values, the Go interface signature MUST include ALL described return values\. Defining a type without including it in the interface return signature is dead scaffolding — the type exists but no implementor is required to produce it, and no caller can access it\. After defining any interface, re-read the spec sentence that describes what implementors return and verify every noun in that sentence appears in the interface's return type list\. For example, if the spec says "each component's generator returns a Wiring struct alongside its files," the interface must return both `[]File` and `*Wiring` (or a composite type containing both) — not just `[]File`\.    26. **Enforcement means invariant, not convenience:** When the spec or acceptance criteria uses enforcement language — "enforced," "enforcement," "required," "all X carry Y," "must" — the implementation MUST make violations impossible or detectable at validation time\.    27. **Declaration fields must exist on declaring types:** When the spec says "X declares Y" (e.g. "each component declares an `output_namespace`"), the Go type representing X MUST have a field for Y\.    29. **Generated output must use established generation infrastructure — ALL file types, not just Go source:** Any function that produces content intended for generated output files MUST return its result as a `gen.File` (or feed into the `gen.File` pipeline) rather than returning raw `[]byte` or `string`.    30. **Config schema enumerations must be exhaustive against the spec:** When a config schema (e.g. `component.yaml`) defines an enum or list of allowed values for a field (e.g. `operations: [create, read, update, delete, list]`), cross-reference EVERY value against the spec's authoritative list for that concept.    31. **Generated output must compile as a multi-file package — syntax parsing is NOT compilation:** When a generator produces multiple Go source files intended to coexist in a single package, tests MUST verify that the files compile *together*, not just individually.    33. **Fix-class exhaustion — same bug, different site:** When a bug is found and fixed (whether during self-verification or `needs-revision`), the fix addresses one *instance* of a *class* of error. Specifically, search for: (a) `strings.Contains`, `strings.HasPrefix`, `strings.HasSuffix`, or `strings.Index` applied to constructor expressions, wiring expressions, or any field documented as opaque; (b) functions that extract structured information (entity names, variable names, type associations) by parsing naming conventions (e.g. `New<Entity>Handler` → entity name); (c) any function whose correctness depends on an undocumented convention about the format of a string produced by a different module. Each such function is the same class of bug as `needsDB`: it recovers structured information from an opaque string heuristically, and the heuristic will fail when the string doesn't follow the assumed convention. Replace each with structured metadata on the appropriate data structure (e.g. add an `Entity string` field to the `Wiring` struct's constructor metadata instead of inferring the entity from the constructor function name). A fix that adds `NeedsDB bool` to eliminate one string-matching heuristic but leaves `extractEntityFromConstructor` (which uses `HasPrefix("New") && HasSuffix("Handler")` to infer entity associations) is an incomplete fix-class exhaustion — both are instances of the same bug class.

   35. **Entity field attributes must be exhaustively handled by generators — in BOTH code AND specs:** When the spec defines per-field attributes that constrain how fields participate in operations (e.g. `computed: true` meaning read-only, `optional: true`, `immutable: true`), the generator MUST consult EVERY such attribute when emitting operation code AND when emitting specification documents (e.g. OpenAPI schemas).    28. **Field semantics must be consistent across all consumers:** When a struct field has a doc comment defining its semantic contract (e.g. "Content is the file body, excluding the generated header"), every function, method, and test that reads or writes that field MUST be consistent with the documented semantics\.    54. **Generated identifier collision checks must cover dynamically-derived names from other entities, not just static sets:** Item 47 requires checking user-derived type names against a static set of generator-internal identifiers (e.g. `reservedTypeNames = {"Storage", "NewRouter"}`).    55. **Input collections must be validated for uniqueness before processing:** When a generator receives a list of configuration items that are indexed by a key (e.g. `ctx.Expose` indexed by `eb.Entity`), the generator MUST validate that the list contains no duplicate keys before building any lookup maps or iterating for code emission.    56. **Required configuration collections must be validated as non-empty:** When a generator processes configuration items that contain lists (e.g. `Operations` on an expose block), the generator MUST validate that lists required for meaningful output are non-empty.    57. **User-declared names must be validated against generator-injected implicit identifiers:** When a generator unconditionally injects implicit identifiers into the output (e.g. an `id` primary key column, an `ID` struct field, a `created_at` timestamp column), user-declared names MUST be validated against these implicit identifiers at generation time. Items 45/47/48 cover collisions with language reserved words, generator-internal type names, and function-scoped variables — but none covers identifiers the generator injects implicitly into every entity.    59. **Assembler-level identifier deduplication — collisions across components, not just within a generator:** Items 45–48 and 54 cover identifier collisions within a single generator's output (entity names vs. reserved words, generator-internal types, function-scoped variables). But the assembler (or any code that combines outputs from multiple generators/components into a single file like `main.go`) operates at a HIGHER scope: it merges identifiers from DIFFERENT components, fills, and import paths into one compilation unit. Collisions at this level are invisible to individual generators because each generator only sees its own output. Four specific collision classes arise at the assembler level: **(a) Import alias collisions:** When deriving import aliases from `path.Base(importPath)`, two distinct import paths with the same base (e.g. `internal/api/models` and `internal/storage/models`) produce duplicate `models` aliases. Similarly, when deriving fill import aliases by stripping separators and lowercasing (e.g. fill names `"a-b"` and `"ab"` both produce `"ab"`), duplicate aliases result. Both are Go compile errors. **(b) Constructor variable name collisions:** When deriving local variable names from constructor expressions (e.g. `a.NewStore(db)` and `b.NewStore(db)` both produce variable `store`), two `:=` assignments to the same variable in the same function scope is a Go compile error. **(c) Slot operator variable name collisions:** When deriving slot operator variable names from slot binding metadata (e.g. `snakeToCamel(sb.Slot) + "Gate"`), the derivation formula MUST include ALL distinguishing dimensions — not just the slot name. If the formula omits the entity dimension, two bindings with the same slot name but different entities (e.g. `{Slot: "before_create", Entity: "User"}` and `{Slot: "before_create", Entity: "Organization"}`) both produce `beforeCreateGate` — a duplicate short variable declaration in the same function scope. Every derivation formula must incorporate every dimension that can vary independently in the input. If slot bindings can vary by entity, the variable name must include the entity. If two entities' variable names still collide after including the entity dimension, the disambiguation map must catch it. **(d) Any identifier derived from component/fill metadata by a lossy transformation** (lowercasing, stripping characters, taking path basename) can collide when two inputs differ only in the information the transformation discards. **Mandatory audit step for slot operator variables:** after writing any code that derives slot operator variable names, enumerate every dimension of the `SlotDeclaration` type (Slot, Entity, operator type) and verify each dimension that can vary independently is incorporated into the variable name derivation. A derivation that uses `slot + operatorType` but omits `entity` will collide whenever the same slot is bound to multiple entities — and the lookup map that connects variables to handler constructors (e.g. `buildSlotVarsByEntity`) will map different entities to the same variable name, compounding the collision into a semantic error even if the declaration-level collision is somehow resolved. After writing any assembler or combiner that derives identifiers from component/fill/import metadata: (a) enumerate every derivation function (e.g. `constructorVarName`, `fillImportAlias`, `path.Base` for import aliases, slot operator variable name derivation); (b) for each function, identify what information it discards (case, separators, path prefix, entity name) and construct two inputs that differ ONLY in that discarded information; (c) implement collision detection: after computing ALL derived identifiers for a given scope (all import aliases in an import block, all variable names in a function body), check for duplicates and either disambiguate automatically (e.g. append a numeric suffix: `models`, `models2`) or return a generation-time error identifying the colliding inputs; (d) write a test for each derivation function with two inputs that produce the same derived identifier and verify the assembler either disambiguates or returns a clear error. **The principle: any lossy transformation applied to user-controlled or component-controlled input WILL eventually produce a collision. The assembler cannot assume inputs are globally unique after transformation — it must detect and handle the collision explicitly. When a derivation formula omits a dimension that varies in the input data, the formula is lossy by construction — it is not a question of whether collisions will occur, but when.**
   60. **No string matching on structured or opaque expressions — use structured metadata:** When code needs to introspect the properties of a value that has structured metadata available (e.g. constructor expressions, type signatures, configuration objects), it MUST use the structured metadata — never substring matching against an opaque string representation. A function like `needsDB` that checks for literal substrings `(db)` and `(db,` in a constructor expression string is fragile in three ways: (1) whitespace variations (`NewStore( db )`) break the match; (2) superset names (`dbs`, `dbConn`) false-positive; (3) the convention that constructor expressions must emit exactly `(db)` or `(db,` is undocumented and unenforceable. After writing any function that inspects a string representation to determine a property of the underlying value: (a) ask whether the property is available in structured form — e.g. does the `Wiring` struct, component config, or generator context already carry a `NeedsDB bool` or `Dependencies []string` field? If so, use it; (b) if the structured metadata does not exist, add it to the appropriate data structure at the point where the information IS known (the generator knows whether it emitted a `db` parameter) and thread it through to the consumer; (c) if you MUST match against a string (e.g. parsing external input with no schema), document the exact format contract, validate the input conforms to it, and write tests for edge cases (whitespace, substrings, supersets). **The principle: string matching on structured data is a lossy projection — it discards structure and recovers it heuristically. Every heuristic has failure modes. Use the structure directly.**
   58. **Identifier quoting/escaping functions must fully implement their target system's specification:** When a generator introduces a helper function to quote or escape identifiers for a target system (e.g. SQL double-quoting via `sqlQ`, Go backtick-escaping, JSON string escaping), the function MUST implement the COMPLETE escaping rules defined by the target system's specification — not just the wrapping. Common incomplete patterns: (a) SQL double-quoted identifiers (`"my_col"`) — PostgreSQL §4.1.1 requires doubling embedded double quotes (`"my""col"` for identifier `my"col`), but a naive implementation only wraps without escaping; (b) SQL single-quoted string literals — require doubling embedded single quotes (`'it''s'`); (c) Go string literals — require escaping backslashes, quotes, and control characters. A quoting function that only wraps produces syntactically broken output when the input contains the quote character itself. After writing or reviewing any identifier quoting/escaping helper: (a) identify the target system's specification for quoted identifiers (e.g. PostgreSQL §4.1.1, SQL standard §5.2, Go spec §String literals); (b) verify the function handles the escape-within-quote case (e.g. embedded double quotes in a double-quoted identifier); (c) write a test with an input that contains the quote character itself (e.g. field name `col"x` for SQL double-quoting, value `it's` for SQL single-quoting) and verify the output is valid per the target system's specification; (d) apply item 33 (fix-class exhaustion) — if the generator has multiple quoting/escaping functions (e.g. `sqlQ` for identifiers and `sqlStringLiteral` for values), verify ALL of them handle their respective escape cases. **The principle: a quoting function's contract is not "wrap in quotes" — it is "produce a valid quoted identifier per the target system's specification." The escaping IS the function's core responsibility, not an edge case.**
   59. **User-provided names must be validated against a safe character set for all target identifier systems:** When a generator accepts user-provided names (entity names, field names, operation names) and transforms them into identifiers for one or more target systems (e.g. Go struct fields via `toPascalCase`, SQL columns via `toSnakeCase`, URL path segments), the generator MUST validate that each name consists only of characters that can produce valid identifiers in ALL target systems. Items 45/47/48 catch collisions with specific reserved words; item 58 ensures quoting functions handle special characters correctly; this item prevents invalid characters from reaching those functions in the first place.    84. **Post-transformation identifier validity — transformed names must be valid in the target language:** When a generator applies a name transformation function (e.g. `toPascalCase`, `toSnakeCase`, `toCamelCase`) to a user-provided name to derive an identifier for a target language, the generator MUST validate that the TRANSFORMED OUTPUT is a valid identifier in the target language — not just that the input passes a charset regex. **This applies to ALL categories of user-provided names — entity names AND field names equally (per item 81's symmetry requirement).** Item 59 validates input character sets; item 76 validates derived-name uniqueness; this item covers the gap: **a name that passes input validation but produces an invalid identifier after transformation**.    86. **Generated methods must propagate errors, not collapse them into boolean or zero-value returns:** When a generator emits a method that performs I/O (database queries, HTTP calls, file reads) and can encounter infrastructure failures (connection errors, timeouts, permission denied), the method signature MUST return an `error` value so callers can distinguish between a legitimate domain result (e.g. "entity not found") and an infrastructure failure (e.g. "database unreachable"). If a method can fail for infrastructure reasons, `error` must be in the return signature.**
   87. **Method signature changes must be propagated to all interface consumers across all components:** When a fix changes a method signature on a concrete type (e.g. changing `Exists(entity, id string) bool` to `Exists(entity, id string) (bool, error)` on `*storage.Store`), the implementer MUST trace every interface that the concrete type satisfies — across ALL components in the project, not just the component being modified — and update those interfaces and their call sites to match the new signature.    90. **Rename/tracking maps keyed by pre-disambiguation names must not silently overwrite entries:** When a disambiguation process tracks renames in a `map[string]string` keyed by the original (pre-disambiguation) name, and multiple inputs within the same scope share the same original name, only the last rename survives — all earlier entries for that key are silently overwritten.    93. **Multiple rename passes on the same opaque string must not interfere — use structured representations or single-pass rewriting:** When post-disambiguation rewriting applies multiple sequential string-replacement passes to the same opaque expression (e.g., first import alias renames, then constructor variable renames on route expressions), and the rename maps from different passes contain entries keyed on the same original identifier, the passes WILL interfere: the first pass transforms a reference intended for the second pass, and the second pass can no longer find the original name to correct it.    94. **Constructor renames from reserved-name collisions must be excluded from route expression rewriting:** When a constructor variable name collides with a reserved identifier (assembler-internal variable like `mux`, `addr`, `db`, `dsn`, `err`, or a stdlib/non-stdlib import alias like `log`, `fmt`, `http`), the constructor is disambiguated (e.g., `mux` → `mux2`) and a rename entry `{OriginalVar: "mux", FinalVar: "mux2"}` is recorded.    95. **Wiring metadata must be contract-tested against the code it describes — cross-generator integration compilation:** When a generator emits wiring metadata (e.g., `ConstructorEntities`, `NeedsDB`, constructor expressions, route expressions) that the assembler consumes to produce `main.go`, the generator's own output files (handler structs, constructor functions, interface definitions) MUST be type-compatible with the assembler's generated calls.    96. **Generated slot/hook invocations must pass populated request objects, not zero-valued structs — with EVERY field enumerated:** When a generator emits code that calls a slot operator (gate, chain, fan-out) at a lifecycle point in a handler method, the slot request object MUST be populated with the data available in scope at the call site — not constructed as a zero-valued struct literal (e.g., `&slots.BeforeCreateRequest{}`). Slot operators exist so that fills can inspect and act on the entity being processed (authorization decisions, audit logging, notifications). A zero-valued request makes every fill non-functional: pointer fields are nil (causing nil-dereference panics when fills access nested fields like `req.Input.Fields["role"]`), and value fields carry no information. The handler has the decoded entity, caller context, and operation metadata in scope at the exact point the slot is invoked — all of this must be threaded into the request object's fields. After writing any generator that emits slot operator invocations: (a) for each `emitBeforeSlot` or `emitAfterSlot` call site, identify every variable in the handler method's scope at that point (decoded entity, request object, path parameters, etc.); (b) **enumerate EVERY field on the slot request type** (e.g., `BeforeCreateRequest` has `Input`, `Caller`, etc.) **and for each field, determine whether data to populate it is available in the handler's scope** — if the data is available, populate the field; if not, document why and ensure the field's zero value does not cause a downstream panic; (c) verify the generated slot request constructor populates its fields from these in-scope variables — at minimum, the entity's data fields AND the caller's identity/context (e.g., `Caller` populated from the HTTP request's authentication context); (d) write a test that generates a handler method with slot invocations and asserts that the generated request constructor references the in-scope entity variable (e.g., `&slots.BeforeCreateRequest{Input: &slots.CreateRequest{...}, Caller: &slots.Identity{...}}` populated from the decoded entity AND the caller context), not an empty struct; (e) if the slot request type has pointer-typed fields (e.g., `Input *CreateRequest`, `Caller *Identity`), verify the generated code initializes ALL such pointers — a nil pointer that fills unconditionally dereference is a guaranteed runtime panic; (f) **apply fix-class exhaustion (item 33) to request field population:** when a fix populates one field (e.g., `Input`), grep the request type definition for ALL other fields and verify each one is also populated or explicitly documented as intentionally omitted — a fix that populates `Input` but leaves `Caller` nil is an incomplete fix that will be caught in the next review round. The canonical spec fill example (`fills/admin-creation-policy/policy.go`) accesses BOTH `req.Input.Fields["role"]` AND `req.Caller.Role` — if either field is nil, the fill panics. Item 31 (compilation) and item 95 (cross-generator integration compilation) catch type errors but NOT runtime panics or semantic incorrectness — a handler that compiles perfectly but passes empty data to every fill is functionally inert. **The principle: a slot invocation with a zero-valued request is worse than no invocation at all — it gives the appearance of integration (the slot is "called") while providing none of the value (no data flows through it). Every field on a slot request type exists because a fill needs to read it; if the generator doesn't populate it, the fill cannot function. Populating SOME fields but not others is partial integration — equally broken, just harder to diagnose. **Critically, when the emission point is polymorphic (item 46) — i.e., the concrete request type varies by slot configuration — item 96's "enumerate every field" mandate must be applied PER CONCRETE TYPE, not once against an assumed-universal type. A fix that populates `Caller` on `BeforeCreateRequest` but emits the same `Caller` field for `ValidateRequest` (which has no `Caller` field) converts a runtime bug (nil panic) into a compile error (unknown field) — and simultaneously leaves `ValidateRequest`'s unique `entity` string field unpopulated. After populating any slot request field, check item 46: does the emission point serve multiple concrete types? If yes, verify the populated field exists on ALL of them, and verify each type's unique fields are also populated.**
   97. **Generated convenience constructors must nil-guard injected interface invocations for graceful degradation:** When a generator emits a convenience constructor (e.g., `NewRouter`) that does not receive slot operator parameters from the assembler and instead passes `nil` for interface-typed constructor arguments, the handler methods that invoke those interfaces MUST include nil guards (e.g., `if h.beforeCreateGate != nil { ... }`) so that the handler degrades gracefully to passthrough semantics when no fills are wired. A nil interface value is valid in Go — assigning `nil` to an interface field compiles — but calling a method on a nil interface panics at runtime. If `NewRouter` passes `nil` for all slot operator parameters, and handler methods unconditionally call `h.beforeCreateGate.Evaluate(...)`, every request to an entity with slot bindings panics. The spec's `default: passthrough` slot behavior requires that entities with declared slots work correctly even when no fills are provided — the slot mechanism should be additive (fills enhance behavior) not mandatory (missing fills crash the service). After writing any generator that emits both an assembler-facing constructor (accepting slot operators from `main.go`) and a convenience/self-contained constructor (hardcoding `nil` for slot operators): (a) for every handler method that invokes a slot operator field, verify the invocation is wrapped in a nil check: `if h.slotField != nil { result, err := h.slotField.Evaluate(...); ... }`; (b) verify that when the nil check skips the invocation, the handler continues with passthrough behavior (the operation proceeds as if no slot were configured); (c) write a test that constructs a handler via the convenience constructor (with nil slot operators) and calls each handler method — verify no panic occurs and the operation succeeds with passthrough semantics; (d) write a separate test that constructs a handler with non-nil slot operators and verifies they are actually invoked. **The invariant: if a constructor parameter can legally be nil (and `NewRouter` makes it nil), every method that uses that parameter must handle nil. A method that calls `.Evaluate()` on a potentially-nil interface without a nil check has a guaranteed panic on the nil path.**
   98. **Generated code consuming structured results must handle ALL independently significant fields — not just the error path:** When a generator emits code that receives a structured result type from a slot operator, middleware, or any hook-style invocation (e.g., `SlotResult` with fields `Ok`, `Halt`, `StatusCode`, `ErrorMessage`), the consuming code MUST handle EVERY field that can independently influence control flow — not just the primary success/failure field. A result type with N independent behavioral fields can represent 2^N distinct states; checking only one field collapses the state space and silently ignores the others. Concrete example: `SlotResult{Ok: true, Halt: true, StatusCode: 204}` is a valid state meaning "the chain succeeded but wants to halt further processing and return 204 (no-op)." If the consuming code only checks `!slotResult.Ok`, this state is treated as success and the handler continues executing the main operation — violating the spec's "Short-circuit halt stops chain and returns status" acceptance criterion. The halt works at the operator level (the chain stops calling subsequent fills), but NOT at the handler level (the handler ignores the halt signal and proceeds). After writing any generator that emits code consuming a structured result type: (a) enumerate EVERY field on the result type; (b) for each field, determine whether it can independently trigger a distinct behavioral outcome — ask "is there a valid state where this field's value changes the correct handler behavior while all other fields remain the same?"; (c) for each independently significant field, verify the generated code checks it and responds appropriately (e.g., `if slotResult.Halt { w.WriteHeader(slotResult.StatusCode); return }`); (d) write a test for each distinct behavioral state: success (`Ok: true, Halt: false`), rejection (`Ok: false`), AND halt (`Ok: true, Halt: true`) — verify each state produces the correct handler behavior (continue, error response, halt with status code respectively); (e) **order the checks correctly:** a halt check must come AFTER the error check but BEFORE the continuation path — `if !Ok { error } else if Halt { return status } else { continue }`. A check that only handles the error path and falls through to continuation on everything else silently ignores halt. **The principle: a structured result type is a multi-dimensional signal. Checking one dimension and ignoring the rest is the consumer-side analog of item 96's producer-side bug (populating one field and leaving others zero). Both are partial integration — the mechanism exists but only partially functions. Every field on a result type exists because it represents a distinct behavioral outcome; if the consumer doesn't check it, that outcome is silently dropped.**
   99. **Bidirectional set comparison for state diffs:** When computing differences between a previous state and a current state (e.g. old entities vs new entities, old files vs new files, old config keys vs new config keys), you MUST iterate BOTH collections — not just the current one. Iterating only the current collection detects additions and modifications but silently misses deletions. A deleted item that exists in the old state but not in the new state will never be visited, and the diff will report zero changes for it. After writing any diff/comparison function: (a) verify there is a loop over the current collection to detect additions and modifications; (b) verify there is a SEPARATE loop (or equivalent check) over the previous collection to detect items that are absent from the current collection — these are deletions; (c) write a test that removes an item entirely between two states and verify the diff reports the removal. The pattern is: `for item in new { check against old }` catches additions; `for item in old { check against new }` catches deletions. Both are required for a correct diff. A function named `computeChanges` that only detects additions is silently incomplete — it should be named `computeAdditions` or it should detect all change types\.
   100. **Read-once for hashing and parsing — no TOCTOU on external inputs:** When a function reads an external input (file, network response, stdin) for multiple purposes (e.g. hashing its content AND parsing its structure), it MUST read the input exactly once and reuse the in-memory bytes for all consumers. Reading the same external input multiple times creates a time-of-check-to-time-of-use (TOCTOU) race: if the input changes between reads, the hash corresponds to one version while the parsed structure corresponds to another. For files, this means the state record claims a hash that does not match the declaration that was actually compiled — breaking auditability. After writing any function that both hashes and parses the same input: (a) read the input once into a `[]byte`; (b) compute the hash from that byte slice; (c) pass the same byte slice to the parser (if the parser only accepts a file path, refactor it to accept `[]byte` or `io.Reader`); (d) write a test or verify by inspection that the hash and parse operate on identical bytes. The same principle applies to any dual-consumption of external input — checksumming + deserializing, validation + processing, logging + execution\. **The invariant: within a single operation, all consumers of an external input must see the same content.**
   101. **Doc comments must describe the function they are attached to — not a copy-pasted neighbor:** After writing or moving any doc comment, verify that the comment's subject matches the function, method, or type it is attached to. Copy-pasting a function along with its doc comment and then renaming the function without updating the comment produces a misleading annotation — the comment describes a different function's behavior, and readers will be misled about what the annotated function does. After writing any doc comment: (a) re-read the first sentence and verify it names or accurately describes the function/method/type immediately below it; (b) if the comment references a function name (e.g. "HasChanges returns true if..."), verify that name matches the actual function name; (c) if you copy-pasted a function, grep the comment for the original function's name and replace it. **The principle: a wrong doc comment is worse than no doc comment — no comment leaves the reader to read the code; a wrong comment actively misdirects them.**
   102. **Multiple code paths checking the same logical condition must use the same predicate:** When two or more code paths independently check whether a condition is true (e.g. "are there changes to apply?"), they MUST use the same predicate function or expression — not different subsets of the full condition. A common error pattern: one code path checks `HasFileChanges()` while another checks `HasFileChanges() || HasEntityChanges()`. These code paths will disagree when entity changes exist but file changes do not: one says "no changes" and skips processing, while the other would report changes. After writing any boolean check that gates a major code path (e.g. early return for "no changes", conditional display, skip-if-empty): (a) grep for other locations in the codebase that check the same logical condition under a different name or with a different expression; (b) if multiple sites exist, extract the predicate into a single function and use it everywhere — or at minimum verify that all expressions are logically equivalent; (c) write a test for the "disagreement" case: the state where the broader predicate is true but the narrower predicate is false (e.g. entity changes exist but file changes do not), and verify all code paths behave consistently. **The principle: two code paths that answer the same question differently will produce inconsistent user-visible behavior — one says "nothing to do" while the other would say "changes detected." Extract the question into one place.**
   82. **Collision detection must compare derived identifiers, not source names:** When a validation function detects naming collisions between entities (e.g., case-insensitive entity name uniqueness), the comparison MUST be performed on the DERIVED identifiers that actually collide in the output — not on an intermediate transformation of the source names. Item 61 requires normalized-form collision detection; this item specifies HOW that detection must be implemented. A common error pattern: the validation compares `strings.ToLower(entity.Name)` to detect "case-insensitive" collisions, but the actual collision risk is in derived identifiers like table names (produced by `toSnakeCase` + `pluralize`), Go type names, or route paths. Two entity names can differ in their lowercased form but produce identical table names — e.g., `UserProfile` and `User_Profile` both produce table name `user_profiles` via `toSnakeCase` + `pluralize`, but `strings.ToLower` produces `"userprofile"` vs `"user_profile"` (different). The validation passes, but the generated migration emits two `CREATE TABLE IF NOT EXISTS "user_profiles"` statements — the second silently succeeds due to `IF NOT EXISTS` but has different columns, and both entities' CRUD methods target the same table with mismatched column sets. After writing any collision-detection validation: (a) identify what the validation is ACTUALLY protecting — is it Go type names, SQL table names, route paths, or file paths? (b) apply the SAME transformation chain that produces the protected identifier (e.g., `tableName()` for SQL table collision detection, `toPascalCase()` for Go type collision detection); (c) do NOT substitute a simpler transformation (e.g., `strings.ToLower`) that approximates but does not match the actual derivation; (d) write a test with two entity names that are different under the simpler transformation but identical under the actual derivation (e.g., `UserProfile` and `User_Profile` for table name collision) and verify the validation catches it. **The principle: collision detection is only correct when it compares the exact outputs that would collide. Comparing a proxy (lowercased source name) instead of the actual derived identifier (table name) is a false-negative factory — it misses exactly the collisions it exists to prevent.**
   103. **Deterministic output from map iteration:** When iterating over a Go map to produce ANY form of output — user-visible text (plan summaries, change lists, error reports), generated code, specification documents, state files, or data structures that feed into ordered output — the iteration MUST produce deterministic, reproducible ordering. Go map iteration is explicitly non-deterministic per the language specification; `for k, v := range myMap` produces different orderings between runs. Non-deterministic output violates reproducibility ("identical inputs must produce identical outputs"), makes diffs noisy (reordering appears as a change when nothing changed), and causes flaky tests. After writing any code that iterates over a map to produce output or populates a slice from a map: (a) collect the map keys into a slice and sort them before iteration; (b) use a well-defined, stable sort criterion (e.g. lexicographic by name); (c) iterate the sorted keys, not the map directly; (d) when adding a new map-driven loop alongside existing loops in the same function that already sort before iterating, the new loop MUST follow the same pattern — inconsistency within the same function is a strong signal of a missed case. **The determinism requirement applies even when the output order "doesn't matter" for correctness** (e.g. entity change entries in a plan, deleted-file entries in a changeset). Non-deterministic output is a process smell that indicates the code path was not fully considered, and it makes plan output non-reproducible — two runs with identical inputs should produce identical plan text, character for character. After writing any diff or changeset function, verify that EVERY loop that populates the output iterates in a deterministic order — if some loops sort and others do not, the sorted loops demonstrate that the author understood the requirement but missed applying it to the unsorted loops\.
   104. **State reconciliation completeness across all tracked dimensions:** When a reconciler manages multiple categories of tracked state (e.g. files AND entities AND config values AND dependency versions), every diff algorithm within the reconciler MUST detect ALL change types (addition, modification, deletion) for EVERY category. Item 99 specifies the technique (bidirectional set comparison) for a single diff; item 33 (fix-class exhaustion) requires applying a fix across all code sites with the same pattern. This item applies both principles to the state model itself: when you implement or fix change detection for one state category, you MUST immediately enumerate ALL other state categories the reconciler tracks and verify each one supports all three change types. Concrete example: a reconciler that detects entity additions and entity deletions (item 99 applied) but only detects file additions (files in new state but not old) without detecting file orphans (files in old state but not new state) has a category-specific gap — the same class of bug (one-directional diff) at a different state dimension. After implementing or fixing any diff logic in a reconciler: (a) list every category of state the reconciler tracks (files, entities, config hashes, dependency versions, etc.); (b) for each category, verify the diff detects additions, modifications, AND deletions; (c) write a test for each category × change-type pair, with emphasis on the deletion case (the most commonly missed); (d) when fixing a deletion-detection bug for one category, apply item 33 (fix-class exhaustion) with the pattern signature "one-directional diff that only iterates the new collection" and grep every other diff function in the reconciler for the same pattern. A reconciler that correctly handles deletions for entities but not for files will leave orphaned files on disk that reference deleted types — breaking downstream compilation and violating "plan shows accurate changeset" requirements. **The principle: a reconciler's state model has N dimensions; each dimension needs three change types. That is 3×N coverage cells. Any cell that is missing is a silent data integrity gap. After adding or fixing any cell, audit all 3×N cells — not just the row or column you touched.**
   105. **Pipeline parameter consistency — no hardcoded defaults for parameterized concepts:** When functions in the same processing pipeline operate on the same configurable concept (e.g. output directory, base URL, registry path), and ANY function in the pipeline exposes that concept as a parameter, EVERY other function in the pipeline that references the same concept MUST either accept it as a parameter or derive it from the same configuration source. Item 49 covers partial parameterization within a single function or generator's output; this item extends it to parameter consistency ACROSS functions in a pipeline. After writing any function that accepts a configurable parameter: (a) trace the full pipeline — every function upstream (that prepares data for this function) and downstream (that acts on this function's output) that references the same concept; (b) grep for hardcoded occurrences of the parameter's default value in the pipeline (e.g. if `outDir` defaults to `"out"`, grep for the literal `"out"` in path construction calls across the pipeline); (c) replace every hardcoded occurrence with the parameterized form or thread the parameter through the call chain; (d) write a test that passes a non-default value for the parameter and verifies that ALL functions in the pipeline consistently use the non-default value — if any function still references the default, the parameterization is incomplete. **The principle: a parameter that is accepted by one function but hardcoded by another in the same pipeline is a ticking inconsistency.    106. **State and snapshot data structures must preserve all spec-required attributes for comparison and display:** When building an internal data structure (state record, snapshot, diff input) that captures entity or component state for later comparison or display, the structure MUST preserve every attribute that (a) the spec shows in example output for that feature, or (b) could change independently and constitute a meaningful state change. Concrete example: if the spec shows plan output as `+ field: display_name (string)` (name + type), but the entity snapshot stores only `[]string` of field names, then (1) `FormatPlan` cannot display types because the type information does not exist in the snapshot, and (2) `computeEntityChanges` cannot detect type changes (e.g. `string` → `int32`) because two snapshots with identical field names but different types are indistinguishable. Item 2 covers lossy type mappings in config schemas; this item extends the same principle to internal state and snapshot data structures used for reconciliation and display.**
   107. **Configurable values must use established resolution mechanisms — no inline defaults that bypass resolution:** When the codebase provides a resolution mechanism for a configurable concept (e.g. `resolveComponentConfig` for component configuration values, a config lookup function for registry-defined defaults), every usage of that concept's value MUST go through the resolution mechanism — never hardcode the default value inline. This is distinct from item 49 (partial parameterization within a generator's output) and item 105 (pipeline parameter consistency across functions): those cover parameters that are explicitly threaded through function signatures. The mechanism exists to centralize default-and-override logic; hardcoding the default elsewhere defeats this centralization. After writing any code that uses a configurable value with a known default: (a) search the codebase for an existing resolution function for that concept (e.g. grep for the config field name in resolver, config, or component-loading code); (b) if a resolution mechanism exists, call it instead of hardcoding the default; (c) if the resolution mechanism's output is already available in the current scope (e.g. via `ctx.ComponentConfig`), use the resolved value from that output; (d) write a test with a non-default override for the concept and verify the override takes effect everywhere the value is used — if any code path still uses the default, the bypass is the cause. **The litmus test: does the codebase already have a function or data structure that resolves this value?    108. **Struct literal completeness at integration boundaries — every field must be evaluated at every construction site:** Checklist item 1 requires that struct fields are assigned a non-zero value by "at least one code path."    109. **Conditional validation guards must defensively cover the absent-value branch:** When a validation check is gated on a configuration value being present (e.g. `if comp.OutputNamespace != ""`), the absent-value branch (`else`, or the implicit fall-through when there is no `else`) MUST explicitly handle the unvalidated case — either by applying an alternative validation, by rejecting the configuration as incomplete, or by asserting a defensive invariant that guarantees the validation is unnecessary.    110. **Aggregated output paths from multiple producers must be validated for uniqueness before writing:** When multiple independent producers (generators, assemblers, slot processors, fill loaders) each contribute output files to a shared collection that will be written to disk, the aggregator MUST validate that no two entries share the same output file path before any file is written.    111. **State-based plan actions must verify on-disk presence before assuming file exists:** When a reconciler computes file actions by comparing generated content against tracked state (e.g. "hash in current state matches hash of newly generated content → unchanged"), the state comparison alone is insufficient — the file may no longer exist on disk despite being tracked in the state (manual deletion, failed prior apply, filesystem corruption).    112. **Module structure must support all import paths the assembler generates:** When a code generator produces both a module definition file (e.g. `go.mod`) and source files that import packages from specific paths, every import path generated MUST be resolvable relative to the module root (the directory containing `go.mod`).
