# Review: Task 015 — CLI — All 11 Commands

## Round 1

- [x] [process-revision-complete] **`stego fill create --slot <s>` does not detect ambiguous slot names.** The spec states: "Each component owns its own slot protos (no sharing between components -- duplication is cheaper than coupling)." This means multiple components can define slots with the same name. At `cmd/stego/main.go:271-282`, the implementation iterates `reg.Components()` (a Go map) and picks the first component whose slot list contains the given name. Map iteration order in Go is non-deterministic, so when two or more components define the same slot name, the command silently picks an arbitrary component — producing different `fill.yaml` `implements` values across invocations. The implementation should detect the ambiguity (count of matching components > 1) and return an error listing the matching components, so the user can disambiguate.

## Round 2

- [x] [process-revision-complete] **`stego fill create` ambiguous-slot error suggests `--component` flag that does not exist.** At `cmd/stego/main.go:292`, the error message for ambiguous slot names instructs the user: "specify --component to disambiguate." However, the `runFillCreate` function's FlagSet at line 225-226 only defines `--slot` — no `--component` flag exists. The user cannot follow this guidance. Passing `--component comp-a` would either be captured by the manual flag parser (lines 232-239) as a flag argument with `comp-a` as its value, causing `fs.Parse` to fail with an "unknown flag" error, or — depending on argument ordering — be treated as a positional argument (the fill name). Either way, disambiguation is impossible. The ambiguity detection itself works correctly (per round 1 fix), but the recovery path is broken: the implementation detects the problem and tells the user what to do, but the suggested action doesn't work. Either implement the `--component` flag in the FlagSet and wire it into the component selection logic, or change the error message to suggest a working alternative (e.g., qualifying the slot name as `--slot <component>.<slot>` and adjusting the matching logic, or removing the suggestion entirely and just listing the components).

## Round 3

- [x] [process-revision-complete] **`stego registry search` output rows are in non-deterministic order.** At `cmd/stego/main.go:552`, `runRegistrySearch` iterates over `reg.Components()` which returns a `map[string]*types.Component`. Go map iteration order is non-deterministic by language specification (the runtime intentionally randomizes it). The same `stego registry search` invocation can produce table rows in different orders across runs. This affects scriptability (output cannot be reliably diff'd or piped), human readability (harder to spot a component in a shifting list), and test determinism. Fix: collect the component names from the map into a slice, `sort.Strings` it, then iterate the sorted slice to look up each component.

- [x] [process-revision-complete] **`stego registry inspect` config section is in non-deterministic order.** At `cmd/stego/main.go:622`, `runRegistryInspect` iterates `for key, field := range comp.Config` — another Go map. When a component has more than one config entry, the "Config:" section of `stego registry inspect` output appears in random order across invocations. The test fixture has only one config key (`port`), so this path is not exercised by tests. Fix: collect config keys into a slice, sort, then iterate in sorted order.

## Round 4

- [ ] **`stego fill create --component` flag is silently ignored when the slot name is unambiguous.** At `cmd/stego/main.go:289-310`, the component selection logic only validates the `--component` flag when multiple components define the requested slot (`len(matchingComps) > 1`). When exactly one component defines the slot, the code unconditionally sets `ownerComp = matchingComps[0]` at line 309, discarding the `--component` value without validation. This means `stego fill create my-fill --slot before_create --component nonexistent-component` silently succeeds and creates a fill implementing `rest-api.before_create`, even though the user explicitly specified `nonexistent-component`. The user receives no feedback that their `--component` value was ignored or invalid. The `--component` validation at lines 299-307 (verifying the named component is among the matching components) should apply regardless of how many components matched — if the user explicitly specifies a component, verify it actually defines the slot.
