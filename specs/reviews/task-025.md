# Review: Task 025 — Patch Operation with Patchable Fields

## Round 1

- [x] [process-revision-complete] **Patchable timestamp field causes compilation failure.** The `needTime` flag in `generateHandler` (`internal/generator/restapi/generator.go:341-356`) only triggers for computed timestamp fields with create/update/upsert operations. It does not account for `OpPatch`. When a patchable field has type `timestamp`, the generated patch request struct contains `*time.Time` but the handler file does not import `"time"`, producing a compilation error (`undefined: time`). Reproduced by generating a collection with `operations: [read, patch]` and `patchable: [scheduled_at]` where `scheduled_at` is type `timestamp` — the generated `handler_events.go` fails `go build` with `undefined: time` at the patch request struct definition.

- [x] [process-revision-complete] **Patch OpenAPI spec omits 404 response.** The patch handler performs a Get before merging (`internal/generator/restapi/generator.go:964`). If the entity does not exist, the handler returns 404 via `NotFound()`. However, the generated OpenAPI spec for the patch operation only declares a `200` response (`internal/generator/restapi/generator.go:1656-1658`). The Read operation — which follows the same Get pattern — correctly declares both `200` and `404` responses (`internal/generator/restapi/generator.go:1554-1557`). The patch operation should declare `404` as well to accurately document the API contract.

## Round 2

- [x] [process-revision-complete] **Patch request struct uses bare reference types instead of spec-prescribed pointer types for jsonb and bytes fields.** The rest-crud archetype spec (`specs/registry/archetypes/rest-crud/spec.md:125`) states: "The generator produces a patch request struct with pointer fields for only the listed fields (`*string`, `*int32`, `*json.RawMessage`, etc.)." The implementation at `internal/generator/restapi/generator.go:967-972` explicitly skips the pointer wrapper for `json.RawMessage` and `[]byte` Go types, using them as bare reference types instead. The corresponding apply logic at lines 1005-1009 uses direct assignment (`cluster.Spec = patch.Spec`) rather than pointer dereference (`cluster.Spec = *patch.Spec`). This creates a semantic divergence: with bare `json.RawMessage`, a client sending `{"spec": null}` produces `json.RawMessage("null")` (a non-nil byte slice containing the literal string "null"), which the `!= nil` guard treats as "provided" and applies to the entity. With the spec's `*json.RawMessage`, both an absent key and an explicit JSON null produce a nil pointer, treating both as "not provided." The implementation violates AC 5 ("Patch request struct uses pointer fields for patchable fields only") and the spec text. Affected fields: any patchable field of type `jsonb` or `bytes`.

## Round 3

- [x] [process-revision-complete] **`generatePatchMethod` introduces hardcoded local variable names not guarded in `handlerScopeIdentifiers`.** The `handlerScopeIdentifiers` map (`internal/generator/restapi/generator.go:1924-1953`) guards entity variable names against collisions with in-scope identifiers in generated handler methods — its own comment (lines 1920-1923) states: "A generated local variable whose name matches any of these will redeclare or shadow the identifier, causing a compile error." `generatePatchMethod` introduces two new hardcoded local variable names — `existing` (line 979: `existing, err := h.store.Get(...)`) and `patch` (line 991: `var patch {PatchReqType}`) — but neither is listed in `handlerScopeIdentifiers`. An entity named `Existing` produces `lower = "existing"`, and line 988 (`existing := existing.(Existing)`) is a `:=` redeclaration of the already-declared `existing` from line 979 — a compile error (`no new variables on left side of :=`). An entity named `Patch` produces `lower = "patch"`, and line 988 (`patch := existing.(Patch)`) declares `patch` which then collides with line 991's `var patch` — a redeclaration compile error. Both `"existing"` and `"patch"` must be added to `handlerScopeIdentifiers` so that `safeVarName` appends `_` when the entity name matches.
