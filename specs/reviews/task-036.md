# Review: Task 036 — Port Resolver Loads Override Components from Registry

## Findings

- [-] [process-revision-complete] **Override `InvalidBinding` errors produce malformed user-facing error messages.** When override validation fails (component not found in registry, component does not provide port, or no component loader configured), the `InvalidBinding` struct is created with `Component: ""` (`resolve.go:134`, `resolve.go:144`, `resolve.go:154`). The `InvalidBinding.Error()` method at `resolve.go:80` formats this as `invalid binding for port "auth-provider" required by "": <reason>`. The `required by ""` fragment is meaningless to users. Override-phase errors are port-level, not component-level, so the `Component` field has no applicable value — but the error format doesn't account for this. Either the `Error()` method should handle the empty-component case (e.g. omitting `required by` when `Component` is empty), or the `InvalidBinding` struct needs a way to distinguish override-phase errors from binding-resolution-phase errors so the message can be formatted appropriately.
- [-] [process-revision-complete] **Override port validation skipped when override component is already in the active set.** The `compProvidesPort` check at `resolve.go:156` is inside the `if _, exists := active[overrideComp]; !exists` block (line 135). When a service override references a component already present in `input.Components` (e.g., overriding `auth-provider` to `otel-tracing`, which is in the archetype's component list but provides `tracing`, not `auth-provider`), the provides-port validation is skipped entirely. The resolver proceeds to delete the archetype default from the active set (line 170) without confirming the override component actually provides the overridden port. The error IS caught later in the binding-resolution phase (line 277), but: (a) the error message attributes the problem to the requiring component (`required by "rest-api"`) rather than identifying the invalid override, and (b) the archetype default has already been unnecessarily removed from the active set. The fix is to move or duplicate the `compProvidesPort` check outside the `!exists` block so it applies to all override components, not just newly loaded ones.
- [-] [process-revision-complete] **No reconciler-level integration test for the ComponentLoader-based override loading path (AC 9).** The existing reconciler test `TestReconcile_ServicePortBindingOverridePassedToResolve` adds `api-key-auth` to the archetype's `components:` list (reconciler_test.go:1399), so the override component is already in `baselineComponents` when passed to `ports.Resolve()`. The `ComponentLoader` is never invoked. AC 9 requires `stego plan` and `stego validate` to work on `examples/user-management-rhsso`, which overrides `auth-provider` to `rh-sso-auth` — a component NOT in the archetype's component list. This path cannot be verified via CLI (the example's `.stego/config.yaml` references a non-existent remote registry), and no reconciler-level test exercises the flow where `ports.Resolve` calls `ComponentLoader` to load an override component from the registry, excludes the replaced default, and returns the modified active set that downstream generators use. The unit tests in `resolve_test.go` do test the ComponentLoader in isolation, but the integration path (Reconcile → resolver loads override → active set → generator execution with override component) is untested.
