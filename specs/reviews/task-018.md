# Review: Task 018 ŌĆö Collection Type System

## Round 1

- [x] [process-revision-complete] **slotVarName produces invalid Go identifiers from hyphenated collection names.** `slotVarName()` in `assembler.go:1460` concatenates the raw collection name into a Go variable name (`base += entity`). Collection names follow kebab-case (spec example: `org-users`), and hyphens are not valid in Go identifiers. The old `SlotDeclaration.Entity` field was always PascalCase (e.g., `User`), so this was safe. The rename to `Collection` introduces the bug ŌĆö `slotVarName("before_create", "org-users", "Gate")` produces `beforeCreateorg-usersGate`, which will cause a compile error in generated `main.go`. The same issue affects `collectAllSlotVarNames`, `buildSlotVarsByEntity`, and `writeSlotWiring`. The collection name must be sanitized (e.g., strip hyphens and apply camelCase) before concatenation.

- [x] [process-revision-complete] **Missing fill-collection cross-validation.** Task acceptance criterion #6 requires: "Validate that fill declarations reference existing collection names." The task description explicitly lists this under "Validation": "Validate that fill declarations reference existing collection names." The current `validateFillsExist` function (`validate.go:806`) only checks that `fills/<name>/fill.yaml` exists on disk ŌĆö it does not parse the fill.yaml or validate that its `Collection` field references a defined collection. This validation is absent.

- [x] [process-revision-complete] **Assembler error messages still say "entity" where they now mean "collection."** `validateSlotBindingUniqueness` in `assembler.go:1375` emits `"for entity %q"` but receives a collection name via `sb.Collection`. Similarly, `validateSlotVarNameUniqueness` at line 1409 uses a struct field named `entity` to hold a collection name, and its error message says `"entity %q"`. These should say "collection" to match the new semantics.

## Round 2

- [x] [process-revision-complete] **`ScopeField()` and `ParentEntity()` are non-deterministic for multi-field scopes, with no validation preventing them.** `Collection.ScopeField()` (`types.go:254`) and `Collection.ParentEntity()` (`types.go:264`) iterate over the `Scope` map and return the first key/value they encounter. Go map iteration order is undefined ŌĆö for a scope like `{org_id: Organization, team_id: Team}`, `ScopeField()` could return either `"org_id"` or `"team_id"` depending on the runtime, and `ParentEntity()` could return either entity. The `Scope` field is `map[string]string` by design (acceptance criterion #2), and `validateCollectionReferences` (`validate.go:450`) validates each scope entry independently but does not restrict the map to a single entry. Since the spec requires deterministic code generation ("STEGO deterministically generates production-ready code"), this creates a latent non-determinism bug. The downstream consumers of these methods ŌĆö `entityBasePath` (path derivation), `generateListMethod` (scope filtering), `validateScopeParentConsistency` (validation), and all nested-routing logic ŌĆö will produce different output across runs when multiple scope entries are present. Fix: add a validation rule in `validateCollectionReferences` that rejects scopes with more than one entry (e.g., "multi-field scopes are not yet supported") until the accessor methods and downstream consumers are updated to handle them deterministically.

- [x] [process-revision-complete] **`Wiring.ConstructorEntities` field name and documentation say "entity" but the field now stores collection names.** The `ConstructorEntities` field in `gen.go:106-112` is documented as "maps constructor index to the entity name it handles." After the task-018 changes, the generator stores the collection name (`eb.Name`, e.g., `"users"`) at `generator.go:150`, and the assembler correctly matches against collection-keyed maps built by `buildSlotVarsByEntity` (`assembler.go:1191`). But the public API documentation is wrong: a new generator implementation reading the godoc would pass entity names (e.g., `"User"`) instead of collection names (e.g., `"users"`), causing silent slot injection failures ŌĆö the assembler's lookup at `assembler.go:781-783` would find no match and skip injection without error. This is the same class of stale "entity" ŌåÆ "collection" terminology caught in round 1 finding #3, but in the cross-component API contract rather than error messages. The field itself, its godoc, the assembler function `buildSlotVarsByEntity`, and the assembler variable `slotVarsByEntity` all need renaming to reference "collection" instead of "entity."
