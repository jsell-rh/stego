# Review: Task 018 — Collection Type System

## Round 1

- [x] [process-revision-complete] **slotVarName produces invalid Go identifiers from hyphenated collection names.** `slotVarName()` in `assembler.go:1460` concatenates the raw collection name into a Go variable name (`base += entity`). Collection names follow kebab-case (spec example: `org-users`), and hyphens are not valid in Go identifiers. The old `SlotDeclaration.Entity` field was always PascalCase (e.g., `User`), so this was safe. The rename to `Collection` introduces the bug — `slotVarName("before_create", "org-users", "Gate")` produces `beforeCreateorg-usersGate`, which will cause a compile error in generated `main.go`. The same issue affects `collectAllSlotVarNames`, `buildSlotVarsByEntity`, and `writeSlotWiring`. The collection name must be sanitized (e.g., strip hyphens and apply camelCase) before concatenation.

- [x] [process-revision-complete] **Missing fill-collection cross-validation.** Task acceptance criterion #6 requires: "Validate that fill declarations reference existing collection names." The task description explicitly lists this under "Validation": "Validate that fill declarations reference existing collection names." The current `validateFillsExist` function (`validate.go:806`) only checks that `fills/<name>/fill.yaml` exists on disk — it does not parse the fill.yaml or validate that its `Collection` field references a defined collection. This validation is absent.

- [x] [process-revision-complete] **Assembler error messages still say "entity" where they now mean "collection."** `validateSlotBindingUniqueness` in `assembler.go:1375` emits `"for entity %q"` but receives a collection name via `sb.Collection`. Similarly, `validateSlotVarNameUniqueness` at line 1409 uses a struct field named `entity` to hold a collection name, and its error message says `"entity %q"`. These should say "collection" to match the new semantics.

## Round 2

- [x] [process-revision-complete] **`ScopeField()` and `ParentEntity()` are non-deterministic for multi-field scopes, with no validation preventing them.** `Collection.ScopeField()` (`types.go:254`) and `Collection.ParentEntity()` (`types.go:264`) iterate over the `Scope` map and return the first key/value they encounter. Go map iteration order is undefined — for a scope like `{org_id: Organization, team_id: Team}`, `ScopeField()` could return either `"org_id"` or `"team_id"` depending on the runtime, and `ParentEntity()` could return either entity. The `Scope` field is `map[string]string` by design (acceptance criterion #2), and `validateCollectionReferences` (`validate.go:450`) validates each scope entry independently but does not restrict the map to a single entry. Since the spec requires deterministic code generation ("STEGO deterministically generates production-ready code"), this creates a latent non-determinism bug. The downstream consumers of these methods — `entityBasePath` (path derivation), `generateListMethod` (scope filtering), `validateScopeParentConsistency` (validation), and all nested-routing logic — will produce different output across runs when multiple scope entries are present. Fix: add a validation rule in `validateCollectionReferences` that rejects scopes with more than one entry (e.g., "multi-field scopes are not yet supported") until the accessor methods and downstream consumers are updated to handle them deterministically.

- [x] [process-revision-complete] **`Wiring.ConstructorEntities` field name and documentation say "entity" but the field now stores collection names.** The `ConstructorEntities` field in `gen.go:106-112` is documented as "maps constructor index to the entity name it handles." After the task-018 changes, the generator stores the collection name (`eb.Name`, e.g., `"users"`) at `generator.go:150`, and the assembler correctly matches against collection-keyed maps built by `buildSlotVarsByEntity` (`assembler.go:1191`). But the public API documentation is wrong: a new generator implementation reading the godoc would pass entity names (e.g., `"User"`) instead of collection names (e.g., `"users"`), causing silent slot injection failures — the assembler's lookup at `assembler.go:781-783` would find no match and skip injection without error. This is the same class of stale "entity" → "collection" terminology caught in round 1 finding #3, but in the cross-component API contract rather than error messages. The field itself, its godoc, the assembler function `buildSlotVarsByEntity`, and the assembler variable `slotVarsByEntity` all need renaming to reference "collection" instead of "entity."

## Round 3

- [x] [process-revision-complete] **Deprecated `entity:` key in slot declarations and fill files is silently accepted, creating a migration gap.** The parser rejects the top-level `expose:` key with a clear migration error (`types.go:305-307`), establishing the precedent that renamed YAML keys should produce actionable errors during migration. However, the `entity:` key within slot declarations (under `slots:` in service.yaml) and within fill files (`fills/*/fill.yaml`) is silently ignored by `yaml.v3`'s default behavior of discarding unknown keys. After task-018, `SlotDeclaration.Entity` was renamed to `SlotDeclaration.Collection` (yaml tag changed from `"entity"` to `"collection"`) and `Fill.Entity` was renamed to `Fill.Collection` (same). A user migrating from the old schema who updates `expose:` → `collections:` (because the parser forces them to) but forgets to update `entity:` → `collection:` in their slot declarations will have slot bindings with empty `Collection` fields. These empty-collection bindings pass validation (`validateSlotBindingCollections` at `validate.go:628` skips empty collections) but produce functionally dead slot operators — `writeSlotWiring` emits the operator variable with `_ =` (unused) and `buildSlotVarsByCollection` (`assembler.go:1197`) skips empty-collection bindings, so no slot operator is ever injected into a handler constructor. The same applies to fill.yaml files: `entity: Widget` is silently ignored, `Fill.Collection` is empty, and `validateFillsExist` (`validate.go:848`) skips validation for empty collections. Fix: either add migration error detection in the `ServiceDeclaration.UnmarshalYAML` for `entity:` keys within slot declaration mappings and in `ParseFill` for fill.yaml files, or add a validation rule that rejects slot bindings and fills with empty `Collection` fields (since the spec says "Slots bind to collections, not entities").

- [x] [process-revision-complete] **18 test fill.yaml fixtures in `validate_test.go` use the deprecated `entity:` key, masking the absence of fill-collection cross-validation coverage.** Test cases at lines 648, 697, 1055, 1135, 1949, 2002, 2060, 2117, 2176, 2525, 2583, 2637, 2690, 2744, 2921, 2977, 3033, and 3091 create fill.yaml files with `entity: Widget` (the pre-task-018 field name) instead of `collection: widgets` (the current field name per acceptance criterion #4 and the spec's fill example). Since `Fill.Entity` was renamed to `Fill.Collection` with yaml tag `"collection"`, the YAML key `entity:` is silently ignored, leaving `Fill.Collection` empty in all 18 fixtures. The fill-collection cross-validation added in round 1 (`validateFillsExist` at `validate.go:843-859`) is only exercised by two tests (`TestValidate_FillExists` at line 445 and `TestValidate_FillBadCollection` at line 497), both of which correctly use `collection:`. The remaining 18 tests pass but do not exercise the new validation path — they test with fills that have no collection binding, which is the exact gap identified in finding #1 above. Fix: update the 18 fill.yaml fixtures to use `collection: widgets` (or the appropriate collection name for each test) so they match the spec's fill format and exercise the fill-collection cross-validation.

## Round 4

- [x] [process-revision-complete] **Multi-field scope validation only runs via `stego validate`, not via `stego plan` or `stego apply`, leaving the non-determinism bug reachable.** The round 2 fix added multi-field scope rejection to `validateCollectionReferences` (validate.go:448–456), which is called by `Validate()`. However, `Reconcile()` (reconciler.go:103) — used by `stego plan` and `stego apply` — does not call `Validate()` or perform any equivalent scope cardinality check. It parses service.yaml independently and passes collections directly to generators. A user running `stego plan` with a multi-field scope (e.g., `scope: {org_id: Organization, team_id: Team}`) bypasses the guard entirely. The generator's validation functions (`validateFieldReferences` at generator.go:1207 and `validateScopeParentConsistency` at generator.go:1475) call `ScopeField()` (types.go:254) and `ParentEntity()` (types.go:264), which iterate the Go `Scope` map and return the first entry found — undefined for maps with more than one entry. Handler generation (`generateListMethod` at generator.go:567–575) uses `ScopeField()` to emit the scope filter in generated code, and `entityBasePath` (generator.go:220–231) uses `ParentEntity()` to derive route paths. Different runs of `stego plan` on the same input can produce different routes, different scope filters, and different parent paths. Before task-018, `Scope` was a `string`, so multi-field scopes were structurally impossible; the type change to `map[string]string` introduced this concern, and the round 2 fix only covered one of two code paths. Fix: either add a multi-field scope cardinality check in `Reconcile()` before generators run, or add the check to the generator's own validation functions (e.g., in `validateFieldReferences` before calling `ScopeField()`).
