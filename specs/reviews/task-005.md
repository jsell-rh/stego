# Review: Task 005 — Port Resolution Engine

## Findings

- [x] [process-revision-complete] **Auto-discovery fallback is not specified.** The spec defines exactly two resolution steps for ports: (1) archetype default bindings and (2) service-level overrides. The implementation adds a third "auto-discovery" step (`resolve.go:132-147`) where, if no explicit binding exists for a required port, the resolver searches the component set for a unique provider. This behavior is not described anywhere in the spec. The spec's Port Resolution section (lines 179-201) says: *"The archetype declares default bindings... The service declaration can override any binding... Unresolved or ambiguous ports are a compile error."* A missing archetype binding IS an unresolved port per spec — auto-discovery silently masks it. This could hide a platform-team omission (forgot to add a binding to the archetype) if there happens to be exactly one provider present. The `TestResolveAutoDiscovery` test validates this non-spec behavior. Either auto-discovery should be removed (and `TestResolveAutoDiscovery` updated to expect an error), or the spec needs to be amended to include it as a resolution strategy.

- [x] [process-revision-complete] **Error messages do not distinguish binding misconfiguration from truly unresolved ports.** When an explicit binding points to a component that doesn't exist (`resolve.go:120-126`, tested by `TestResolveBindingToNonExistentComponent`) or to a component that doesn't provide the required port (`TestResolveBindingToComponentThatDoesNotProvidePort`), the error is the same `UnresolvedPort` type as when no component anywhere provides the port (`resolve.go:135-137`). The spec says "Unresolved or ambiguous ports are a compile error" with the implication of clear diagnostics. An error like `"binding for port 'storage-adapter' references component 'otel-tracing', which does not provide it"` is actionably different from `"unresolved port 'storage-adapter': no component provides it"`. The current code produces the latter for both cases, sending the user hunting for missing providers when the real problem is a misconfigured binding. A distinct error type (e.g. `InvalidBinding`) or at minimum a different message would satisfy the "clear messages" requirement from the task description.

- [x] [process-revision-complete] **Self-resolution is not prevented.** A component that both `requires` and `provides` the same port name would resolve to itself — either via auto-discovery (if it is the sole provider) or via an explicit binding that names it. This creates a circular dependency: the component depends on itself to fulfill its own requirement. The spec describes `requires`/`provides` as a cross-component wiring mechanism. The `providers` index (`resolve.go:73-77`) includes every component's provides without excluding self-references, and `componentProvides` (`resolve.go:167-178`) does not check whether the bound component is the same as the requiring component. A guard like `if bound == compName { ... }` on the explicit-binding path and filtering self from `providerList` on the auto-discovery path would prevent this class of error.

- [x] [process-revision-complete] **`AmbiguousPort` is dead code and the acceptance criterion for ambiguous port errors is unmet.** The `AmbiguousPort` struct (`resolve.go:52-58`), the `ResolutionError.Ambiguous` field (`resolve.go:23`), the `hasErrors()` check for `Ambiguous` (`resolve.go:42`), and the ambiguity formatting in `Error()` (`resolve.go:35-36`) are all defined but **nothing ever appends to `resErr.Ambiguous`** — no code path populates it. The acceptance criteria explicitly require *"Clear error on ambiguous port (two providers for same port)"* and *"Unit tests for happy path + both error cases"* (unresolved and ambiguous). The old `TestResolveAmbiguousPort` test was removed with auto-discovery, and the remaining `TestResolveAmbiguousResolvedByBinding` only tests the success path (binding disambiguates). The spec says: *"Unresolved or ambiguous ports are a compile error."* In the current binding-only model, when no binding exists and multiple components provide a port, the resolver reports it as `UnresolvedPort` — losing the actionable distinction. A user seeing `"unresolved port 'storage-adapter': no component provides it"` when two components DO provide it (but no binding picks one) gets a misleading diagnostic. The fix is to rebuild the providers index (removed with auto-discovery) and use it only for error classification on the no-binding path: 0 providers → `UnresolvedPort`, 2+ providers → `AmbiguousPort`. This satisfies the acceptance criteria, the spec, and provides better diagnostics without reintroducing auto-discovery (1 provider still requires an explicit binding).
