# Review: Task 009 — rest-api Component Generator

## Round 1

- [ ] **1. `component.yaml` operations enum missing `upsert`** — The spec defines six operations: `create`, `read`, `update`, `delete`, `list`, and `upsert` (spec §Entity & Operation Features). The component's config schema at `registry/components/rest-api/component.yaml:18` declares only `[create, read, update, delete, list]`. The generator code supports upsert, but the declarative config schema used for validation omits it.

- [ ] **2. `update` and `upsert` both map to `http.MethodPut` — duplicate switch case** — In `generator.go:169-170` (`OpUpdate`) and `generator.go:181-182` (`OpUpsert`), both operations generate `case http.MethodPut:`. If a service declaration exposes both `update` and `upsert` on the same entity, the generated `switch` contains duplicate `case` branches, which is a Go compile error. This violates the acceptance criterion "Generator produces compilable Go handler code."

- [ ] **3. `extractPathParam` uses `r.PathValue()` but routes are registered without pattern parameters** — The router registers routes as `mux.Handle("/users/", http.StripPrefix("/users", userHandler))` (generator.go:384-385) using bare path prefixes without Go 1.22 `{param}` syntax. `extractPathParam` (generator.go:404) calls `r.PathValue(name)`, which only returns values when the mux pattern includes `{param}` placeholders. Since no patterns use this syntax, `PathValue` always returns `""`. This breaks parent ID extraction in nested routing and scope field extraction in list operations. The spec (§Nested routing) requires "parent existence verification at each level" — the code is emitted but non-functional at runtime.

- [ ] **4. Generated entity struct types are empty** — `generateRouter` (generator.go:372) emits `type <Entity> struct{}` for each exposed entity. The handler code decodes JSON into these types (`json.NewDecoder(r.Body).Decode(&user)`) and encodes them back — but an empty struct cannot hold any data. Worse, `generateCreateMethod` (generator.go:231) emits field assignments like `nodepool.ClusterID = extractPathParam(...)` on these empty structs, which is a compile error. The generator has access to entity fields via `ctx.Entities` but does not use them to populate the struct definitions.

- [ ] **5. Tests do not verify generated code compiles as a unit** — The acceptance criterion states "Tests verify generated code compiles." The test `TestGenerate_GeneratedGoCodeCompiles` (generator_test.go:580) only checks that individual files pass `go/format.Source` (syntax validity) and contain `package api`. Cross-file references — handlers referencing `Storage` interface and entity types from `router.go`, field assignments on empty structs — are not validated. The generated output will fail `go build` when multiple files are compiled together (e.g., finding 4's field assignment on an empty struct).
