# Review: Task 009 — rest-api Component Generator

## Round 1

- [x] [process-revision-complete] **1. `component.yaml` operations enum missing `upsert`** — The spec defines six operations: `create`, `read`, `update`, `delete`, `list`, and `upsert` (spec §Entity & Operation Features). The component's config schema at `registry/components/rest-api/component.yaml:18` declares only `[create, read, update, delete, list]`. The generator code supports upsert, but the declarative config schema used for validation omits it.

- [x] [process-revision-complete] **2. `update` and `upsert` both map to `http.MethodPut` — duplicate switch case** — In `generator.go:169-170` (`OpUpdate`) and `generator.go:181-182` (`OpUpsert`), both operations generate `case http.MethodPut:`. If a service declaration exposes both `update` and `upsert` on the same entity, the generated `switch` contains duplicate `case` branches, which is a Go compile error. This violates the acceptance criterion "Generator produces compilable Go handler code."

- [x] [process-revision-complete] **3. `extractPathParam` uses `r.PathValue()` but routes are registered without pattern parameters** — The router registers routes as `mux.Handle("/users/", http.StripPrefix("/users", userHandler))` (generator.go:384-385) using bare path prefixes without Go 1.22 `{param}` syntax. `extractPathParam` (generator.go:404) calls `r.PathValue(name)`, which only returns values when the mux pattern includes `{param}` placeholders. Since no patterns use this syntax, `PathValue` always returns `""`. This breaks parent ID extraction in nested routing and scope field extraction in list operations. The spec (§Nested routing) requires "parent existence verification at each level" — the code is emitted but non-functional at runtime.

- [x] [process-revision-complete] **4. Generated entity struct types are empty** — `generateRouter` (generator.go:372) emits `type <Entity> struct{}` for each exposed entity. The handler code decodes JSON into these types (`json.NewDecoder(r.Body).Decode(&user)`) and encodes them back — but an empty struct cannot hold any data. Worse, `generateCreateMethod` (generator.go:231) emits field assignments like `nodepool.ClusterID = extractPathParam(...)` on these empty structs, which is a compile error. The generator has access to entity fields via `ctx.Entities` but does not use them to populate the struct definitions.

- [x] [process-revision-complete] **5. Tests do not verify generated code compiles as a unit** — The acceptance criterion states "Tests verify generated code compiles." The test `TestGenerate_GeneratedGoCodeCompiles` (generator_test.go:580) only checks that individual files pass `go/format.Source` (syntax validity) and contain `package api`. Cross-file references — handlers referencing `Storage` interface and entity types from `router.go`, field assignments on empty structs — are not validated. The generated output will fail `go build` when multiple files are compiled together (e.g., finding 4's field assignment on an empty struct).

## Round 2

- [x] [process-revision-complete] **6. Scope filtering is non-functional — `r.PathValue(scope_field)` never matches a path parameter** — In `generateListMethod` (generator.go:282), when `eb.Scope` is set, the handler emits `scopeValue := r.PathValue("org_id")`. However, no route pattern contains a `{org_id}` path parameter. Without a parent, the route is `GET /users` (zero path params). With a parent (e.g., `parent: Organization`), the route is `GET /organizations/{organization_id}/users` — the path param is `{organization_id}`, not `{org_id}`. In both cases `r.PathValue("org_id")` returns `""` and list always queries with an empty scope value. The spec (§Entity & Operation Features) shows scope on the User expose block, meaning filtered list is a required behavior. The test `TestGenerate_ScopeFiltering` (generator_test.go:311) only asserts string presence (`"org_id"`, `"scopeValue"`) — it does not verify the scope value is actually extractable from the route at runtime.

- [x] [process-revision-complete] **7. OpenAPI spec omits path parameters for nested routes** — When generating OpenAPI paths for nested entities (e.g., `/clusters/{cluster_id}/nodepools`), the `generateOpenAPI` function (generator.go:470-580) declares path parameters only for `{id}` on item-level operations (read, update, delete). Parent path parameters like `{cluster_id}` are never declared in the operation's `parameters` array, despite appearing in the URL template. Per OpenAPI 3.0.3 §8.3.3, every path parameter placeholder in the path template MUST have a corresponding parameter declaration. The generated spec is invalid — tools like `swagger-ui`, `openapi-generator`, or spec validators will reject it. The test `TestGenerate_OpenAPISpec` (generator_test.go:435) only validates the basic User entity (no nesting) and doesn't check nested route parameter declarations.

- [x] [process-revision-complete] **8. Computed fields are included in write operations and request schemas** — The spec (§Computed/derived fields) states: "Computed/derived fields are read-only fields populated by a fill, never written via the API." The generator does not reference `Field.Computed` anywhere (confirmed: zero matches for `computed` or `Computed` in `generator.go`). Consequently: (a) generated `create`, `update`, and `upsert` handlers decode the full JSON body into the entity struct including computed fields — API consumers can submit values for fields that should be read-only; (b) generated OpenAPI schemas include computed fields without `readOnly: true`, so there is no signal to API consumers that these fields are read-only; (c) the entity struct definition in `router.go` correctly includes computed fields for responses, but nothing prevents them from being written. The spec example shows `status_conditions` as `computed: true, filled_by: status-aggregator` — this field must not be writable via the API.

## Round 3

- [ ] [process-revision-complete] **9. `openapi.json` is invalid JSON when rendered via `File.Bytes()`** — The `gen.File.Bytes()` method (gen.go:61) prepends `// Code generated by stego. DO NOT EDIT.` to all files, including `openapi.json`. This Go-comment prefix makes the rendered JSON file syntactically invalid — no JSON parser will accept it. The spec (§Generated Code Structure) states "All generated files carry a `// Code generated by stego. DO NOT EDIT.` header", but this rule uses Go comment syntax which is incompatible with JSON. Any downstream tool (OpenAPI validators, swagger-ui, openapi-generator) consuming the rendered file will fail to parse it. The test `TestGenerate_OpenAPISpec` (generator_test.go:490) reads `f.Content` directly rather than `f.Bytes()`, masking the issue. The test `TestGenerate_FilesBytesIncludesHeader` (generator_test.go:860) asserts the header on all files including JSON, confirming the invalid output is the intended behavior of the current code.

- [ ] [process-revision-complete] **10. Upsert handler does not set parent ID from path parameter** — When an entity has a parent and exposes upsert, `generateUpsertMethod` (generator.go:310-342) calls `emitParentCheck` to verify the parent exists but does not assign the parent's path parameter to the entity's ref field. Compare to `generateCreateMethod` (generator.go:219-222), which correctly emits `lower.ParentRefField = r.PathValue(parentIDParam)`. The spec (§Entity & Operation Features) shows `AdapterStatus` with `parent: NodePool` and `operations: [list, upsert]` — the upserted entity would be persisted without its parent reference, breaking the parent-child relationship. Since upsert creates a record if it does not exist, the parent ID is required for new records, identical to the create case.

- [ ] [process-revision-complete] **11. Handler unconditionally imports `"encoding/json"` — compile error for delete-only entities** — `generateHandler` (generator.go:136-138) always emits `import "encoding/json"` regardless of which operations are exposed. The `json` package is used in create, read, update, list, and upsert methods but NOT in delete. If an entity exposes only `delete`, the generated handler has an unused import, which is a Go compilation error (`imported and not used`). The `go/format.Source` call (generator.go:190) is a formatter, not a compiler — it does not catch unused imports. The compilation test `TestGenerate_GeneratedCodeCompilesAsPackage` (generator_test.go:960) uses entities with all 6 operations and does not cover this edge case. The acceptance criterion states "Generator produces compilable Go handler code for CRUD operations."
