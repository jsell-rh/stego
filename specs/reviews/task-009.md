# Review: Task 009 — rest-api Component Generator

## Round 1

- [x] [process-revision-complete] **1. `component.yaml` operations enum missing `upsert`** — The spec defines six operations: `create`, `read`, `update`, `delete`, `list`, and `upsert` (spec §Entity & Operation Features). The component's config schema at `registry/components/rest-api/component.yaml:18` declares only `[create, read, update, delete, list]`. The generator code supports upsert, but the declarative config schema used for validation omits it.

- [x] [process-revision-complete] **2. `update` and `upsert` both map to `http.MethodPut` — duplicate switch case** — In `generator.go:169-170` (`OpUpdate`) and `generator.go:181-182` (`OpUpsert`), both operations generate `case http.MethodPut:`. If a service declaration exposes both `update` and `upsert` on the same entity, the generated `switch` contains duplicate `case` branches, which is a Go compile error. This violates the acceptance criterion "Generator produces compilable Go handler code."

- [x] [process-revision-complete] **3. `extractPathParam` uses `r.PathValue()` but routes are registered without pattern parameters** — The router registers routes as `mux.Handle("/users/", http.StripPrefix("/users", userHandler))` (generator.go:384-385) using bare path prefixes without Go 1.22 `{param}` syntax. `extractPathParam` (generator.go:404) calls `r.PathValue(name)`, which only returns values when the mux pattern includes `{param}` placeholders. Since no patterns use this syntax, `PathValue` always returns `""`. This breaks parent ID extraction in nested routing and scope field extraction in list operations. The spec (§Nested routing) requires "parent existence verification at each level" — the code is emitted but non-functional at runtime.

- [x] [process-revision-complete] **4. Generated entity struct types are empty** — `generateRouter` (generator.go:372) emits `type <Entity> struct{}` for each exposed entity. The handler code decodes JSON into these types (`json.NewDecoder(r.Body).Decode(&user)`) and encodes them back — but an empty struct cannot hold any data. Worse, `generateCreateMethod` (generator.go:231) emits field assignments like `nodepool.ClusterID = extractPathParam(...)` on these empty structs, which is a compile error. The generator has access to entity fields via `ctx.Entities` but does not use them to populate the struct definitions.

- [x] [process-revision-complete] **5. Tests do not verify generated code compiles as a unit** — The acceptance criterion states "Tests verify generated code compiles." The test `TestGenerate_GeneratedGoCodeCompiles` (generator_test.go:580) only checks that individual files pass `go/format.Source` (syntax validity) and contain `package api`. Cross-file references — handlers referencing `Storage` interface and entity types from `router.go`, field assignments on empty structs — are not validated. The generated output will fail `go build` when multiple files are compiled together (e.g., finding 4's field assignment on an empty struct).

## Round 2

- [x] [process-revision-complete] **6. Scope filtering is non-functional — `r.PathValue(scope_field)` never matches a path parameter** — In `generateListMethod` (generator.go:282), when `eb.Scope` is set, the handler emits `scopeValue := r.PathValue("org_id")`. However, no route pattern contains a `{org_id}` path parameter. Without a parent, the route is `GET /users` (zero path params). With a parent (e.g., `parent: Organization`), the route is `GET /organizations/{organization_id}/users` — the path param is `{organization_id}`, not `{org_id}`. In both cases `r.PathValue("org_id")` returns `""` and list always queries with an empty scope value. The spec (§Entity & Operation Features) shows scope on the User expose block, meaning filtered list is a required behavior. The test `TestGenerate_ScopeFiltering` (generator_test.go:311) only asserts string presence (`"org_id"`, `"scopeValue"`) — it does not verify the scope value is actually extractable from the route at runtime.

- [x] [process-revision-complete] **7. OpenAPI spec omits path parameters for nested routes** — When generating OpenAPI paths for nested entities (e.g., `/clusters/{cluster_id}/nodepools`), the `generateOpenAPI` function (generator.go:470-580) declares path parameters only for `{id}` on item-level operations (read, update, delete). Parent path parameters like `{cluster_id}` are never declared in the operation's `parameters` array, despite appearing in the URL template. Per OpenAPI 3.0.3 §8.3.3, every path parameter placeholder in the path template MUST have a corresponding parameter declaration. The generated spec is invalid — tools like `swagger-ui`, `openapi-generator`, or spec validators will reject it. The test `TestGenerate_OpenAPISpec` (generator_test.go:435) only validates the basic User entity (no nesting) and doesn't check nested route parameter declarations.

- [x] [process-revision-complete] **8. Computed fields are included in write operations and request schemas** — The spec (§Computed/derived fields) states: "Computed/derived fields are read-only fields populated by a fill, never written via the API." The generator does not reference `Field.Computed` anywhere (confirmed: zero matches for `computed` or `Computed` in `generator.go`). Consequently: (a) generated `create`, `update`, and `upsert` handlers decode the full JSON body into the entity struct including computed fields — API consumers can submit values for fields that should be read-only; (b) generated OpenAPI schemas include computed fields without `readOnly: true`, so there is no signal to API consumers that these fields are read-only; (c) the entity struct definition in `router.go` correctly includes computed fields for responses, but nothing prevents them from being written. The spec example shows `status_conditions` as `computed: true, filled_by: status-aggregator` — this field must not be writable via the API.

## Round 4

- [x] [process-revision-complete] **12. Generated handlers never set `Content-Type: application/json` — mismatch with generated OpenAPI spec** — Every handler method that writes JSON responses (`create`, `read`, `update`, `list`, `upsert`) uses `json.NewEncoder(w).Encode(...)` without calling `w.Header().Set("Content-Type", "application/json")`. Go's `net/http` auto-detects `Content-Type` via `http.DetectContentType`, which returns `text/plain; charset=utf-8` for JSON payloads (the sniffer does not recognize JSON). The generated OpenAPI spec declares `application/json` as the response media type via the `jsonContent` helper (generator.go:730-734). The runtime behavior and the declared API contract disagree: the OpenAPI spec promises `application/json` responses, but the generated handlers serve `text/plain; charset=utf-8`. The spec (§MVP Scope) lists both "handlers" and "OpenAPI" as rest-api component outputs — they should be consistent.

- [x] [process-revision-complete] **13. `emitClearComputedFields` emits `= nil` for computed timestamp fields — compile error** — In `emitClearComputedFields` (generator.go:699-717), the `default` case (line 713-714) emits `entity.Field = nil` for any type not explicitly handled. `fieldTypeToGo(FieldTypeTimestamp)` returns `"time.Time"` (line 762), which is a value type (struct), not a pointer or slice. Assigning `nil` to a `time.Time` field is a compile error: `cannot use nil as time.Time value in assignment`. A computed timestamp field (e.g., `{name: last_aggregated, type: timestamp, computed: true, filled_by: aggregator}`) would produce uncompilable code. The correct zero value is `time.Time{}`. The test `TestGenerate_ComputedFieldsCompilesAsPackage` (generator_test.go:774) covers jsonb and bool computed fields but not timestamp, so this case is untested. The acceptance criterion states "Generator produces compilable Go handler code for CRUD operations."

- [x] [process-revision-complete] **14. `component.yaml` config schema missing expose block fields from spec §Entity & Operation Features** — The `component.yaml` config schema (lines 10-21) declares expose items with only `entity`, `operations`, and `scope`. The spec (§Entity & Operation Features) defines four additional expose block properties: `parent` (nested routing), `path_prefix` (custom URL paths), `upsert_key` (natural-key conflict resolution), and `concurrency` (optimistic concurrency). The generator code handles all four — they are fields on `types.ExposeBlock` and used throughout `generator.go`. But the component's config schema omits them. Since `stego validate` "checks service.yaml against registry" (spec §CLI Interface) and the config schema is the component's declared interface, a valid service declaration using any of these four fields would be unrecognized by the schema. The generator works, but the component's self-description is incomplete.

## Round 5

- [x] [process-revision-complete] **15. OpenAPI spec omits response body schema for `create`, `update`, and `upsert` operations** — The handlers for `create` (generator.go:261-263), `update` (generator.go:297-298), and `upsert` (generator.go:381-383) all write JSON response bodies via `json.NewEncoder(w).Encode(...)` and set `Content-Type: application/json` (per finding #12 fix). However, the OpenAPI spec only declares response content for `read` and `list` operations. The `create` response (generator.go:565-567) declares `"201": {Description: "Created"}` with no `Content` field. Similarly, `update` (generator.go:604-605) declares `"200": {Description: "Updated"}` and `upsert` (generator.go:633-634) declares `"200": {Description: "Upserted"}` — both without `Content`. Meanwhile, `read` (generator.go:585) and `list` (generator.go:544-548) correctly declare response content with `jsonContent(...)`. The generated handlers and the generated OpenAPI spec disagree on whether these operations return a body. Client code generators (e.g., `openapi-generator`) will treat these as no-body responses and discard the JSON the server sends. The spec (§MVP Scope) lists both "handlers" and "OpenAPI" as rest-api component outputs — they should be consistent.

- [x] [process-revision-complete] **16. Wiring routes reference unexported handler methods — unusable from `cmd/main.go`** — The `Wiring.Routes` entries (generator.go:64-80) produce expressions like `mux.HandleFunc("POST /users", userHandler.create)`. The handler methods (`create`, `read`, `update`, `delete`, `list`, `upsert`) are all lowercase/unexported (generator.go:245, 268, 284, 302, 316, 350). The spec (§Generated Code Structure) says "Each component's generator returns a Wiring struct (imports, constructors, routes) alongside its files, and the compiler assembles the shared files from all wiring declarations." The task description says "Returns Wiring for main.go assembly." Since `cmd/main.go` is in package `main`, it cannot access unexported methods from package `api`. The wiring route expressions are non-functional code fragments that would produce a compile error if the compiler embedded them in `main.go`. The `NewRouter` function in `router.go` works correctly because it registers routes within the same package, but the wiring contract — the component's interface with the compiler — is broken. The `Wiring` struct doc (gen.go:79-80) says routes are "route registration expressions for main.go assembly" — they must be usable in package `main`.

## Round 6

No test covers the parent-only List path — `TestGenerate_ScopeFilteringWithParent` (generator_test.go:311) only covers the case where both scope AND parent are set.

## Round 7

- [x] [process-revision-complete] **19. `parentRefFieldName` fallback produces compile errors when entity lacks a ref field to the parent** — `parentRefFieldName` (generator.go:761-768) searches entity fields for a `ref` type with `To == parent`. If none is found, it falls back to `parent + "ID"` (e.g., `"ClusterID"` for parent `"Cluster"`). This fallback name is emitted as a struct field access in `generateCreateMethod` (line 255: `%s.%s = r.PathValue(...)`) and `generateUpsertMethod` (line 360). If an entity declares `parent: Foo` but has no field of type `ref` with `to: Foo`, the generated code references a non-existent struct field — a compile error. The same issue exists in `parentRefRawFieldName` (generator.go:773-780), whose fallback `strings.ToLower(parent) + "_id"` is passed to `store.List` in the parent-only branch (line 332-334). The spec (§Nested routing) shows parent relationships backed by explicit ref fields, but the generator silently accepts the mismatch and produces uncompilable code instead of returning an error. The acceptance criterion states "Generator produces compilable Go handler code for CRUD operations." No test covers an entity with `parent` set but no matching ref field.

- [x] [process-revision-complete] **20. OpenAPI entity schemas omit `required` array — all fields appear optional** — The spec's entity type system (§Entity Type System) distinguishes required from optional fields: `{name: metadata, type: jsonb, optional: true}` is optional; fields without `optional: true` (e.g., `{name: email, type: string, unique: true}`) are implicitly required. The `fieldToOpenAPISchema` function (generator.go:704-732) never reads `Field.Optional`. The `openAPISchema` struct (generator.go:884-892) has no `Required []string` field. Consequently, the generated OpenAPI schema for every entity omits the `required` array — all fields appear optional to API consumers and code generators. For example, the spec's User entity has `email` (required), `role` (required), and `org_id` (required) — but the generated schema does not mark any of them as required. Per OpenAPI 3.0.3 §4.7.24.2, `required` lists "the required properties." The rest-api component "provides: openapi-spec" (component.yaml:41) — the generated spec should accurately reflect the entity's field requirements.

- [x] [process-revision-complete] **21. Entity names that are Go keywords produce compile errors in generated handler code** — `strings.ToLower(entity.Name)` is used as a local variable name throughout handler methods: `var <lower> <Entity>` (generator.go:247), `json.NewDecoder(r.Body).Decode(&<lower>)` (line 248), and many more. Go has 25 reserved keywords (`type`, `map`, `range`, `func`, `select`, `switch`, `case`, `default`, `break`, `continue`, `return`, `var`, `const`, `if`, `else`, `for`, `go`, `defer`, `chan`, `struct`, `interface`, `package`, `import`, `goto`, `fallthrough`). If an entity is named with any of these words (e.g., entity "Type"), the generated code contains statements like `var type Type` — a Go syntax error that `format.Source` will reject. The generator should either detect and escape keyword collisions or return an error. The acceptance criterion states "Generator produces compilable Go handler code for CRUD operations."

## Round 8

- [x] [process-revision-complete] **22. OpenAPI entity schemas omit field constraint attributes — `minLength`, `maxLength`, `pattern`, `minimum`, `maximum`, and integer/float format are lost** — The spec (§Entity Type System) defines constraint attributes as "essential complexity" that describes "what the data IS": `min_length`, `max_length`, `pattern` (strings), `min`, `max` (numerics). The `Field` struct in `types.go:63-67` carries all of these (`MinLength *int`, `MaxLength *int`, `Pattern string`, `Min *float64`, `Max *float64`). However, `fieldToOpenAPISchema` (generator.go:729-757) ignores all of them — it maps only the base type, format (for bytes and timestamp), enum values, and readOnly. The `openAPISchema` struct (generator.go:936-945) lacks the corresponding JSON fields (`minLength`, `maxLength`, `pattern`, `minimum`, `maximum`). Consequently, an entity field declared as `{name: name, type: string, min_length: 3, max_length: 53, pattern: "^[a-z0-9]"}` (spec §Entity Type System example) generates an OpenAPI schema of just `{"type": "string"}` — all three constraints are silently dropped. Additionally, `int32` and `int64` both map to `{"type": "integer"}` with no `format` (generator.go:733-734), and `float` and `double` both map to `{"type": "number"}` (generator.go:735-736) — OpenAPI consumers cannot distinguish precision. Per OpenAPI 3.0.3 §4.7.24.1, `minLength`, `maxLength`, `pattern`, `minimum`, `maximum`, and `format` are standard schema properties. The rest-api component declares `provides: openapi-spec` (component.yaml:42). The generated spec should reflect the entity's declared constraints. No test covers constraint attributes in OpenAPI output — `TestGenerate_OpenAPIFieldTypes` (generator_test.go:716) only checks base types, not constraints.

- [x] [process-revision-complete] **23. Nested routing `checkParent` only verifies the immediate parent — ancestor levels are unverified for multi-level nesting** — The spec (§Nested routing) states: "The compiler generates parent existence verification at each level." For multi-level nesting (e.g., Cluster → NodePool → AdapterStatus), `entityBasePath` (generator.go:111-124) correctly builds the full hierarchical path by recursing up the parent chain: `/clusters/{cluster_id}/nodepools/{nodepool_id}/adapterstatuss`. However, `generateHandler` (generator.go:188-204) only generates a `checkParent` helper for the entity's immediate parent. For AdapterStatus with `parent: NodePool`, the generated `checkParent` extracts `r.PathValue("nodepool_id")` and calls `h.store.Exists("NodePool", nodepoolID)`. The `{cluster_id}` path parameter is present in the URL but never extracted or verified — a request to `GET /clusters/nonexistent/nodepools/1/adapterstatuss` succeeds if NodePool 1 exists, regardless of whether `nonexistent` is a valid Cluster. The spec says "at each level," which means all ancestor entities in the URL hierarchy should be verified, not just the immediate parent. The spec's own example (§Nested routing) shows a three-level hierarchy (Cluster → NodePool → AdapterStatus) as the motivating case. No test covers multi-level nesting beyond two levels.

## Round 9

- [x] [process-revision-complete] **24. Circular parent references cause infinite loop in `entityBasePath` and `collectAncestors` — generator hangs instead of returning an error** — `entityBasePath` (generator.go:111-124) recursively calls itself at line 118 when resolving nested parent paths: `parentPath := entityBasePath(parentEB, exposeMap)`. `collectAncestors` (generator.go:815-831) iterates the parent chain in a `for` loop at line 818. Neither function detects cycles. If the service declaration contains a circular parent relationship (e.g., entity A declares `parent: B` and entity B declares `parent: A`, both in the expose list), `entityBasePath` produces infinite recursion (stack overflow) and `collectAncestors` produces an infinite loop (unbounded memory growth). The generator hangs or crashes instead of returning a descriptive error. The spec's reconciler pattern (§Compiler Process) operates on user-provided service declarations, and `stego validate` exists to "check service.yaml against registry" (§CLI Interface). A generator that hangs on invalid-but-parseable input violates the acceptance criterion "Generator produces compilable Go handler code" — it produces no output at all. No test covers circular parent references.

- [x] [process-revision-complete] **25. OpenAPI entity schemas omit `default` constraint attribute — field defaults are invisible to API consumers** — The spec (§Entity Type System) lists `default` as a constraint attribute in the same finite set as `min_length`, `max_length`, `pattern`, `min`, `max`, `optional`, `values`, and `computed`. Finding 22 addressed `minLength`, `maxLength`, `pattern`, `minimum`, `maximum`, and `format` — all now propagated to the OpenAPI schema. However, `default` was not included. The `Field` struct in types.go:71 carries `Default any`. The `openAPISchema` struct (generator.go:985-999) has no `Default` field. `fieldToOpenAPISchema` (generator.go:733-787) does not read `f.Default`. Consequently, a field declared as `{name: role, type: enum, values: [admin, member], default: member}` generates an OpenAPI schema with `{"type": "string", "enum": ["admin", "member"]}` — the default value `member` is silently dropped. Per OpenAPI 3.0.3 §4.7.24.1, `default` is a standard schema property that "declares the value of the parameter that the server will use if none is provided." The rest-api component declares `provides: openapi-spec` (component.yaml:42). Client code generators (e.g., `openapi-generator`) use `default` to set initial values in generated client models. The test `TestGenerate_OpenAPIConstraintAttributes` (generator_test.go:1710) validates `minLength`, `maxLength`, `pattern`, `minimum`, `maximum`, and `format` but not `default`.

## Round 10

- [x] [process-revision-complete] **26. `update` handler does not assign parent ID from path parameter — nested entity parent reference can be overwritten or cleared by client** — `generateCreateMethod` (generator.go:279-286) and `generateUpsertMethod` (generator.go:393-399) both override the entity's parent ref field with the path parameter value after JSON decode: `entity.ClusterID = r.PathValue("cluster_id")`. `generateUpdateMethod` (generator.go:313-330) does not — it decodes the request body, clears computed fields, and passes the entity directly to `store.Update` without setting the parent field. For a nested route like `PUT /clusters/{cluster_id}/nodepools/{id}`, a client can send a request body with a different `cluster_id` value (pointing to a different cluster) or omit it entirely (zeroing the field to `""`). In both cases, the stored entity's parent reference will not match the URL hierarchy. This breaks the nested routing invariant: a NodePool accessed at `/clusters/1/nodepools/5` could end up with `cluster_id: "2"` or `cluster_id: ""` after an update. The spec (§Nested routing) states: "Nested routing is expressed via `parent` on an expose block. The compiler generates parent existence verification at each level." The `checkAncestors` call verifies the parent exists, but the entity's parent field is not pinned to the verified parent's ID. `create` and `upsert` both enforce this correctly; `update` is the only write operation that does not. No test covers `update` with `parent` — `TestGenerate_GeneratedCodeCompilesAsPackage` (generator_test.go:1069) includes `update` with `parent` but only verifies compilation, not parent ID assignment behavior.

## Round 12

- [x] [process-revision-complete] **28. Entity names whose lowercase form matches handler method identifiers produce compile errors — `safeVarName` does not guard against in-scope parameter, receiver, or import names** — `safeVarName` (generator.go:923-928) prevents collisions with Go keywords and predeclared identifiers (the `goReservedWords` set at generator.go:905-918). ## Round 13

- [x] [process-revision-complete] **29. Generated Go package name and wiring metadata are hardcoded to `"api"` / `"internal/api"` instead of deriving from `ctx.OutputNamespace`** — The generator uses `ctx.OutputNamespace` for file path construction (generator.go:261, 543, 738 via `path.Join(ns, ...)`) but hardcodes the same value in three other places: (a) package declaration `"package api"` (generator.go:184, 467) — should be `"package " + filepath.Base(ctx.OutputNamespace)`; (b) wiring imports `[]string{"internal/api"}` (generator.go:107) — should be `[]string{ctx.OutputNamespace}`; (c) wiring constructor prefix `"api.New%sHandler(store)"` (generator.go:63) — should use `filepath.Base(ctx.OutputNamespace)` as the package qualifier. The `gen.Context.OutputNamespace` field (gen.go:44) exists to decouple generators from their component.yaml config. File paths correctly use the parameter, but package name and wiring do not.
