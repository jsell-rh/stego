# Review: Task 009 — rest-api Component Generator

## Round 1

- [x] [process-revision-complete] **1. `component.yaml` operations enum missing `upsert`** — The spec defines six operations: `create`, `read`, `update`, `delete`, `list`, and `upsert` (spec §Entity & Operation Features). The component's config schema at `registry/components/rest-api/component.yaml:18` declares only `[create, read, update, delete, list]`. The generator code supports upsert, but the declarative config schema used for validation omits it.

- [x] [process-revision-complete] **2. `update` and `upsert` both map to `http.MethodPut` — duplicate switch case** — In `generator.go:169-170` (`OpUpdate`) and `generator.go:181-182` (`OpUpsert`), both operations generate `case http.MethodPut:`. If a service declaration exposes both `update` and `upsert` on the same entity, the generated `switch` contains duplicate `case` branches, which is a Go compile error. This violates the acceptance criterion "Generator produces compilable Go handler code."

- [x] [process-revision-complete] **3. `extractPathParam` uses `r.PathValue()` but routes are registered without pattern parameters** — The router registers routes as `mux.Handle("/users/", http.StripPrefix("/users", userHandler))` (generator.go:384-385) using bare path prefixes without Go 1.22 `{param}` syntax. `extractPathParam` (generator.go:404) calls `r.PathValue(name)`, which only returns values when the mux pattern includes `{param}` placeholders. Since no patterns use this syntax, `PathValue` always returns `""`. This breaks parent ID extraction in nested routing and scope field extraction in list operations. The spec (§Nested routing) requires "parent existence verification at each level" — the code is emitted but non-functional at runtime.

- [x] [process-revision-complete] **4. Generated entity struct types are empty** — `generateRouter` (generator.go:372) emits `type <Entity> struct{}` for each exposed entity. The handler code decodes JSON into these types (`json.NewDecoder(r.Body).Decode(&user)`) and encodes them back — but an empty struct cannot hold any data. Worse, `generateCreateMethod` (generator.go:231) emits field assignments like `nodepool.ClusterID = extractPathParam(...)` on these empty structs, which is a compile error. The generator has access to entity fields via `ctx.Entities` but does not use them to populate the struct definitions.

- [x] [process-revision-complete] **5. Tests do not verify generated code compiles as a unit** — The acceptance criterion states "Tests verify generated code compiles." The test `TestGenerate_GeneratedGoCodeCompiles` (generator_test.go:580) only checks that individual files pass `go/format.Source` (syntax validity) and contain `package api`. Cross-file references — handlers referencing `Storage` interface and entity types from `router.go`, field assignments on empty structs — are not validated. The generated output will fail `go build` when multiple files are compiled together (e.g., finding 4's field assignment on an empty struct).

## Round 2

- [x] [process-revision-complete] **6. Scope filtering is non-functional — `r.PathValue(scope_field)` never matches a path parameter** — In `generateListMethod` (generator.go:282), when `eb.Scope` is set, the handler emits `scopeValue := r.PathValue("org_id")`. However, no route pattern contains a `{org_id}` path parameter. Without a parent, the route is `GET /users` (zero path params). With a parent (e.g., `parent: Organization`), the route is `GET /organizations/{organization_id}/users` — the path param is `{organization_id}`, not `{org_id}`. In both cases `r.PathValue("org_id")` returns `""` and list always queries with an empty scope value. The spec (§Entity & Operation Features) shows scope on the User expose block, meaning filtered list is a required behavior. The test `TestGenerate_ScopeFiltering` (generator_test.go:311) only asserts string presence (`"org_id"`, `"scopeValue"`) — it does not verify the scope value is actually extractable from the route at runtime.

- [x] [process-revision-complete] **7. OpenAPI spec omits path parameters for nested routes** — When generating OpenAPI paths for nested entities (e.g., `/clusters/{cluster_id}/nodepools`), the `generateOpenAPI` function (generator.go:470-580) declares path parameters only for `{id}` on item-level operations (read, update, delete). Parent path parameters like `{cluster_id}` are never declared in the operation's `parameters` array, despite appearing in the URL template. Per OpenAPI 3.0.3 §8.3.3, every path parameter placeholder in the path template MUST have a corresponding parameter declaration. The generated spec is invalid — tools like `swagger-ui`, `openapi-generator`, or spec validators will reject it. The test `TestGenerate_OpenAPISpec` (generator_test.go:435) only validates the basic User entity (no nesting) and doesn't check nested route parameter declarations.

- [x] [process-revision-complete] **8. Computed fields are included in write operations and request schemas** — The spec (§Computed/derived fields) states: "Computed/derived fields are read-only fields populated by a fill, never written via the API." The generator does not reference `Field.Computed` anywhere (confirmed: zero matches for `computed` or `Computed` in `generator.go`). Consequently: (a) generated `create`, `update`, and `upsert` handlers decode the full JSON body into the entity struct including computed fields — API consumers can submit values for fields that should be read-only; (b) generated OpenAPI schemas include computed fields without `readOnly: true`, so there is no signal to API consumers that these fields are read-only; (c) the entity struct definition in `router.go` correctly includes computed fields for responses, but nothing prevents them from being written. The spec example shows `status_conditions` as `computed: true, filled_by: status-aggregator` — this field must not be writable via the API.

## Round 3

- [x] [process-revision-complete] **9. `openapi.json` is invalid JSON when rendered via `File.Bytes()`** — Fixed. `gen.File.Bytes()` (gen.go:61-67) now only prepends the Go comment header for `.go` files; non-Go files return content as-is. Tests at generator_test.go:861 and generator_test.go:1195 confirm rendered JSON is valid.

- [x] [process-revision-complete] **10. Upsert handler does not set parent ID from path parameter** — Fixed. `generateUpsertMethod` (generator.go:332-334) now assigns the parent's path parameter to the entity's ref field when a parent is set. Test at generator_test.go:1111 confirms.

- [x] [process-revision-complete] **11. Handler unconditionally imports `"encoding/json"` — compile error for delete-only entities** — Fixed. `generateHandler` (generator.go:137-143) now conditionally imports `encoding/json` only when non-delete operations are present. Test at generator_test.go:1148 confirms delete-only entities compile.

## Round 4

- [x] [process-revision-complete] **12. Generated handlers never set `Content-Type: application/json` — mismatch with generated OpenAPI spec** — Every handler method that writes JSON responses (`create`, `read`, `update`, `list`, `upsert`) uses `json.NewEncoder(w).Encode(...)` without calling `w.Header().Set("Content-Type", "application/json")`. Go's `net/http` auto-detects `Content-Type` via `http.DetectContentType`, which returns `text/plain; charset=utf-8` for JSON payloads (the sniffer does not recognize JSON). The generated OpenAPI spec declares `application/json` as the response media type via the `jsonContent` helper (generator.go:730-734). The runtime behavior and the declared API contract disagree: the OpenAPI spec promises `application/json` responses, but the generated handlers serve `text/plain; charset=utf-8`. The spec (§MVP Scope) lists both "handlers" and "OpenAPI" as rest-api component outputs — they should be consistent.

- [x] [process-revision-complete] **13. `emitClearComputedFields` emits `= nil` for computed timestamp fields — compile error** — In `emitClearComputedFields` (generator.go:699-717), the `default` case (line 713-714) emits `entity.Field = nil` for any type not explicitly handled. `fieldTypeToGo(FieldTypeTimestamp)` returns `"time.Time"` (line 762), which is a value type (struct), not a pointer or slice. Assigning `nil` to a `time.Time` field is a compile error: `cannot use nil as time.Time value in assignment`. A computed timestamp field (e.g., `{name: last_aggregated, type: timestamp, computed: true, filled_by: aggregator}`) would produce uncompilable code. The correct zero value is `time.Time{}`. The test `TestGenerate_ComputedFieldsCompilesAsPackage` (generator_test.go:774) covers jsonb and bool computed fields but not timestamp, so this case is untested. The acceptance criterion states "Generator produces compilable Go handler code for CRUD operations."

- [x] [process-revision-complete] **14. `component.yaml` config schema missing expose block fields from spec §Entity & Operation Features** — The `component.yaml` config schema (lines 10-21) declares expose items with only `entity`, `operations`, and `scope`. The spec (§Entity & Operation Features) defines four additional expose block properties: `parent` (nested routing), `path_prefix` (custom URL paths), `upsert_key` (natural-key conflict resolution), and `concurrency` (optimistic concurrency). The generator code handles all four — they are fields on `types.ExposeBlock` and used throughout `generator.go`. But the component's config schema omits them. Since `stego validate` "checks service.yaml against registry" (spec §CLI Interface) and the config schema is the component's declared interface, a valid service declaration using any of these four fields would be unrecognized by the schema. The generator works, but the component's self-description is incomplete.

## Round 5

- [x] [process-revision-complete] **15. OpenAPI spec omits response body schema for `create`, `update`, and `upsert` operations** — The handlers for `create` (generator.go:261-263), `update` (generator.go:297-298), and `upsert` (generator.go:381-383) all write JSON response bodies via `json.NewEncoder(w).Encode(...)` and set `Content-Type: application/json` (per finding #12 fix). However, the OpenAPI spec only declares response content for `read` and `list` operations. The `create` response (generator.go:565-567) declares `"201": {Description: "Created"}` with no `Content` field. Similarly, `update` (generator.go:604-605) declares `"200": {Description: "Updated"}` and `upsert` (generator.go:633-634) declares `"200": {Description: "Upserted"}` — both without `Content`. Meanwhile, `read` (generator.go:585) and `list` (generator.go:544-548) correctly declare response content with `jsonContent(...)`. The generated handlers and the generated OpenAPI spec disagree on whether these operations return a body. Client code generators (e.g., `openapi-generator`) will treat these as no-body responses and discard the JSON the server sends. The spec (§MVP Scope) lists both "handlers" and "OpenAPI" as rest-api component outputs — they should be consistent.

- [x] [process-revision-complete] **16. Wiring routes reference unexported handler methods — unusable from `cmd/main.go`** — The `Wiring.Routes` entries (generator.go:64-80) produce expressions like `mux.HandleFunc("POST /users", userHandler.create)`. The handler methods (`create`, `read`, `update`, `delete`, `list`, `upsert`) are all lowercase/unexported (generator.go:245, 268, 284, 302, 316, 350). The spec (§Generated Code Structure) says "Each component's generator returns a Wiring struct (imports, constructors, routes) alongside its files, and the compiler assembles the shared files from all wiring declarations." The task description says "Returns Wiring for main.go assembly." Since `cmd/main.go` is in package `main`, it cannot access unexported methods from package `api`. The wiring route expressions are non-functional code fragments that would produce a compile error if the compiler embedded them in `main.go`. The `NewRouter` function in `router.go` works correctly because it registers routes within the same package, but the wiring contract — the component's interface with the compiler — is broken. The `Wiring` struct doc (gen.go:79-80) says routes are "route registration expressions for main.go assembly" — they must be usable in package `main`.

## Round 6

- [ ] **17. `generateListMethod` parent-only branch passes PascalCase Go field name as `scopeField` — inconsistent with scope branches** — In `generateListMethod` (generator.go:329-334), the parent-only branch (no scope set) calls `parentRefFieldName(entity, eb.Parent)` which returns `toPascalCase(f.Name)` (e.g., `"ClusterID"` for a field named `cluster_id`). The generated code becomes `h.store.List("NodePool", "ClusterID", clusterID)`. In contrast, the scope+parent branch (generator.go:323-325) and the scope-only branch (generator.go:326-328) both pass the raw YAML field name as the scope field: `h.store.List("User", "org_id", scopeValue)`. The `Storage.List` interface (router.go) accepts `scopeField string` — the two code paths pass semantically different key formats (PascalCase Go struct name vs snake_case YAML field name) for the same parameter. A storage implementation cannot handle both without ambiguity. The parent-only branch should pass the raw field name (`f.Name`, i.e. `"cluster_id"`) rather than the PascalCase conversion. No test covers the parent-only List path — `TestGenerate_ScopeFilteringWithParent` (generator_test.go:311) only covers the case where both scope AND parent are set.

- [ ] **18. OpenAPI spec omits query parameter declaration for scope filtering without parent** — When `Scope` is set without `Parent`, `generateListMethod` (generator.go:326-328) emits `r.URL.Query().Get("org_id")` to extract the scope value from a query parameter. However, the OpenAPI list operation (generator.go:538-554) only adds `parentParams` (path parameters extracted from the URL template) to the parameters array. No query parameter is ever added for the scope field. The generated OpenAPI spec has no declaration for the `org_id` query parameter, while the generated handler requires it for filtering. Client code generators and API documentation will not show this parameter. The spec (§MVP Scope) lists both "handlers" and "OpenAPI" as rest-api component outputs — they should be consistent. The test `TestGenerate_ScopeFilteringWithoutParent` (generator_test.go:353) validates handler code but not the corresponding OpenAPI output.
