# Review: Task 028 — Upsert Operation with Conflict Resolution

## Round 1

- [-] [process-revision-complete] **postgres-adapter: COUNT query error not checked** (`internal/generator/postgresadapter/generator.go:767`). The pre-existence COUNT query `s.db.WithContext(ctx).Model(&v).Where(whereClause).Count(&existingCount)` does not check for errors. If the database query fails (connection lost, invalid query, etc.), `existingCount` remains at its zero value (`0`), and the method silently proceeds to the INSERT. This means a database error is swallowed and the `created` return value defaults to `true` regardless of actual state. All other database operations in the generated store check errors — this is an omission. **Process fix: added checklist item 192 — every intermediate I/O result in generated methods must have its error checked.**

- [-] [process-revision-complete] **postgres-adapter: TOCTOU race in insert/update distinction** (`internal/generator/postgresadapter/generator.go:756-831`). The pre-existence check (COUNT at line 767) and the actual upsert (INSERT...ON CONFLICT at line 824) are two separate database queries with no transactional isolation between them. Under concurrent upserts targeting the same natural key, the `created` return value can be incorrect: a concurrent INSERT between the COUNT and the CREATE would cause the CREATE to hit ON CONFLICT (update path), but `existingCount == 0` would still report `created = true`, producing a 201 instead of the correct 200. The reverse race (concurrent DELETE between COUNT and INSERT) would produce 200 instead of 201. Consider wrapping both operations in a transaction, or using PostgreSQL's `xmax` system column from a `RETURNING` clause to determine insert vs update atomically. **Process fix: added checklist item 193 — multi-step database operations requiring atomicity must use transactions or single-statement alternatives.**

## Round 2

- [ ] **rest-api: `handlerScopeIdentifiers` missing `created` and `upsertKey`** (`internal/generator/restapi/generator.go:2159-2191`). The `generateUpsertMethod` introduces two hardcoded local variables in the generated handler body: `created` (line 964, via `created, err := h.store.Upsert(...)`) and `upsertKey` (line 955, via `upsertKey := []string{...}`). Neither is listed in `handlerScopeIdentifiers`. Per process checklist item 188, every hardcoded local introduced by a `generate*Method` function must be added to the set. The `created` omission has practical impact: an entity named "Created" produces `safeVarName(strings.ToLower("Created"))` = `"created"`, which collides with the `created` return-value variable in the upsert handler, causing a type-mismatch compile error (`bool` vs entity type) in the generated code. The `upsertKey` omission is process-only (camelCase cannot be produced by `strings.ToLower`, so no realistic collision exists), but item 188 requires all hardcoded locals to be registered regardless.

- [ ] **postgres-adapter: transaction isolation level insufficient for TOCTOU prevention** (`internal/generator/postgresadapter/generator.go:815`). The generated transaction uses `s.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error { ... })` with no isolation level specification. GORM's `Transaction()` defaults to the database's default isolation level, which is Read Committed for PostgreSQL. Under Read Committed, each statement within the transaction sees the latest committed data — not a snapshot. A concurrent INSERT that commits between the COUNT (line 827) and the INSERT...ON CONFLICT (line 831) within the same transaction will cause the `created` flag to be incorrect: COUNT returns 0 (row didn't exist when COUNT ran), but INSERT hits ON CONFLICT UPDATE (row now exists from the concurrent commit), producing `created = true` when the actual operation was an update. This is the same TOCTOU race from round 1, now moved inside a transaction that doesn't prevent it. Process checklist item 193 explicitly requires SERIALIZABLE or REPEATABLE READ isolation for transaction-wrapped pre-check + mutation patterns. GORM's `Transaction` accepts `*sql.TxOptions` as a variadic second argument: `s.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error { ... }, &sql.TxOptions{Isolation: sql.LevelSerializable})`.
