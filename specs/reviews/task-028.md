# Review: Task 028 — Upsert Operation with Conflict Resolution

## Round 1

- [ ] **postgres-adapter: COUNT query error not checked** (`internal/generator/postgresadapter/generator.go:767`). The pre-existence COUNT query `s.db.WithContext(ctx).Model(&v).Where(whereClause).Count(&existingCount)` does not check for errors. If the database query fails (connection lost, invalid query, etc.), `existingCount` remains at its zero value (`0`), and the method silently proceeds to the INSERT. This means a database error is swallowed and the `created` return value defaults to `true` regardless of actual state. All other database operations in the generated store check errors — this is an omission.

- [ ] **postgres-adapter: TOCTOU race in insert/update distinction** (`internal/generator/postgresadapter/generator.go:756-831`). The pre-existence check (COUNT at line 767) and the actual upsert (INSERT...ON CONFLICT at line 824) are two separate database queries with no transactional isolation between them. Under concurrent upserts targeting the same natural key, the `created` return value can be incorrect: a concurrent INSERT between the COUNT and the CREATE would cause the CREATE to hit ON CONFLICT (update path), but `existingCount == 0` would still report `created = true`, producing a 201 instead of the correct 200. The reverse race (concurrent DELETE between COUNT and CREATE) would produce 200 instead of 201. Consider wrapping both operations in a transaction, or using PostgreSQL's `xmax` system column from a `RETURNING` clause to determine insert vs update atomically.
