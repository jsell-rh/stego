- [x] [process-revision-complete] **Upsert: optimistic concurrency WHERE clause appended to DO NOTHING produces invalid SQL.** In `emitUpsertMethod`, the optimistic concurrency `WHERE` clause (line 464) is appended unconditionally after both the `DO UPDATE SET` and `DO NOTHING` branches. - [x] [process-revision-complete] **SQL string literals not escaped in migration DDL.** `columnDefinition` interpolates field values into SQL with single-quote wrapping but no escaping of embedded single quotes: (a) `f.Pattern` at line 579 (`CHECK (col ~ 'pattern')`), (b) enum `f.Values` entries at line 565 (`CHECK (col IN ('val1', 'val2'))`), (c) string defaults in `sqlDefault` at line 597 (`DEFAULT 'value'`). ## Round 2

- [x] [process-revision-complete] **Generated model structs use non-pointer types for nullable columns, causing runtime scan failures.** Optional fields (`optional: true`) and computed fields (`computed: true`) produce nullable columns in the migration DDL (no `NOT NULL`). However, `generateModels` maps all fields to non-pointer Go types (e.g., `string`, `int32`, `time.Time`) regardless of nullability. When `database/sql.Rows.Scan` encounters a SQL NULL value for these columns, it cannot convert NULL into a non-pointer type and returns a runtime error like `"converting NULL to string is unsupported"`. Affected types: `string`, `int32`, `int64`, `float32`, `float64`, `bool`, `time.Time`. (`[]byte` and `json.RawMessage` handle nil correctly.) The generated Read and List methods scan directly into struct fields (`&v.FieldName`), so any entity with optional or computed fields of the affected types will fail at runtime when the column is NULL. Ref: `generator.go:112-116` (model field emission), `generator.go:237-240` (Read scan args), `generator.go:338-340` (List scan args).

- [x] [process-revision-complete] **`emitUpdateMethod` produces invalid SQL when an entity has only computed fields.** If every field in an entity is `computed: true`, then `writeColumns(e)` returns an empty slice. `emitUpdateMethod` builds `setClauses` from `writeCols`, producing an empty slice. `strings.Join([]string{}, ", ")` returns `""`, so the generated SQL becomes `UPDATE table_name SET  WHERE id = $1` â€” a syntax error. - [x] [process-revision-complete] **`toSnakeCase` mishandles consecutive uppercase letters (acronyms).** The function inserts an underscore before every uppercase rune unconditionally (`if i > 0 && unicode.IsUpper(r)`). For entity names containing acronyms â€” e.g., `HTTPServer`, `APIKey`, `UserID` â€” this produces `h_t_t_p_server`, `a_p_i_key`, `user_i_d` instead of the expected `http_server`, `api_key`, `user_id`. These malformed snake_case names propagate to table names, column references, and all generated SQL. While no current spec examples use acronym entity names, entity names are user-supplied and the spec imposes no restriction against acronyms. Ref: `generator.go:673-682`.

- [x] [process-revision-complete] **`unique_composite` field names not validated against entity fields, and duplicate constraints emitted.** The `generateMigration` function (lines 532-536) iterates all fields and emits `UNIQUE (f.UniqueComposite...)` for each field whose `UniqueComposite` slice is non-empty. Ref: `generator.go:532-536`, spec Â§Entity Type System `unique_composite: [field1, field2]`.

## Round 3

- [x] [process-revision-complete] **`ref` type `to` attribute silently ignored â€” no FOREIGN KEY constraint generated.** The spec's Entity Type System defines `ref` as a stego-specific type with a `to` attribute pointing to another entity (e.g., `{name: org_id, type: ref, to: Organization}`). The spec states: "The component decides HOW to enforce them (SQL constraints, middleware, ORM annotations)." For a PostgreSQL adapter, the standard enforcement is a `REFERENCES` clause. Currently, `columnDefinition` maps `ref` to plain `TEXT` and never inspects `Field.To`. The `generateMigration` function never emits `REFERENCES` or `FOREIGN KEY` clauses. The `to` attribute is dead metadata â€” referential integrity is not enforced at the database level. Ref: `generator.go:614-662` (`columnDefinition`), `generator.go:574-609` (`generateMigration`), spec Â§Entity Type System (`ref` type), spec Â§Entity Type System enforcement principle.

- [x] [process-revision-complete] **No validation that `ref` field's `to` attribute references an existing entity.** The generator validates entity name uniqueness (`validateEntityUniqueness`), case-insensitive collisions (`validateCaseInsensitiveUniqueness`), reserved names (`validateReservedNames`), and `unique_composite` field references (`validateUniqueComposites`). However, `ref` fields' `to` values are never checked against the entity list. A typo like `to: Organziation` produces no error â€” the generator silently generates a plain TEXT column. Since `to` is already ignored (finding above), the missing validation compounds the issue: the user gets no feedback that their ref target is wrong. Ref: `generator.go:25-43` (validation calls in `Generate`), `types.go:75` (`To` field on `Field` struct).

- [x] [process-revision-complete] **Migration SQL file uses non-standard header instead of spec-mandated text.** The spec's first enforcement rule states: "All generated files carry a `// Code generated by stego. ## Round 4

- [x] [process-revision-complete] **Migration SQL emits FOREIGN KEY references before referenced tables exist.** `generateMigration` emits `CREATE TABLE` statements in the order entities appear in `ctx.Entities`. When entity A has a `ref` field pointing to entity B, the generator emits `REFERENCES B_table(id)` inline. If A appears before B in the entity list, PostgreSQL rejects the migration because the referenced table does not yet exist. Example: `basicContext()` in tests defines `User` (with `org_id ref to Organization`) before `Organization`. The generated SQL is `CREATE TABLE IF NOT EXISTS users (... org_id TEXT NOT NULL REFERENCES organizations(id)); CREATE TABLE IF NOT EXISTS organizations (...)` â€” this fails at execution time with `relation "organizations" does not exist`. The generator needs to topologically sort entities by ref dependencies before emitting migration DDL, or use deferred foreign key constraints. Ref: `generator.go:584-613` (entity iteration order), `generator.go:597-599` (REFERENCES emission), `generator_test.go:14-40` (`basicContext` entity ordering).

- [x] [process-revision-complete] **`ref` field with empty `to` attribute silently generates a plain TEXT column with no error.** Both `validateRefTargets` (line 908) and the REFERENCES emission in `generateMigration` (line 597) guard on `f.To != ""`. A `ref` field with an omitted or empty `to` attribute passes all validation and produces a bare `TEXT` column with no foreign key constraint. The `ref` type's defining characteristic per the spec is its `to` attribute (e.g., `{name: org_id, type: ref, to: Organization}`). A ref without a target is semantically invalid â€” the generator should return a compile error rather than silently degrading to a plain TEXT column. Ref: `generator.go:597` (migration guard), `generator.go:908` (validation guard), spec Â§Entity Type System (`ref` type definition).

## Round 5

- [x] [process-revision-complete] **Optimistic concurrency `WHERE` clause references hardcoded `generation` column without validating the entity has such a field.** `emitUpsertMethod` unconditionally emits `WHERE EXCLUDED.generation > table.generation` for every entity when `concurrency == "optimistic"` (line 482â€“484). The generator has full knowledge of each entity's fields at generation time, but never checks whether a `generation` field exists. If the Upsert method is called at runtime with `concurrency: "optimistic"` for an entity that lacks a `generation` field (e.g., an entity with only `label` and `category`), the generated SQL references a non-existent column and PostgreSQL returns a runtime error. The generator should either (a) only emit the optimistic concurrency branch for entities that have a `generation` field, or (b) emit a runtime guard that returns an error when optimistic concurrency is requested for an entity without a `generation` column. Ref: `generator.go:482-484`, spec Â§Entity & Operation Features (`concurrency: optimistic`).

- [x] [process-revision-complete] **No validation of field name uniqueness within an entity.** The generator validates entity name uniqueness (`validateEntityUniqueness`), case-insensitive entity name collisions (`validateCaseInsensitiveUniqueness`), and reserved entity names (`validateReservedNames`). However, field names within an entity are never checked for duplicates. If two fields share the same name (e.g., two fields both named `email`), the generator produces: (a) duplicate Go struct fields â€” the model struct has two `Email` fields, causing a Go compilation error; (b) duplicate SQL column definitions â€” the migration has two `email` columns, causing a PostgreSQL error. The generator should validate field name uniqueness per entity and return a compile error. Ref: `generator.go:25-48` (validation calls), `generator.go:117-131` (model struct emission), `generator.go:660-672` (migration column emission).

- [x] [process-revision-complete] **SQL identifiers derived from field names are not quoted, producing invalid SQL when names collide with PostgreSQL reserved words.** Field names from the entity definition are interpolated directly into DDL (`columnDefinition`, line 694â€“742) and DML (`emitCreateMethod`, `emitReadMethod`, `emitUpdateMethod`, `emitDeleteMethod`, `emitListMethod`, `emitUpsertMethod`) without quoting. PostgreSQL reserved words used as field names â€” e.g., `order`, `check`, `default`, `group`, `table`, `select`, `primary` â€” produce syntactically invalid SQL. For example, a field named `order` generates `order INTEGER NOT NULL` in the migration, which PostgreSQL rejects because `ORDER` is a reserved keyword. The standard PostgreSQL mitigation is to double-quote identifiers (`"order" INTEGER NOT NULL`). The generator should either double-quote all column identifiers in generated SQL, or validate field names against PostgreSQL reserved words at generation time. Ref: `generator.go:694-742` (columnDefinition), `generator.go:190-233` (emitCreateMethod), `generator.go:238-270` (emitReadMethod), spec Â§Entity Type System (field names are user-supplied, no restriction against reserved words).

## Round 6

- [ ] **User-declared field named `id` collides with the implicit ID primary key, producing uncompilable Go code and invalid SQL.** The generator unconditionally adds an `ID string` field to every model struct (line 124) and an `"id" TEXT PRIMARY KEY` column to every migration (line 683). However, no validation prevents a user from declaring a field with `name: id` in their entity definition. When such a field exists: (a) `generateModels` emits two struct fields both named `ID` (via `toPascalCase("id")` â†’ `"ID"`) â€” the generated Go code fails to compile due to duplicate field names; (b) `generateMigration` emits two `"id"` column definitions â€” PostgreSQL rejects the duplicate column; (c) `emitCreateMethod`/`emitReadMethod` reference `v.ID` ambiguously. The existing `validateFieldUniqueness` check (line 1054) only validates among declared fields, not against the implicit `id`. The generator should reject any entity that declares a field named `id` (case-insensitive, since `toPascalCase("ID")` and `toPascalCase("Id")` both produce `"ID"`). Ref: `generator.go:124` (implicit ID in model), `generator.go:683` (implicit id in DDL), `generator.go:1054-1072` (validateFieldUniqueness â€” does not check implicit id).

- [ ] **`sqlQ` does not escape double quotes within identifiers, producing malformed SQL for field names containing `"`.** The `sqlQ` helper (line 798â€“800) wraps a SQL identifier in double quotes via simple concatenation (`"\"" + id + "\""`) but does not escape embedded double quotes in the identifier string. PostgreSQL standard requires doubling embedded double quotes: the identifier `my"field` must be rendered as `"my""field"`. While uncommon, field names are user-supplied (spec Â§Entity Type System) and no validation prevents a double-quote character. The same issue affects `sqlQuotedColList` (line 804) and every call site that uses `sqlQ`. Either `sqlQ` should double embedded quotes (`strings.ReplaceAll(id, "\"", "\"\"")`) or the generator should validate field/entity names against a safe character set. Ref: `generator.go:798-800` (`sqlQ`), `generator.go:804-809` (`sqlQuotedColList`), PostgreSQL Â§4.1.1 (quoted identifiers).

- [ ] **No validation of field name character set, producing opaque `format.Source` errors for names that are not valid Go identifiers.** Field names are user-supplied (spec Â§Entity Type System) and the generator converts them to Go struct field names via `toPascalCase` (line 896â€“906).
