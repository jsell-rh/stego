# Review: Task 010 — postgres-adapter Component Generator

## Round 1

- [x] [process-revision-complete] **tableName pluralization produces invalid table names.** `tableName` appends `"s"` unconditionally (`toSnakeCase(name) + "s"`). For entities whose snake_case form already ends in 's' (e.g. `AdapterStatus` → `adapter_statuss`, `Address` → `addresss`), the generated table name is grammatically wrong and diverges from PostgreSQL conventions. The test `TestTableNameDerivation` codifies `"adapter_statuss"` as expected, but the generated SQL (`CREATE TABLE IF NOT EXISTS adapter_statuss`) and all query methods reference this malformed name. Ref: `generator.go:633-635`, `generator_test.go:788`.

- [x] [process-revision-complete] **Upsert: optimistic concurrency WHERE clause appended to DO NOTHING produces invalid SQL.** In `emitUpsertMethod`, the optimistic concurrency `WHERE` clause (line 464) is appended unconditionally after both the `DO UPDATE SET` and `DO NOTHING` branches. When all non-key writable columns are empty (all columns are upsert key columns, or the remaining non-key columns are all computed), `setClauses` is empty, the code emits `ON CONFLICT (...) DO NOTHING`, and then appends `WHERE EXCLUDED.generation > table.generation`. PostgreSQL does not support a `WHERE` clause on `DO NOTHING`; the generated query would fail at runtime. Ref: `generator.go:458-465`.

- [x] [process-revision-complete] **SQL string literals not escaped in migration DDL.** `columnDefinition` interpolates field values into SQL with single-quote wrapping but no escaping of embedded single quotes: (a) `f.Pattern` at line 579 (`CHECK (col ~ 'pattern')`), (b) enum `f.Values` entries at line 565 (`CHECK (col IN ('val1', 'val2'))`), (c) string defaults in `sqlDefault` at line 597 (`DEFAULT 'value'`). A pattern like `^[a-z']`, an enum value containing `'`, or a default like `it's` would produce syntactically broken SQL. Standard SQL escaping requires doubling single quotes (`''`). Ref: `generator.go:565,579,597`.

- [x] [process-revision-complete] **`unique_composite` constraint attribute silently dropped.** The spec defines `unique_composite: [field1, field2]` as a constraint attribute (spec §Entity Type System). The `types.Field` struct carries `UniqueComposite []string` (types.go:69). However, `columnDefinition` never checks `f.UniqueComposite`, and `generateMigration` never emits a table-level `UNIQUE (field1, field2)` constraint. If an entity declares `unique_composite`, the constraint is silently ignored in the generated migration. Ref: `generator.go:542-591`, spec "Constraint attributes" list.

## Round 2

- [x] [process-revision-complete] **Generated model structs use non-pointer types for nullable columns, causing runtime scan failures.** Optional fields (`optional: true`) and computed fields (`computed: true`) produce nullable columns in the migration DDL (no `NOT NULL`). However, `generateModels` maps all fields to non-pointer Go types (e.g., `string`, `int32`, `time.Time`) regardless of nullability. When `database/sql.Rows.Scan` encounters a SQL NULL value for these columns, it cannot convert NULL into a non-pointer type and returns a runtime error like `"converting NULL to string is unsupported"`. Affected types: `string`, `int32`, `int64`, `float32`, `float64`, `bool`, `time.Time`. (`[]byte` and `json.RawMessage` handle nil correctly.) The generated Read and List methods scan directly into struct fields (`&v.FieldName`), so any entity with optional or computed fields of the affected types will fail at runtime when the column is NULL. Ref: `generator.go:112-116` (model field emission), `generator.go:237-240` (Read scan args), `generator.go:338-340` (List scan args).

- [x] [process-revision-complete] **`emitUpdateMethod` produces invalid SQL when an entity has only computed fields.** If every field in an entity is `computed: true`, then `writeColumns(e)` returns an empty slice. `emitUpdateMethod` builds `setClauses` from `writeCols`, producing an empty slice. `strings.Join([]string{}, ", ")` returns `""`, so the generated SQL becomes `UPDATE table_name SET  WHERE id = $1` — a syntax error. The same edge case also affects `emitCreateMethod` (the INSERT would be `INSERT INTO table (id) VALUES ($1)`, which is valid) but `emitUpdateMethod` has no guard against the empty-SET-clause case. Ref: `generator.go:279-288`.

- [x] [process-revision-complete] **`toSnakeCase` mishandles consecutive uppercase letters (acronyms).** The function inserts an underscore before every uppercase rune unconditionally (`if i > 0 && unicode.IsUpper(r)`). For entity names containing acronyms — e.g., `HTTPServer`, `APIKey`, `UserID` — this produces `h_t_t_p_server`, `a_p_i_key`, `user_i_d` instead of the expected `http_server`, `api_key`, `user_id`. These malformed snake_case names propagate to table names, column references, and all generated SQL. While no current spec examples use acronym entity names, entity names are user-supplied and the spec imposes no restriction against acronyms. Ref: `generator.go:673-682`.

- [x] [process-revision-complete] **`unique_composite` field names not validated against entity fields, and duplicate constraints emitted.** The `generateMigration` function (lines 532-536) iterates all fields and emits `UNIQUE (f.UniqueComposite...)` for each field whose `UniqueComposite` slice is non-empty. Two problems: (a) the field names in `UniqueComposite` are interpolated directly into the SQL DDL without verifying they correspond to actual fields in the entity — a typo like `unique_composite: [usr_id, org_id]` would produce a `UNIQUE` constraint on a non-existent column, failing at migration time with no stego-level error; (b) if two fields in the same entity both declare `unique_composite: [field_a, field_b]`, the same `UNIQUE (field_a, field_b)` constraint is emitted twice, which PostgreSQL rejects as a duplicate constraint error. Ref: `generator.go:532-536`, spec §Entity Type System `unique_composite: [field1, field2]`.

## Round 3

- [x] [process-revision-complete] **`ref` type `to` attribute silently ignored — no FOREIGN KEY constraint generated.** The spec's Entity Type System defines `ref` as a stego-specific type with a `to` attribute pointing to another entity (e.g., `{name: org_id, type: ref, to: Organization}`). The spec states: "The component decides HOW to enforce them (SQL constraints, middleware, ORM annotations)." For a PostgreSQL adapter, the standard enforcement is a `REFERENCES` clause. Currently, `columnDefinition` maps `ref` to plain `TEXT` and never inspects `Field.To`. The `generateMigration` function never emits `REFERENCES` or `FOREIGN KEY` clauses. The `to` attribute is dead metadata — referential integrity is not enforced at the database level. Ref: `generator.go:614-662` (`columnDefinition`), `generator.go:574-609` (`generateMigration`), spec §Entity Type System (`ref` type), spec §Entity Type System enforcement principle.

- [x] [process-revision-complete] **No validation that `ref` field's `to` attribute references an existing entity.** The generator validates entity name uniqueness (`validateEntityUniqueness`), case-insensitive collisions (`validateCaseInsensitiveUniqueness`), reserved names (`validateReservedNames`), and `unique_composite` field references (`validateUniqueComposites`). However, `ref` fields' `to` values are never checked against the entity list. A typo like `to: Organziation` produces no error — the generator silently generates a plain TEXT column. Since `to` is already ignored (finding above), the missing validation compounds the issue: the user gets no feedback that their ref target is wrong. Ref: `generator.go:25-43` (validation calls in `Generate`), `types.go:75` (`To` field on `Field` struct).

- [x] [process-revision-complete] **Migration SQL file uses non-standard header instead of spec-mandated text.** The spec's first enforcement rule states: "All generated files carry a `// Code generated by stego. DO NOT EDIT.` header." The migration file (`001_initial.sql`) emits `-- Generated by stego postgres-adapter component.` (line 578) instead of the spec-mandated text. Since the `gen.File.Bytes()` method only prepends headers for `.go` files, the postgres-adapter manually writes its own SQL header — but uses different text than what the spec requires. The SQL-comment equivalent should be `-- Code generated by stego. DO NOT EDIT.` Ref: `generator.go:577-578`, spec §Generated Code Structure rule 1, `gen.go:61-67` (`File.Bytes`).

## Round 4

- [x] [process-revision-complete] **Migration SQL emits FOREIGN KEY references before referenced tables exist.** `generateMigration` emits `CREATE TABLE` statements in the order entities appear in `ctx.Entities`. When entity A has a `ref` field pointing to entity B, the generator emits `REFERENCES B_table(id)` inline. If A appears before B in the entity list, PostgreSQL rejects the migration because the referenced table does not yet exist. Example: `basicContext()` in tests defines `User` (with `org_id ref to Organization`) before `Organization`. The generated SQL is `CREATE TABLE IF NOT EXISTS users (... org_id TEXT NOT NULL REFERENCES organizations(id)); CREATE TABLE IF NOT EXISTS organizations (...)` — this fails at execution time with `relation "organizations" does not exist`. The generator needs to topologically sort entities by ref dependencies before emitting migration DDL, or use deferred foreign key constraints. Ref: `generator.go:584-613` (entity iteration order), `generator.go:597-599` (REFERENCES emission), `generator_test.go:14-40` (`basicContext` entity ordering).

- [x] [process-revision-complete] **`ref` field with empty `to` attribute silently generates a plain TEXT column with no error.** Both `validateRefTargets` (line 908) and the REFERENCES emission in `generateMigration` (line 597) guard on `f.To != ""`. A `ref` field with an omitted or empty `to` attribute passes all validation and produces a bare `TEXT` column with no foreign key constraint. The `ref` type's defining characteristic per the spec is its `to` attribute (e.g., `{name: org_id, type: ref, to: Organization}`). A ref without a target is semantically invalid — the generator should return a compile error rather than silently degrading to a plain TEXT column. Ref: `generator.go:597` (migration guard), `generator.go:908` (validation guard), spec §Entity Type System (`ref` type definition).

## Round 5

- [x] [process-revision-complete] **Optimistic concurrency `WHERE` clause references hardcoded `generation` column without validating the entity has such a field.** `emitUpsertMethod` unconditionally emits `WHERE EXCLUDED.generation > table.generation` for every entity when `concurrency == "optimistic"` (line 482–484). The generator has full knowledge of each entity's fields at generation time, but never checks whether a `generation` field exists. If the Upsert method is called at runtime with `concurrency: "optimistic"` for an entity that lacks a `generation` field (e.g., an entity with only `label` and `category`), the generated SQL references a non-existent column and PostgreSQL returns a runtime error. The generator should either (a) only emit the optimistic concurrency branch for entities that have a `generation` field, or (b) emit a runtime guard that returns an error when optimistic concurrency is requested for an entity without a `generation` column. Ref: `generator.go:482-484`, spec §Entity & Operation Features (`concurrency: optimistic`).

- [x] [process-revision-complete] **No validation of field name uniqueness within an entity.** The generator validates entity name uniqueness (`validateEntityUniqueness`), case-insensitive entity name collisions (`validateCaseInsensitiveUniqueness`), and reserved entity names (`validateReservedNames`). However, field names within an entity are never checked for duplicates. If two fields share the same name (e.g., two fields both named `email`), the generator produces: (a) duplicate Go struct fields — the model struct has two `Email` fields, causing a Go compilation error; (b) duplicate SQL column definitions — the migration has two `email` columns, causing a PostgreSQL error. The generator should validate field name uniqueness per entity and return a compile error. Ref: `generator.go:25-48` (validation calls), `generator.go:117-131` (model struct emission), `generator.go:660-672` (migration column emission).

- [x] [process-revision-complete] **SQL identifiers derived from field names are not quoted, producing invalid SQL when names collide with PostgreSQL reserved words.** Field names from the entity definition are interpolated directly into DDL (`columnDefinition`, line 694–742) and DML (`emitCreateMethod`, `emitReadMethod`, `emitUpdateMethod`, `emitDeleteMethod`, `emitListMethod`, `emitUpsertMethod`) without quoting. PostgreSQL reserved words used as field names — e.g., `order`, `check`, `default`, `group`, `table`, `select`, `primary` — produce syntactically invalid SQL. For example, a field named `order` generates `order INTEGER NOT NULL` in the migration, which PostgreSQL rejects because `ORDER` is a reserved keyword. The standard PostgreSQL mitigation is to double-quote identifiers (`"order" INTEGER NOT NULL`). The generator should either double-quote all column identifiers in generated SQL, or validate field names against PostgreSQL reserved words at generation time. Ref: `generator.go:694-742` (columnDefinition), `generator.go:190-233` (emitCreateMethod), `generator.go:238-270` (emitReadMethod), spec §Entity Type System (field names are user-supplied, no restriction against reserved words).

## Round 6

- [-] [process-revision-complete] **User-declared field named `id` collides with the implicit ID primary key, producing uncompilable Go code and invalid SQL.** The generator unconditionally adds an `ID string` field to every model struct (line 124) and an `"id" TEXT PRIMARY KEY` column to every migration (line 683). However, no validation prevents a user from declaring a field with `name: id` in their entity definition. When such a field exists: (a) `generateModels` emits two struct fields both named `ID` (via `toPascalCase("id")` → `"ID"`) — the generated Go code fails to compile due to duplicate field names; (b) `generateMigration` emits two `"id"` column definitions — PostgreSQL rejects the duplicate column; (c) `emitCreateMethod`/`emitReadMethod` reference `v.ID` ambiguously. The existing `validateFieldUniqueness` check (line 1054) only validates among declared fields, not against the implicit `id`. The generator should reject any entity that declares a field named `id` (case-insensitive, since `toPascalCase("ID")` and `toPascalCase("Id")` both produce `"ID"`). Ref: `generator.go:124` (implicit ID in model), `generator.go:683` (implicit id in DDL), `generator.go:1054-1072` (validateFieldUniqueness — does not check implicit id).

- [-] [process-revision-complete] **`sqlQ` does not escape double quotes within identifiers, producing malformed SQL for field names containing `"`.** The `sqlQ` helper (line 798–800) wraps a SQL identifier in double quotes via simple concatenation (`"\"" + id + "\""`) but does not escape embedded double quotes in the identifier string. PostgreSQL standard requires doubling embedded double quotes: the identifier `my"field` must be rendered as `"my""field"`. While uncommon, field names are user-supplied (spec §Entity Type System) and no validation prevents a double-quote character. A field named `col"x` would produce `"col"x"` in all generated DDL and DML — syntactically broken SQL that fails at migration or query time. The same issue affects `sqlQuotedColList` (line 804) and every call site that uses `sqlQ`. Either `sqlQ` should double embedded quotes (`strings.ReplaceAll(id, "\"", "\"\"")`) or the generator should validate field/entity names against a safe character set. Ref: `generator.go:798-800` (`sqlQ`), `generator.go:804-809` (`sqlQuotedColList`), PostgreSQL §4.1.1 (quoted identifiers).

- [-] [process-revision-complete] **No validation of field name character set, producing opaque `format.Source` errors for names that are not valid Go identifiers.** Field names are user-supplied (spec §Entity Type System) and the generator converts them to Go struct field names via `toPascalCase` (line 896–906). However, `toPascalCase` only splits on underscores and title-cases segments — it does not validate or sanitize the input. Field names containing hyphens (e.g., `foo-bar` → `Foo-bar`), leading digits (e.g., `123` → `123`), spaces, or other non-identifier characters produce Go struct fields that are not valid identifiers. The failure surfaces only at `format.Source` (line 139) as an opaque formatting error with no indication of which field name is problematic. The generator should validate that each field name matches a safe identifier pattern (e.g., `^[a-zA-Z_][a-zA-Z0-9_]*$`) during the validation phase and return a clear error naming the offending field. Ref: `generator.go:896-906` (`toPascalCase`), `generator.go:139` (`format.Source` error), spec §Entity Type System (field names are user-supplied).

## Round 7

- [-] [process-revision-complete] **No entity name charset validation, producing opaque `format.Source` errors for names that are not valid Go identifiers.** The generator validates field names against `^[a-zA-Z_][a-zA-Z0-9_]*$` via `validateFieldNameCharset` (lines 1092–1108), but has no equivalent validation for entity names. Entity names are user-supplied (spec §Entity Type System) and are used directly as Go struct type names (line 138: `fmt.Fprintf(&buf, "type %s struct {\n", e.Name)`) and variable types (line 224: `var v %s`). An entity named `my-entity`, `123Invalid`, or `hello world` would produce `type my-entity struct {...}` or `type 123Invalid struct {...}` — invalid Go — and fail at `format.Source` with an opaque formatting error. The generator should validate entity names against the same safe identifier pattern used for field names. Ref: `generator.go:138` (entity name as Go type), `generator.go:224` (entity name as variable type), `generator.go:1092-1108` (`validateFieldNameCharset` — applies to fields only, not entities).

- [-] [process-revision-complete] **`reservedTypeNames` does not guard against Go keywords or predeclared identifiers, producing uncompilable generated code.** The `reservedTypeNames` set (line 101–104) only contains `Store` and `NewStore`. Entity names that match Go keywords (`func`, `return`, `type`, `struct`, `if`, `for`, `switch`, `case`, `package`, `import`, `var`, `const`, `map`, `chan`, `range`, `go`, `defer`, `select`, `interface`, `break`, `continue`, `fallthrough`, `goto`, `default`) produce invalid Go syntax when used as type names (`type func struct {...}` — parse error). Entity names that match Go predeclared identifiers (`error`, `any`, `string`, `int`, `int32`, `int64`, `float32`, `float64`, `bool`, `byte`, `rune`, `true`, `false`, `nil`) shadow those identifiers within the generated package. Since the generated `store.go` method signatures use `any`, `error`, and `string` as built-in types (e.g., `func (s *Store) Create(entity string, value any) error`), an entity named `error` or `any` would cause the method signatures to reference the entity struct instead of the built-in type, producing compilation failures. Ref: `generator.go:101-104` (`reservedTypeNames`), `generator.go:211` (Create method signature uses `any` and `error`), `generator.go:260` (Read method uses `string` and `error`).

- [-] [process-revision-complete] **Table name collision detection compares lowercased entity names, not derived table names.** `validateCaseInsensitiveUniqueness` (lines 1005–1023) compares `strings.ToLower(e.Name)` to detect entity name collisions. However, the actual collision risk is in the derived table name produced by `tableName()` (which applies `toSnakeCase` + `pluralize`). Two entity names that differ in their lowercased form can still produce identical table names. Example: `UserProfile` and `User_Profile` — `strings.ToLower("UserProfile")` = `"userprofile"`, `strings.ToLower("User_Profile")` = `"user_profile"` — these are different, so validation passes. But `tableName("UserProfile")` and `tableName("User_Profile")` both produce `"user_profiles"`. The generated migration would emit two `CREATE TABLE IF NOT EXISTS "user_profiles"` statements — the second silently succeeds due to `IF NOT EXISTS` but has different columns. Both entities' CRUD methods would target the same table with mismatched column sets, causing runtime SQL errors. The validation should compare `tableName(e.Name)` values instead of `strings.ToLower(e.Name)` values. Ref: `generator.go:1005-1023` (`validateCaseInsensitiveUniqueness`), `generator.go:861-863` (`tableName`), `generator.go:889-908` (`toSnakeCase` — underscore in input has no effect on output).

## Round 8

- [-] [process-revision-complete] **No validation of derived Go struct field name (PascalCase) uniqueness within an entity, producing uncompilable generated code.** `validateFieldUniqueness` (lines 1098–1116) checks raw field names for duplicates, but the generated Go struct uses `toPascalCase(f.Name)` for field names (line 169). The field name charset validation (`^[a-zA-Z_][a-zA-Z0-9_]*$`) allows uppercase letters and leading underscores, permitting multiple raw field names that map to the same PascalCase identifier. Examples: (a) `_name` and `name` — `toPascalCase("_name")` splits on `_` into `["", "name"]`, skips the empty segment, produces `"Name"`; `toPascalCase("name")` also produces `"Name"`. (b) `foo_bar` and `fooBar` — `toPascalCase("foo_bar")` produces `"FooBar"`; `toPascalCase("fooBar")` also produces `"FooBar"`. In both cases, the raw names differ (passing `validateFieldUniqueness`) but the generated struct has two fields with the same Go name, causing `format.Source` to fail with an opaque error. The generator should validate `toPascalCase(f.Name)` uniqueness within each entity, not just raw name uniqueness. Ref: `generator.go:169` (`toPascalCase` in model generation), `generator.go:1098-1116` (`validateFieldUniqueness` — checks raw names only), `generator.go:940-949` (`toPascalCase` — drops leading underscores, doesn't distinguish `fooBar` from `foo_bar`).

- [-] [process-revision-complete] **`enum` fields with empty `Values` silently degrade to unconstrained TEXT columns with no validation error.** The spec defines `enum` as a stego-specific type with `values: [a, b, c]` (spec §Entity Type System). An enum's defining characteristic is its finite set of allowed values. The generator's `columnDefinition` (line 784) guards the CHECK constraint emission with `len(f.Values) > 0`, so an enum field with an empty or omitted `values` attribute silently produces a plain `TEXT` column with no CHECK constraint — indistinguishable from a `string` field. No validation catches this: neither `validateFieldNameCharset`, `validateFieldUniqueness`, nor any other validator checks that enum fields have at least one value. The generator should validate that every field with `type: enum` has a non-empty `Values` slice and return a compile error otherwise. Ref: `generator.go:784` (CHECK constraint guard), `generator.go:29-73` (validation calls — no enum values check), spec §Entity Type System (`values: [a, b, c]` for enums).

## Round 9

- [-] [process-revision-complete] **Duplicate field names within a single `unique_composite` constraint not validated, producing SQL that PostgreSQL rejects.** `validateUniqueComposites` (lines 609–651) validates that each referenced field name exists in the entity, but does not check for duplicate entries within a single constraint's field list. A declaration like `unique_composite: [user_id, user_id]` passes validation and produces `UNIQUE ("user_id", "user_id")` in the migration DDL. PostgreSQL rejects duplicate columns in UNIQUE constraints with `ERROR: column "user_id" appears twice in unique constraint`. The deduplication logic (lines 634–648) only prevents the same constraint from being emitted *multiple times across fields* — it does not detect duplicates *within* a single constraint. The generator should validate that each `unique_composite` slice contains no duplicate field names and return a compile error otherwise. Ref: `generator.go:609-651` (`validateUniqueComposites` — no intra-constraint duplicate check), `generator.go:753-758` (constraint emission).

- [-] [process-revision-complete] **Duplicate enum values not validated, producing semantically invalid CHECK constraints.** `validateEnumValues` (lines 1216–1232) checks that enum fields have at least one value (`len(f.Values) == 0`) but does not check for duplicate entries within the `Values` slice. A declaration like `values: [admin, admin, member]` passes validation and produces `CHECK ("role" IN ('admin', 'admin', 'member'))` — valid SQL but semantically wrong (the duplicate implies a configuration error, analogous to duplicate field names or duplicate entity names). The spec defines enum values as a finite set (`values: [a, b, c]`, spec §Entity Type System), which implies uniqueness. The generator validates uniqueness for entity names, field names, derived field names, and unique_composite constraints, but not for enum values. The generator should validate that enum `Values` are unique and return a compile error for duplicates. Ref: `generator.go:1216-1232` (`validateEnumValues` — only checks empty, not duplicates), `generator.go:794-799` (CHECK constraint emission with all values including duplicates).

## Round 10

- [-] [process-revision-complete] **`toPascalCase` produces invalid Go identifiers from underscore-only and underscore-prefixed-digit field names, causing opaque `format.Source` errors.** The field name charset validation (`^[a-zA-Z_][a-zA-Z0-9_]*$`, line 21) accepts names consisting solely of underscores (e.g., `_`, `__`) and underscore-prefixed digit-only names (e.g., `_123`, `_42`). The `toPascalCase` function (lines 957–967) splits on underscores and skips empty segments, producing: (a) empty string for underscore-only names (`toPascalCase("_")` → `""`, `toPascalCase("__")` → `""`), and (b) digit-leading strings for underscore-prefixed digit names (`toPascalCase("_123")` → `"123"`). Both are invalid Go identifiers. In `generateModels`, an empty `goName` produces an anonymous/embedded field declaration (`string \`json:"_"\``) — syntactically valid but semantically wrong. In store methods (e.g., `emitCreateMethod` line 281), `"v."+toPascalCase("_")` produces `"v."` — a syntax error that `format.Source` rejects with an opaque formatting error giving no indication which field name is problematic. A digit-leading name like `"123"` similarly produces `123 string` in the struct — invalid Go. No existing validation catches this: `validateFieldNameCharset` accepts these names, `validateDerivedFieldUniqueness` checks uniqueness but not validity, and `validateNoImplicitIDCollision` only checks for `"ID"`. The generator should validate that `toPascalCase(f.Name)` produces a non-empty string that is a valid Go identifier (starts with a letter or underscore), or tighten the charset regex to require at least one alphabetic character. Ref: `generator.go:957-967` (`toPascalCase`), `generator.go:21` (charset regex), `generator.go:179` (model field emission), `generator.go:281` (store scan arg).

## Round 11

- [ ] **`reservedTypeNames` does not include import aliases used in generated code, producing uncompilable output for entities named `sql`, `json`, `fmt`, or `strings`.** The generated `store.go` unconditionally imports `database/sql`, `encoding/json`, `fmt`, and `strings` (lines 215–219). The generated `models.go` conditionally imports `encoding/json` and `time` (lines 168–177). `reservedTypeNames` (lines 122–147) includes Go keywords, predeclared identifiers, and generator-internal names (`Store`, `NewStore`) but does not include the import aliases `sql`, `json`, `fmt`, `strings`, or `time`. An entity named `sql` produces `type sql struct { ... }` in both models.go and store.go. In store.go, this shadows the `database/sql` import alias, causing `type Store struct { db *sql.DB }` (line 223) to resolve `sql` to the entity struct instead of the `database/sql` package — a compile error. The same applies to `json` (shadows `encoding/json`, breaking `json.Marshal`/`json.Unmarshal` calls), `fmt` (shadows `fmt`, breaking `fmt.Errorf` calls), `strings` (shadows `strings`, breaking `strings.Join` calls), and `time` (shadows `time` when timestamp fields exist, breaking `time.Time` references in models). The generator should add these import aliases to `reservedTypeNames`. Ref: `generator.go:122-147` (`reservedTypeNames`), `generator.go:215-219` (store.go imports), `generator.go:168-177` (models.go conditional imports).

- [ ] **Entity names consisting solely of underscores (e.g. `_`, `__`) pass all validation but produce uncompilable generated code.** The entity name charset validation (`^[a-zA-Z_][a-zA-Z0-9_]*$`, line 21) accepts `_` and `__`. These are not in `reservedTypeNames`. The generator produces `type _ struct { ... }` — valid Go syntax (the blank identifier discards the type name). However, the store methods use the entity name as a type: `var v _` in `emitCreateMethod` (line 267), `emitReadMethod` (line 311), `emitUpdateMethod` (line 359), `emitListMethod` (line 458), and `emitUpsertMethod` (line 496). Since `_` as a type reference is the blank identifier and not a named type, these declarations fail during compilation. Unlike field names — which have `validateDerivedFieldValidity` (lines 1208–1225) to catch post-transformation validity issues — entity names have no equivalent check. The generator should validate that entity names are not the blank identifier `_`, or more broadly, that they produce valid, usable Go type names.
