# Review: Task 006 — Generator Interface & gen.Context

## Findings

- [ ] **`Generator.Generate` does not return `Wiring`.** The spec states: *"Each component's generator returns a `Wiring` struct (imports, constructors, routes) alongside its files, and the compiler assembles the shared files from all wiring declarations."* The task description explicitly lists: *"Wiring struct — imports, constructors, routes that each component returns for shared file assembly."* The `Wiring` type is defined (`gen.go:58-69`) but is not part of the `Generator` interface's return signature — `Generate` returns `([]File, error)`. A component implementing `Generator` has no mechanism to communicate its imports, constructors, or routes to the compiler. The `Wiring` struct is dead infrastructure: defined but unreachable through the interface contract. The fix is to include `Wiring` in the return: either `Generate(ctx Context) ([]File, *Wiring, error)` or a composite return type.

- [ ] **Generated file header is opt-in, not enforced.** The acceptance criteria requires *"Generated file header enforcement."* The spec states: *"All generated files carry a `// Code generated by stego. DO NOT EDIT.` header."* The word "enforcement" implies it should be impossible (or at least detectable) when a file lacks the header. Currently, `File.Content` is a public field that does not include the header, and `WithHeader()` (`gen.go:52-54`) is a convenience method that callers may freely skip. Nothing prevents a generator from producing files that are written via `f.Content` without the header. Compare with namespace validation: `ValidateNamespace` actively rejects violations. An analogous `ValidateHeaders([]File) error` (or making the header part of the `File` constructor so it cannot be omitted) would satisfy the "enforcement" criterion. Without it, header presence is a convention, not an invariant.

- [ ] **`Component` type has no `OutputNamespace` field; the declaration half of namespace enforcement is missing.** The task description says: *"Output namespace enforcement: each component declares `output_namespace`, generator can only write files under it."* The spec says: *"Each component declares an `output_namespace` (e.g. `internal/api`) and can only write files under it."* `ValidateNamespace` (`gen.go:74-99`) handles the *enforcement* half — given a namespace string, it rejects files outside it. But nothing allows a component to *declare* its namespace. The `Component` struct (`types.go:198-206`) has no `OutputNamespace` field, and `gen.Context` (`gen.go:23-39`) does not carry the component's declared namespace either. The validator takes namespace as a bare `string` parameter, so the caller must supply it from an unknown source. The declaration must exist somewhere — either as a field on `Component` (since the spec says the component declares it) or on `gen.Context` (so the generator/compiler can access it). Without it, namespace enforcement cannot be wired end-to-end.
