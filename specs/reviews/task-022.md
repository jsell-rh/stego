# Review: Task 022 — GORM-Based postgres-adapter

## Round 1

- [x] [process-revision-complete] **DAO method naming mismatch**: AC #3 requires methods named `Get` and `Replace` (matching the spec at rest-crud/spec.md:328-329: `Get(ctx, id)`, `Replace(ctx, entity)`). The implementation provides `Read` and `Update` instead (`generator.go:419`, `generator.go:439`). Rename `Read` to `Get` and `Update` to `Replace` to match the spec's DAO contract. *Process fix: expanded checklist item 4 to explicitly cover method/function names and parameter lists on generated interfaces, not just type names. Added checklist item 165 (generated interface contracts must exactly match spec-defined method signatures — names, parameters, AND behavioral semantics).*

- [x] [process-revision-complete] **DAO methods missing `context.Context` parameter**: The spec defines every DAO method with `ctx` as the first parameter (rest-crud/spec.md:327-333: `Create(ctx, entity)`, `Get(ctx, id)`, `Replace(ctx, entity)`, `Delete(ctx, id)`, `List(ctx, listArgs)`, `Upsert(ctx, entity, upsertKey, concurrency)`, `Exists(ctx, id)`). The implementation omits `context.Context` from all seven Store methods. Context is required for GORM's `db.WithContext(ctx)` and for OpenTelemetry trace propagation (the archetype includes `otel-tracing`). *Process fix: checklist item 4 now explicitly requires verifying all specified parameters are present. Checklist item 165 makes parameter completeness a first-class verification step for generated interface contracts.*

- [x] [process-revision-complete] **List method lacks pagination**: Both the spec (rest-crud/spec.md:331) and the task description (task-022.md:33) specify `List(ctx, listArgs) — g2.Offset(offset).Limit(limit).Find(&list) with pagination`. The implementation at `generator.go:501-539` does not support offset or limit — it calls `query.Find(&result)` without pagination parameters. *Process fix: checklist item 165 now requires verifying behavioral semantics described alongside method signatures, not just the method name and parameter list. Pagination specified in a method's description is a behavioral contract, not a suggestion.*

- [x] [process-revision-complete] **GORM DB connection never closed**: `writeGORMDBSetup` (`assembler.go:425-434`) does not close the database connection. The raw SQL equivalent (`writeDBSetup` at `assembler.go:413-423`) has `defer db.Close()`. The GORM version should retrieve the underlying `*sql.DB` via `db.DB()` and defer its close. *Process fix: added checklist item 166 (generated resource acquisition must have corresponding lifecycle management). Requires verifying every Open/Dial/Connect has a corresponding Close/Shutdown, and specifically requires diffing parallel code paths for the same resource type.*

- [x] [process-revision-complete] **Migrations not wired to run at startup**: The spec states "GORM AutoMigrate at startup" (rest-crud/spec.md:337). The generator produces `Register`, `Migrate`, and `init()` registration infrastructure (`generator.go:654-717`), but nothing in the wiring or assembler calls `Migrate(db)` in the generated `main.go`. The migration infrastructure is generated but never invoked at runtime. The wiring (`generator.go:124-134`) should include a mechanism (e.g. a pre-constructor call or a new Wiring field) to ensure `storage.Migrate(db)` is called after DB setup and before constructors. *Process fix: added checklist item 167 (generated infrastructure must be invoked in the application lifecycle). Registration without invocation is dead code — the generator must produce BOTH the registration and the invocation, and wire them together at the correct lifecycle point.*

## Round 2

- [x] [process-revision-complete] **Optimistic concurrency WHERE clause uses a table name that may diverge from GORM's naming**: The generated upsert code at `generator.go:614-617` produces a SQL expression `EXCLUDED."generation" > "adapter_statuses"."generation"` where the table name is computed by the generator's own `tableName`→`pluralize` function (`generator.go:841-861`). GORM's default naming strategy uses `jinzhu/inflection` for pluralization, which handles irregular English plurals (Person→people, Child→children, Goose→geese). The generator's `pluralize` function does not — it would produce "persons", "childs", "gooses". If an entity has an irregular plural form, the ON CONFLICT WHERE clause references a table name that doesn't match the actual GORM-created table, causing a runtime SQL error. The table qualifier is unnecessary in PostgreSQL's `ON CONFLICT ... DO UPDATE SET ... WHERE` context — the unqualified column name `"generation"` correctly refers to the existing row. Remove the table name qualifier: `EXCLUDED."generation" > "generation"`. *Process fix: added checklist item 168 (framework-derived identifiers must not be re-implemented by the generator). When the generated code depends on matching a framework-derived identifier, the generator must either use the framework's own derivation function, override the framework's naming, or avoid the reference entirely. Re-implementing framework-internal logic with a different algorithm guarantees divergence on edge cases.*

- [x] [process-revision-complete] **PostDBCalls expressions not subject to import alias disambiguation**: The assembler applies import alias renames to constructor expressions (`writeConstructors` at `assembler.go:220` receives `imports.ImportRenames` and calls `replaceAliasInExpr`) and to route expressions. However, `writePostDBCalls` at `assembler.go:446-457` emits each `PostDBCalls` entry verbatim without applying any import renames. If an import alias collision causes the postgres-adapter's `storage` alias to be renamed (e.g. to `storage2`), the PostDBCall `storage.Migrate(db)` would reference a non-existent alias, producing a compilation error in the generated `main.go`. The fix: pass `imports.ImportRenames` into `writePostDBCalls` and apply renames to each call expression, matching the logic already used for constructors. *Process fix: added checklist item 169 (new wiring expression categories must receive all existing expression-processing treatments). When adding a new expression field to wiring metadata, the implementer must audit every processing step applied to existing expression categories and apply the same treatments to the new category. The mandatory audit procedure: grep the assembler for every function that processes existing expression categories and add the new category to each applicable function.*

- [x] [process-revision-complete] **List handler parses `offset`/`limit` query parameters instead of spec-defined `page`/`size`**: The rest-api handler changes introduced in this task (`restapi/generator.go:596-599`) parse `offset` and `limit` directly from query parameters. The spec (rest-crud/spec.md:175-193) defines the list query parameters as `page` (1-indexed page number, default: 1) and `size` (items per page, default: 100, max: 65500), with the conversion `OFFSET (page-1)*size LIMIT size`. The handler should parse `page`/`size`, apply the default values, clamp `size` to 65500, convert to `offset`/`limit`, and pass those to `store.List`. The current implementation exposes raw database-level pagination parameters to the API consumer and omits the defaults and clamping the spec requires. *Process fix: added checklist item 170 (generated API handlers must use spec-defined external parameters, not internal implementation parameters). When the spec defines API-level query parameters with conversion logic, the handler must parse the spec-defined names, apply defaults and constraints, and convert internally — not expose storage-level parameters directly to the API consumer.*

## Round 3

- [x] [process-revision-complete] **Meta timestamp fields (`CreatedTime`, `UpdatedTime`) are never populated**: The spec (rest-crud/spec.md:286-287) says the postgres-adapter "Uses GORM as the ORM, following the proven rh-trex/hyperfleet-api pattern." GORM auto-manages timestamp fields only when they are conventionally named `CreatedAt`/`UpdatedAt`. The spec defines them as `CreatedTime`/`UpdatedTime` (rest-crud/spec.md:305-306), which are non-conventional names. The generated Meta struct at `postgresadapter/generator.go:226-229` does not include `gorm:"autoCreateTime"` or `gorm:"autoUpdateTime"` tags on these fields. Additionally, neither `emitCreateMethod` (generator.go:386-417) nor `emitReplaceMethod` (generator.go:442-482) sets these fields manually before the GORM call. As a result, every record created or updated via the generated store will have zero-value timestamps (`0001-01-01T00:00:00Z`). Either add `gorm:"autoCreateTime"` to `CreatedTime` and `gorm:"autoUpdateTime"` to `UpdatedTime` in the Meta struct, or set them explicitly in the store methods before the GORM write call. *Process fix: added checklist item 171 (framework auto-behaviors tied to naming conventions must be explicitly configured when spec names diverge). When a spec says "following the [framework] pattern" but defines field names that differ from the framework's conventions, the implementer must check every auto-behavior the framework provides and add explicit configuration where the framework's naming convention is not met.*

- [x] [process-revision-complete] **`unique_composite` constraint not reflected in generated GORM tags**: The spec (spec.md:100) defines `unique_composite: [field1, field2]` as a constraint attribute in the finite constraint set. The IR supports it via `types.Field.UniqueComposite` (types/types.go:75). The compiler validates it (validate.go:368-374). But `buildGormTag` (generator.go:305-336) only handles `f.Unique` (line 323-325) and completely ignores `f.UniqueComposite`. GORM supports composite unique indexes via `gorm:"uniqueIndex:idx_name"` applied to each participating field, where the shared index name groups them. Entities declaring `unique_composite` constraints (e.g., the `upsert_key` pattern from rest-crud/spec.md:76-77) will not have the uniqueness constraint enforced at the database level. Add handling for `f.UniqueComposite` in `buildGormTag` that emits `uniqueIndex:composite_<joined_field_names>` (or similar) on each field in the group. *Process fix: added checklist item 172 (attribute-to-annotation mapping functions must exhaustively cover ALL IR attributes in their domain). Requires opening the input type definition, listing every attribute, and verifying the mapping function has a code path for each in-domain attribute. Also requires diffing attribute coverage against any predecessor function when migrating backends (e.g., raw SQL → GORM).*

- [x] [process-revision-complete] **`min_length` constraint not reflected in generated GORM tags**: The spec (rest-crud/spec.md:320) explicitly states "GORM tags are derived from entity field constraints (unique, min/max length, optional, type)". The parenthetical lists "min/max length" — both min and max. `buildGormTag` (generator.go:305-336) handles `max_length` as `size:N` (line 327-329) but does not handle `min_length` at all, despite `types.Field.MinLength` being present in the IR (types/types.go:69). GORM supports CHECK constraints via the `check:` tag (e.g., `gorm:"check:length(column_name) >= 3"`), which maps directly to a PostgreSQL `CHECK` constraint enforced at the database level. Add handling for `f.MinLength` in `buildGormTag` that emits a `check:length(<column_name>) >= N` tag. *Process fix: checklist item 172 also covers this finding — the same `buildGormTag` mapping function that missed `UniqueComposite` also missed `MinLength`. The mandatory audit procedure (open the input type, list every attribute, verify a code path for each) would have caught both in a single pass. Additionally, the spec's explicit parenthetical "(unique, min/max length, optional, type)" enumerates the attributes — checklist item 35 requires cross-referencing the spec's attribute list against the generator's code paths.*

## Round 4

- [x] [process-revision-complete] **Replace method's `Save` overwrites computed fields with zero values**: The `emitReplaceMethod` (generator.go:452-493) zeroes computed fields (lines 479-485) then calls `s.db.WithContext(ctx).Save(&v)` (line 487). GORM's `Save` performs a full-column update of ALL columns, including the now-zeroed computed fields. This writes nil/zero values to computed field columns in the database, destroying any data previously populated by fills (e.g., a `status_conditions` field filled by `status-aggregator`). The task description (task-022.md:26) states "Computed fields included in model but excluded from create/update inputs." The spec (rest-crud/spec.md:323) states "computed fields are included in the model but excluded from create/update inputs." "Excluded from update inputs" means the Replace method should not modify computed columns at all — their existing values should be preserved, not overwritten with zeros. The `writeCols` slice is already computed at line 459 but not used to constrain the Save. Fix: replace `s.db.WithContext(ctx).Save(&v).Error` with a selective column update that only modifies writable (non-computed) columns, e.g. `s.db.WithContext(ctx).Model(&<Entity>{}).Where("id = ?", id).Select(writeCols).Updates(&v).Error`. This leaves computed columns untouched in the database. *Process fix: added checklist item 173 (computed restriction sets must actually constrain the framework API calls they govern). Requires verifying that every computed restriction set — writeCols, updateCols, selectFields — is consumed by the framework API call it was computed for. A restriction set with zero consumers is dead code whose purpose has been defeated. Specifically, GORM's `Save` is never the correct API for a selective update — it performs a full-column update regardless of any restriction set the method computed.*

- [x] [process-revision-complete] **Upsert method does not clear computed fields before INSERT**: The `emitCreateMethod` (generator.go:413-420) correctly zeroes all computed fields before calling `s.db.WithContext(ctx).Create(&v)`. The `emitUpsertMethod` (generator.go:562-644) does not — it unmarshals the request body into `v` (lines 574-581) and then calls `s.db.WithContext(ctx).Clauses(onConflict).Create(&v)` (line 638) without clearing computed fields first. On the INSERT path (no conflict), computed field values from the request body are written to the database rather than being excluded. The ON CONFLICT UPDATE path correctly restricts updates to `writeCols` (non-computed columns, line 610-611), but the INSERT path has no such restriction. This is inconsistent with `emitCreateMethod`'s behavior and with the spec requirement that computed fields are "excluded from create/update inputs" (rest-crud/spec.md:323, task-022.md:26). Fix: add the same computed-field zeroing loop from `emitCreateMethod` (lines 413-420) into `emitUpsertMethod`, after the json.Unmarshal and before the upsert key validation. *Process fix: added checklist item 174 (input-accepting persistence operations must share ALL data-sanitization steps — diff method bodies pairwise as a mandatory audit). Requires a mandatory pairwise diff of Create vs Upsert, Create vs Replace, and Upsert vs Replace, with every difference justified by a specific spec reason. Upsert is specifically flagged as the highest-risk operation because its unique complexity (ON CONFLICT, concurrency) dominates attention and crowds out shared semantics like computed field clearing that it inherits from Create.*

## Round 5

- [x] [process-revision-complete] **Store.List does not perform COUNT(\*) query or return total count for pagination**: The spec (rest-crud/spec.md:185) defines the list response `total` field as "total count of matching records across all pages." The spec's pagination mechanics (rest-crud/spec.md:189) explicitly state "Count total matching records first (`SELECT COUNT(*)`)." The task description (task-022.md:33) specifies "List(ctx, listArgs) — g2.Offset(offset).Limit(limit).Find(&list) with pagination." The generated `emitListMethod` (generator.go:517-561) returns `(any, error)` where `any` is only the paginated result slice — no `COUNT(*)` query is performed, and no total count is included in the return value. The corresponding rest-api Storage interface (restapi/generator.go:744) mirrors this signature: `List(...) (any, error)`. Without a total count from the store, the rest-api handler (restapi/generator.go:589-644) cannot construct the spec-required pagination envelope (`{kind, page, size, total, items}` — rest-crud/spec.md:163-171). The "with pagination" descriptor in both the spec and task description denotes the full pagination mechanics, not just OFFSET/LIMIT — the spec defines these as a two-step process: COUNT first, then FETCH. Fix: (a) change the generated Store.List to perform a scoped `db.Count(&total)` query before the paginated `Find`, and return both the results and total count (e.g., via a `ListResult` struct or an additional return value `(any, int64, error)`), and (b) update the rest-api Storage interface's List method signature to match, so the handler can include `total` in the response. *Process fix: checklist item 175 (generated query methods must implement ALL spec-defined query mechanics — not just the result-fetching step). Requires re-reading the spec's multi-step query description, verifying every step is implemented, and verifying the method's return type can carry all data the spec requires in the response. The return type is a contract — if it can't carry a total count, the total is architecturally impossible. Additionally, checklist item 25 (interface return types must match spec contracts) would have caught the return type deficiency: the spec says the list response includes `total`, but the Store.List return type has no slot for it.*

## Round 6

- [ ] [process-revision-complete] **List response `size` field reports requested page size, not actual item count**: The spec (rest-crud/spec.md:184) defines the list response `size` field as "the actual number of items returned (may be less than requested on the last page)." The generated list handler at `restapi/generator.go:642` emits `"size": size` where `size` is the parsed `?size=` query parameter (default 100, clamped to 65500). On the last page, when fewer items than the requested page size remain, the response incorrectly reports the full requested page size rather than the actual count of returned items. For example, with `total=103`, `page=2`, `size=100`, the response would report `"size": 100` while the `items` array contains only 3 entries. The spec's parenthetical "(may be less than requested on the last page)" makes this distinction explicit — `size` is NOT the requested page size, it is the actual count of items in the response. Fix: after the `store.List` call returns the items, compute the actual item count (e.g., via `reflect.ValueOf(<items>).Len()` since the items variable is typed as `any`, or by adding a typed length helper) and use that as the `size` value in the response envelope instead of the query parameter. *Process fix: checklist item 176 (response envelope fields must be populated from the actual response data, not from the request parameters). When a spec defines a response field that describes a property of the response payload (e.g., "actual number of items returned"), the value must be derived from the response, not echoed from the request. Request-echo is only correct when the spec says so (e.g., "the requested page number"). The spec's field descriptions are the source of truth for whether a value is request-derived or response-derived.*
