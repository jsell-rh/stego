# Task 014 Review — Validation & Drift Detection

## Round 1

- [x] [process-revision-complete] **Finding 1: `validateFieldTypes` does not check constraint-type applicability.** The spec's Entity Type System section (spec.md:207-228) annotates constraints to specific field types: "min_length, max_length, pattern (strings)" and "min, max (numerics)". The validate function checks that the field type itself is valid, and handles ref-specific (`to`) and enum-specific (`values`) attributes, but does not verify that constraint attributes are applied only to their designated field types. A declaration like `{name: count, type: int32, min_length: 5}` passes validation despite `min_length` being a string-only constraint per the spec. Similarly, `{name: label, type: string, values: [a, b]}` passes despite `values` being enum-only, and `{name: role, type: string, to: Widget}` is silently accepted despite `to` being ref-specific. The spec defines a finite, typed constraint set — the validate function enforces the field type but not the constraint-to-type mapping within that set. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 2: `validateFieldTypes` does not check `computed`/`filled_by` consistency.** The spec (spec.md:46-54) defines computed fields as the pairing `computed: true, filled_by: <fill>`. The validate function does not check that a field with `computed: true` has `filled_by` set, or that a field with `filled_by` set also has `computed: true`. A field `{name: status, type: jsonb, computed: true}` (missing `filled_by`) passes validation, as does `{name: status, type: jsonb, filled_by: aggregator}` (missing `computed: true`). The spec presents these as a pair: "Computed/derived fields are read-only fields populated by a fill, never written via the API" with the example always showing both attributes together. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 3: `validateFillsExist` silently ignores non-`os.IsNotExist` errors from `os.Stat`.** `validateFillsExist` (validate.go:310-313) checks `os.IsNotExist(err)` and only appends a validation error when that specific condition is true. When `os.Stat` returns a different error — such as permission denied (`EACCES`) — the condition is false and no error is appended, so the fill is silently treated as existing. This can produce a false positive: validation reports no fill errors when the function could not actually verify fill existence due to filesystem permission issues. In contrast, `DetectDrift` (drift.go:57-64) correctly handles non-`IsNotExist` read errors by returning them as infrastructure Go errors. The task acceptance criteria state "validate catches: ... missing fill."

A service binding to this mixin-added slot would receive a false positive "slot not defined by any component" validation error. The task description says "Validate slot bindings match available slots."

- [x] [process-revision-complete] **Finding 5: `unique_composite` field name references are not validated against existing fields within the entity.** The spec (spec.md:213) defines `unique_composite: [field1, field2]` as a constraint attribute where the list values reference other field names within the same entity. ## Round 4

- [x] [process-revision-complete] **Finding 8: Duplicate entity names are not detected by `validateFieldTypes` or `validateExposeReferences`.** Both functions build entity name lookup maps — `validateFieldTypes` at validate.go:184-187 and `validateExposeReferences` at validate.go:314-317 — by iterating `entities` and inserting into `map[string]bool`. When two entities share the same name, the second silently overwrites the first. No error is reported. Entities are referenced by name throughout the service declaration: ref field `to` targets (spec.md:226), expose block `entity` and `parent` references (spec.md:36-44), scope and upsert_key field lookups (spec.md:143, spec.md:23-30), and slot binding `entity` references (spec.md:348). Duplicate entity names make all of these references ambiguous — the lookup maps cannot distinguish which entity is intended. The spec's entity examples (spec.md:219-227, 326-332) always show distinct entity names, and generated code (Go struct names, SQL table names) requires unique names. A service.yaml with two `- name: Widget` entity entries passes validation. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 9: Duplicate field names within an entity are not detected by `validateFieldTypes`.** The function iterates each entity's fields at validate.go:190-306 without checking that field names are unique within the entity. Duplicate field names cannot be expressed in target-language representations: Go struct field names must be unique, SQL column names must be unique within a table, and OpenAPI schema property names must be unique. The spec's field examples (spec.md:219-227) always show distinct field names within each entity.
