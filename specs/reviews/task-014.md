## Round 1

- [x] [process-revision-complete] **Finding 1: `validateFieldTypes` does not check constraint-type applicability.** The spec's Entity Type System section (spec.md:207-228) annotates constraints to specific field types: "min_length, max_length, pattern (strings)" and "min, max (numerics)". The validate function checks that the field type itself is valid, and handles ref-specific (`to`) and enum-specific (`values`) attributes, but does not verify that constraint attributes are applied only to their designated field types. A declaration like `{name: count, type: int32, min_length: 5}` passes validation despite `min_length` being a string-only constraint per the spec. Similarly, `{name: label, type: string, values: [a, b]}` passes despite `values` being enum-only, and `{name: role, type: string, to: Widget}` is silently accepted despite `to` being ref-specific. The spec defines a finite, typed constraint set â€” the validate function enforces the field type but not the constraint-to-type mapping within that set. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 2: `validateFieldTypes` does not check `computed`/`filled_by` consistency.** The spec (spec.md:46-54) defines computed fields as the pairing `computed: true, filled_by: <fill>`. The validate function does not check that a field with `computed: true` has `filled_by` set, or that a field with `filled_by` set also has `computed: true`. A field `{name: status, type: jsonb, computed: true}` (missing `filled_by`) passes validation, as does `{name: status, type: jsonb, filled_by: aggregator}` (missing `computed: true`). The spec presents these as a pair: "Computed/derived fields are read-only fields populated by a fill, never written via the API" with the example always showing both attributes together. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 3: `validateFillsExist` silently ignores non-`os.IsNotExist` errors from `os.Stat`.** `validateFillsExist` (validate.go:310-313) checks `os.IsNotExist(err)` and only appends a validation error when that specific condition is true. When `os.Stat` returns a different error â€” such as permission denied (`EACCES`) â€” the condition is false and no error is appended, so the fill is silently treated as existing. This can produce a false positive: validation reports no fill errors when the function could not actually verify fill existence due to filesystem permission issues. In contrast, `DetectDrift` (drift.go:57-64) correctly handles non-`IsNotExist` read errors by returning them as infrastructure Go errors. The task acceptance criteria state "validate catches: ... missing fill."

## Round 2

- [x] [process-revision-complete] **Finding 4: `validateSlotNames` does not include mixin-added slots (`adds_slots`) in the available slot set.** The function builds its available slot set exclusively from `comp.Slots` across resolved components (validate.go:329-334). The comment at validate.go:337-338 states "they are already included in the component set via collectComponentNames, so their slots are covered" â€” this is incorrect. `collectComponentNames` (reconciler.go:395-406) adds only the mixin's `AddsComponents` (component names) to the component list; it does not merge the mixin's `AddsSlots` into any component's slot definitions. A mixin like the spec's `event-publisher` example (spec.md:296-312) adds a slot `on_entity_changed` directly via `adds_slots`, with proto `stego.mixins.event_publisher.slots.OnEntityChanged` â€” this slot is defined by the mixin, not by any component. A service binding to this mixin-added slot would receive a false positive "slot not defined by any component" validation error. The task description says "Validate slot bindings match available slots."

- [x] [process-revision-complete] **Finding 5: `unique_composite` field name references are not validated against existing fields within the entity.** The spec (spec.md:213) defines `unique_composite: [field1, field2]` as a constraint attribute where the list values reference other field names within the same entity. The validate function checks cross-entity references â€” `to` on ref fields is validated against entity names (validate.go:197-203) â€” and same-type referencing attributes like `values` on enums for non-emptiness (validate.go:205-210), but does not validate that `unique_composite` field name references resolve to existing fields in the entity's own field list. A field `{name: email, type: string, unique_composite: [nonexistent_field]}` passes validation. The task description says "Validate entity type system constraints are valid."

## Round 3

- [ ] **Finding 6: `scope` field reference on expose blocks is not validated against entity fields.** The spec (spec.md:143) defines `scope` as `{ type: field-ref, optional: true }` in the component config schema â€” it is explicitly typed as a field reference. The spec example (spec.md:338-340) shows `scope: org_id` on the User expose block, where `org_id` is a field on the User entity. `validateExposeReferences` (validate.go:312-334) checks that the expose block's `entity` and `parent` references resolve to defined entity names, but does not check that the `scope` value (when set) resolves to an existing field on the referenced entity. An expose block `{entity: User, operations: [list], scope: nonexistent_field}` passes validation despite `nonexistent_field` not existing on User. The task description says "Validate expose blocks reference defined entities."

- [ ] **Finding 7: `upsert_key` field name references on expose blocks are not validated against entity fields.** The spec (spec.md:23-30) defines upsert with `upsert_key: [resource_type, resource_id, adapter]` â€” a list of field names that must exist on the entity to form the natural key for conflict resolution. This is structurally identical to `unique_composite: [field1, field2]` (spec.md:213), which was validated in Finding 5's fix (validate.go:276-289): both are lists of field name references within an entity. `upsert_key` field names receive no validation â€” an expose block with `upsert_key: [nonexistent_field]` passes. The task description says "Validate expose blocks reference defined entities."
