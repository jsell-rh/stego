# Task 014 Review — Validation & Drift Detection

## Round 1

- [ ] **Finding 1: `validateFieldTypes` does not check constraint-type applicability.** The spec's Entity Type System section (spec.md:207-228) annotates constraints to specific field types: "min_length, max_length, pattern (strings)" and "min, max (numerics)". The validate function checks that the field type itself is valid, and handles ref-specific (`to`) and enum-specific (`values`) attributes, but does not verify that constraint attributes are applied only to their designated field types. A declaration like `{name: count, type: int32, min_length: 5}` passes validation despite `min_length` being a string-only constraint per the spec. Similarly, `{name: label, type: string, values: [a, b]}` passes despite `values` being enum-only, and `{name: role, type: string, to: Widget}` is silently accepted despite `to` being ref-specific. The spec defines a finite, typed constraint set — the validate function enforces the field type but not the constraint-to-type mapping within that set. The task description says "Validate entity type system constraints are valid."

- [ ] **Finding 2: `validateFieldTypes` does not check `computed`/`filled_by` consistency.** The spec (spec.md:46-54) defines computed fields as the pairing `computed: true, filled_by: <fill>`. The validate function does not check that a field with `computed: true` has `filled_by` set, or that a field with `filled_by` set also has `computed: true`. A field `{name: status, type: jsonb, computed: true}` (missing `filled_by`) passes validation, as does `{name: status, type: jsonb, filled_by: aggregator}` (missing `computed: true`). The spec presents these as a pair: "Computed/derived fields are read-only fields populated by a fill, never written via the API" with the example always showing both attributes together. The task description says "Validate entity type system constraints are valid."

- [ ] **Finding 3: `validateFillsExist` silently ignores non-`os.IsNotExist` errors from `os.Stat`.** `validateFillsExist` (validate.go:310-313) checks `os.IsNotExist(err)` and only appends a validation error when that specific condition is true. When `os.Stat` returns a different error — such as permission denied (`EACCES`) — the condition is false and no error is appended, so the fill is silently treated as existing. This can produce a false positive: validation reports no fill errors when the function could not actually verify fill existence due to filesystem permission issues. In contrast, `DetectDrift` (drift.go:57-64) correctly handles non-`IsNotExist` read errors by returning them as infrastructure Go errors. The task acceptance criteria state "validate catches: ... missing fill."
