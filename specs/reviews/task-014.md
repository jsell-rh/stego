# Task 014 Review — Validation & Drift Detection

## Round 1

- [x] [process-revision-complete] **Finding 1: `validateFieldTypes` does not check constraint-type applicability.** The spec's Entity Type System section (spec.md:207-228) annotates constraints to specific field types: "min_length, max_length, pattern (strings)" and "min, max (numerics)". The validate function checks that the field type itself is valid, and handles ref-specific (`to`) and enum-specific (`values`) attributes, but does not verify that constraint attributes are applied only to their designated field types. A declaration like `{name: count, type: int32, min_length: 5}` passes validation despite `min_length` being a string-only constraint per the spec. Similarly, `{name: label, type: string, values: [a, b]}` passes despite `values` being enum-only, and `{name: role, type: string, to: Widget}` is silently accepted despite `to` being ref-specific. The spec defines a finite, typed constraint set — the validate function enforces the field type but not the constraint-to-type mapping within that set. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 2: `validateFieldTypes` does not check `computed`/`filled_by` consistency.** The spec (spec.md:46-54) defines computed fields as the pairing `computed: true, filled_by: <fill>`. The validate function does not check that a field with `computed: true` has `filled_by` set, or that a field with `filled_by` set also has `computed: true`. A field `{name: status, type: jsonb, computed: true}` (missing `filled_by`) passes validation, as does `{name: status, type: jsonb, filled_by: aggregator}` (missing `computed: true`). The spec presents these as a pair: "Computed/derived fields are read-only fields populated by a fill, never written via the API" with the example always showing both attributes together. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 3: `validateFillsExist` silently ignores non-`os.IsNotExist` errors from `os.Stat`.** `validateFillsExist` (validate.go:310-313) checks `os.IsNotExist(err)` and only appends a validation error when that specific condition is true. When `os.Stat` returns a different error — such as permission denied (`EACCES`) — the condition is false and no error is appended, so the fill is silently treated as existing. This can produce a false positive: validation reports no fill errors when the function could not actually verify fill existence due to filesystem permission issues. In contrast, `DetectDrift` (drift.go:57-64) correctly handles non-`IsNotExist` read errors by returning them as infrastructure Go errors. The task acceptance criteria state "validate catches: ... missing fill."

## Round 2

- [x] [process-revision-complete] **Finding 4: `validateSlotNames` does not include mixin-added slots (`adds_slots`) in the available slot set.** The function builds its available slot set exclusively from `comp.Slots` across resolved components (validate.go:329-334). The comment at validate.go:337-338 states "they are already included in the component set via collectComponentNames, so their slots are covered" — this is incorrect. `collectComponentNames` (reconciler.go:395-406) adds only the mixin's `AddsComponents` (component names) to the component list; it does not merge the mixin's `AddsSlots` into any component's slot definitions. A mixin like the spec's `event-publisher` example (spec.md:296-312) adds a slot `on_entity_changed` directly via `adds_slots`, with proto `stego.mixins.event_publisher.slots.OnEntityChanged` — this slot is defined by the mixin, not by any component. A service binding to this mixin-added slot would receive a false positive "slot not defined by any component" validation error. The task description says "Validate slot bindings match available slots."

- [x] [process-revision-complete] **Finding 5: `unique_composite` field name references are not validated against existing fields within the entity.** The spec (spec.md:213) defines `unique_composite: [field1, field2]` as a constraint attribute where the list values reference other field names within the same entity. The validate function checks cross-entity references — `to` on ref fields is validated against entity names (validate.go:197-203) — and same-type referencing attributes like `values` on enums for non-emptiness (validate.go:205-210), but does not validate that `unique_composite` field name references resolve to existing fields in the entity's own field list. A field `{name: email, type: string, unique_composite: [nonexistent_field]}` passes validation. The task description says "Validate entity type system constraints are valid."

## Round 3

- [x] [process-revision-complete] **Finding 6: `scope` field reference on expose blocks is not validated against entity fields.** The spec (spec.md:143) defines `scope` as `{ type: field-ref, optional: true }` in the component config schema — it is explicitly typed as a field reference. The spec example (spec.md:338-340) shows `scope: org_id` on the User expose block, where `org_id` is a field on the User entity. `validateExposeReferences` (validate.go:312-334) checks that the expose block's `entity` and `parent` references resolve to defined entity names, but does not check that the `scope` value (when set) resolves to an existing field on the referenced entity. An expose block `{entity: User, operations: [list], scope: nonexistent_field}` passes validation despite `nonexistent_field` not existing on User. The task description says "Validate expose blocks reference defined entities."

- [x] [process-revision-complete] **Finding 7: `upsert_key` field name references on expose blocks are not validated against entity fields.** The spec (spec.md:23-30) defines upsert with `upsert_key: [resource_type, resource_id, adapter]` — a list of field names that must exist on the entity to form the natural key for conflict resolution. This is structurally identical to `unique_composite: [field1, field2]` (spec.md:213), which was validated in Finding 5's fix (validate.go:276-289): both are lists of field name references within an entity. `upsert_key` field names receive no validation — an expose block with `upsert_key: [nonexistent_field]` passes. The task description says "Validate expose blocks reference defined entities."

## Round 4

- [x] [process-revision-complete] **Finding 8: Duplicate entity names are not detected by `validateFieldTypes` or `validateExposeReferences`.** Both functions build entity name lookup maps — `validateFieldTypes` at validate.go:184-187 and `validateExposeReferences` at validate.go:314-317 — by iterating `entities` and inserting into `map[string]bool`. When two entities share the same name, the second silently overwrites the first. No error is reported. Entities are referenced by name throughout the service declaration: ref field `to` targets (spec.md:226), expose block `entity` and `parent` references (spec.md:36-44), scope and upsert_key field lookups (spec.md:143, spec.md:23-30), and slot binding `entity` references (spec.md:348). Duplicate entity names make all of these references ambiguous — the lookup maps cannot distinguish which entity is intended. The spec's entity examples (spec.md:219-227, 326-332) always show distinct entity names, and generated code (Go struct names, SQL table names) requires unique names. A service.yaml with two `- name: Widget` entity entries passes validation. The task description says "Validate entity type system constraints are valid."

- [x] [process-revision-complete] **Finding 9: Duplicate field names within an entity are not detected by `validateFieldTypes`.** The function iterates each entity's fields at validate.go:190-306 without checking that field names are unique within the entity. Two fields with the same name both pass all individual field validations independently. The `unique_composite` reference validation (validate.go:277-279) and the `validateExposeReferences` scope/upsert_key validation (validate.go:318-322) also build field name lookup maps from entity fields that silently collapse duplicates. Duplicate field names cannot be expressed in target-language representations: Go struct field names must be unique, SQL column names must be unique within a table, and OpenAPI schema property names must be unique. The spec's field examples (spec.md:219-227) always show distinct field names within each entity. A service.yaml with two `- { name: label, type: string }` entries in the same entity passes validation. The task description says "Validate entity type system constraints are valid."

## Round 5

- [x] [process-revision-complete] **Finding 10: `upsert_key` on an expose block is not validated against the operations list — specifying `upsert_key` without the `upsert` operation passes validation.** The spec (spec.md:23-30) shows `upsert_key` exclusively in the context of the `upsert` operation: `operations: [list, upsert]` paired with `upsert_key: [resource_type, resource_id, adapter]`. The `upsert_key` attribute provides the natural-key fields for upsert's conflict resolution — it has no semantic meaning without the `upsert` operation. The validate function checks that `upsert_key` field names reference existing fields on the entity (validate.go:372-380, Finding 7 fix), but does not check that `upsert` is present in the expose block's operations list. An expose block `{entity: Widget, operations: [create, read], upsert_key: [resource_type, resource_id]}` passes validation despite the upsert_key having no operation to serve. This follows the same pattern as Finding 1's constraint-type applicability: `values` is rejected on non-enum fields (validate.go:279-284), `to` is rejected on non-ref fields (validate.go:287-292), but `upsert_key` is accepted on expose blocks that lack the `upsert` operation. The task description says "Validate expose blocks reference defined entities."

## Round 6

- [x] [process-revision-complete] **Finding 11: `upsert` operation without `upsert_key` passes validation — the converse of Finding 10.** The spec (spec.md:22-30) introduces the upsert operation as "Upsert supports natural-key conflict resolution" and shows the only example with `upsert_key: [resource_type, resource_id, adapter]` providing the natural key fields. The `upsert_key` defines which fields form the natural key for conflict resolution — without it, the upsert operation has no defined conflict resolution strategy. Finding 10 validated one direction: `upsert_key` requires the `upsert` operation. The converse — the `upsert` operation requires `upsert_key` — is not validated. The validate function at validate.go:382-397 only enters the upsert/upsert_key pairing check via `if len(eb.UpsertKey) > 0`, so an expose block with `operations: [upsert]` but no `upsert_key` bypasses the check entirely. An expose block `{entity: Widget, operations: [upsert]}` passes validation. The rest-api generator confirms the consequence: at generator.go:603-604, when `UpsertKey` is empty, the generated handler passes `upsertKey := []string{}` to the store's `Upsert` call — an empty conflict key with no conflict resolution fields. The task description says "Validate expose blocks reference defined entities."

## Round 7

- [x] [process-revision-complete] **Finding 12: `concurrency` attribute on an expose block without `upsert` operation passes validation.** The spec (spec.md:23-30) shows `concurrency: optimistic` exclusively paired with the upsert operation: `operations: [list, upsert]` with `concurrency: optimistic # only update if generation is newer`. The `concurrency` attribute controls how upsert handles conflict resolution — it has no semantic meaning without the `upsert` operation. This follows the exact pattern of Findings 10 and 11: `upsert_key` without `upsert` was flagged (Finding 10), `upsert` without `upsert_key` was flagged (Finding 11), but `concurrency` without `upsert` is not checked. The rest-api generator confirms the dependency: at restapi/generator.go:607-611, the `concurrency` attribute is only read in the upsert handler code path. An expose block `{entity: Widget, operations: [create, read], concurrency: optimistic}` passes validation despite the concurrency attribute having no operation to serve. The task description says "Validate expose blocks reference defined entities."

- [x] [process-revision-complete] **Finding 13: Invalid `concurrency` values are not rejected by the validate function.** The spec (spec.md:29) defines only one concurrency mode: `concurrency: optimistic`. The types define only `ConcurrencyOptimistic = "optimistic"` (types.go:222-224). The validate function checks field types against `ValidFieldTypes` (validate.go:212) and operations against `ValidOperations` (validate.go:411), but does not check `concurrency` values against a valid set — there is no `ValidConcurrencyModes` map or equivalent. An expose block with `concurrency: pessimistic` passes validation despite `pessimistic` not being a defined concurrency mode. The generated code passes the raw value through to the postgres-adapter's `Upsert` function (restapi/generator.go:611), where it is silently treated as no concurrency — the generated store only checks for `"optimistic"` (postgresadapter/generator.go:564). Invalid values produce silently degraded behavior rather than a validation error.

- [x] [process-revision-complete] **Finding 14: `parent` reference on expose blocks is not validated against the expose list — only against the entities list.** The spec (spec.md:33-35) says: "Nested routing is expressed via `parent` on an expose block. The compiler generates parent existence verification at each level." For the compiler to generate parent existence verification, the parent entity must have its own expose block — without it, there is no route for the parent and no handler to verify the parent exists. `validateExposeReferences` (validate.go:355-360) checks that the parent entity exists in the entities list, but does not check that it is in the expose list. An expose block `{entity: NodePool, parent: Cluster, operations: [create]}` passes validation when Cluster is defined in entities but NOT in the expose list. The rest-api generator catches this at generation time (restapi/generator.go:1295-1307, `validateParentReferences`), but the validate function does not — meaning `stego validate` reports "Validation passed" while `stego plan` subsequently fails. This is inconsistent with `validateSlotBindingEntitiesCollect` (validate.go:461-489), which validates slot binding entity references against the expose list (not just the entities list). Both are entity references that must resolve to exposed entities for the generated code to function, but only slot binding entities receive the expose-list check.

## Round 8

- [x] [process-revision-complete] **Finding 15: Duplicate expose blocks for the same entity pass `stego validate`.** The spec's expose block examples (spec.md:36-44, 336-340) show each entity appearing exactly once in the expose list. The rest-api generator rejects duplicates via `validateExposeUniqueness` (restapi/generator.go:1212-1225), which detects that duplicate entries cause duplicate type declarations and silent map overwrites. The validate function's `validateExposeReferences` builds `exposedEntities` (validate.go:348-351) using `map[string]bool` that silently collapses duplicates — it never counts or flags them. A service.yaml with two expose blocks for "Widget" (e.g. with different operations or scopes) passes `stego validate` but fails `stego plan` with "duplicate expose blocks for entities: Widget". This is inconsistent with duplicate entity name detection (Finding 8, validate.go:188-196) and duplicate field name detection (Finding 9, validate.go:200-209), which both detect duplicates during validation. The task description says "Validate expose blocks reference defined entities."

- [x] [process-revision-complete] **Finding 16: Duplicate operations within an expose block pass `stego validate`.** The `validateExposeOps` function (validate.go:444-458) checks that each operation is in `ValidOperations`, but does not check for duplicates within a single expose block. An expose block `{entity: Widget, operations: [create, create, read]}` passes validation. The rest-api generator rejects this via `validateOperationUniqueness` (restapi/generator.go:1557-1592), which detects that duplicate operations produce duplicate method declarations (Go compile error), duplicate route registrations (Go 1.22 ServeMux runtime panic), and duplicate OpenAPI operation entries (silent overwrite). `stego validate` passes but `stego plan` fails. This follows the pattern of duplicate entity names (Finding 8) and duplicate field names (Finding 9) — the validate function checks value validity but not uniqueness within lists. The task description says "Validate expose blocks reference defined entities."

## Round 9

- [x] [process-revision-complete] **Finding 17: `short_circuit: true` on a slot binding without a chain operator passes validation.** The spec (spec.md:56-66) defines `short_circuit: true` exclusively in the context of chain steps: `chain: [...] short_circuit: true # enables halt semantics`. The spec's protobuf definition (spec.md:67-74) describes the `halt` field as "stop the chain, return this result" — halt semantics are chain-specific. The slot operators code (operators.go:95-116) only references `ShortCircuit` in the chain operator struct and its evaluation method. The assembler (assembler.go:917-921) only reads `sb.ShortCircuit` inside the `if len(sb.Chain) > 0` block — when the binding uses a gate or fan-out, `ShortCircuit` is silently ignored. A slot binding `{slot: before_create, entity: User, gate: [rbac-policy], short_circuit: true}` passes validation despite `short_circuit` having no effect on gate operations. This follows the same pattern as Findings 10 and 12: `upsert_key` without `upsert` was flagged (Finding 10), `concurrency` without `upsert` was flagged (Finding 12), but `short_circuit` without `chain` is not checked. The task description says "Validate slot bindings match available slots."

- [x] [process-revision-complete] **Finding 18: Duplicate slot binding declarations (same slot, entity, operator) pass `stego validate`.** The assembler detects these via `validateSlotBindingUniqueness` (assembler.go:1349-1380), which checks that no two slot bindings share the same (slot, entity, operator) composite key. Duplicate bindings produce duplicate `:=` variable declarations in the generated main.go — a Go compile error. `validateSlotBindingUniqueness` is only called from `generateMainGo` (assembler.go:114), not from the `Validate` function. A service.yaml with two `{slot: before_create, entity: User, gate: [...]}` entries passes `stego validate` but fails `stego plan`. This follows the pattern of Findings 15 and 16: duplicate expose blocks (Finding 15) and duplicate operations (Finding 16) were both detected by generators/assembler but not by the validate function. Both were fixed by adding duplicate detection to validate. Duplicate slot bindings have the same structural pattern — lists whose elements should be unique — but receive no uniqueness check in validate. The task description says "Validate slot bindings match available slots."

## Round 10

- [x] [process-revision-complete] **Finding 19: Empty operations list in an expose block passes `stego validate`.** The rest-api generator's `validateExposeOperations` (restapi/generator.go:1438-1450) rejects expose blocks with empty operations lists: `"expose blocks with no operations: %s; each expose block must have at least one operation"`. The validate function's `validateExposeOps` (validate.go:470-491) checks for invalid operation values (line 483) and duplicate operations (line 475), but never checks whether the operations list is empty. When `eb.Operations` is empty or nil, the inner `for _, op := range eb.Operations` loop body never executes, so no errors are produced. An expose block `{entity: Widget, operations: []}` passes `stego validate` but fails `stego plan`. The spec's expose block examples (spec.md:36-44, 336-340) always show at least one operation — an expose block exists to expose operations, and one with no operations is semantically void. This follows the exact pattern of Findings 15 and 16: the rest-api generator catches a condition during `stego plan` that the validate function does not catch during `stego validate`. The task description says "Validate expose blocks reference defined entities."

- [x] [process-revision-complete] **Finding 20: Duplicate enum values within a field's `values` list pass `stego validate`.** The postgres-adapter's `validateEnumValues` (postgresadapter/generator.go:1334-1343) detects duplicate enum values: it builds a `seen` map and reports `"entity %s: field %q has duplicate enum value %q"`. The validate function checks that enum fields have at least one value (validate.go:241-246) and that the `values` attribute is only used on enum fields (validate.go:285-289), but does not check for duplicate values within the list. A field `{name: role, type: enum, values: [admin, admin, member]}` passes `stego validate` but fails `stego plan`. Duplicate enum values produce duplicate Go const declarations (compile error) and duplicate SQL CHECK constraint values. The spec (spec.md:214, 226) shows `values: [a, b, c]` and `values: [admin, member]` — always distinct. This follows the same pattern as duplicate field names (Finding 9), duplicate operations (Finding 16), and duplicate expose blocks (Finding 15): lists whose elements should be unique receive no uniqueness check in validate, while the generators catch it downstream. The task description says "Validate entity type system constraints are valid."
