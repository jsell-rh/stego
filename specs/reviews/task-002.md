# Review: Task 002 — Core Domain Types

## Findings

- [x] **`Port` struct is dead code.** `Port` is defined (`types.go:85-87`) but never referenced by `Component` or any other type. `Component.Requires` and `Component.Provides` are `[]string`, not `[]Port`. The task description specifies "Port — name (for requires/provides)", implying it should be used by `Component`. Either wire `[]Port` into `Component.Requires`/`Component.Provides`, or remove `Port` entirely. Dead types pollute the API surface and will confuse consumers.

- [x] **`ConfigField` cannot represent nested config schemas.** The spec shows recursive config structures (e.g., `expose.items` containing typed sub-fields `entity`, `operations`, `scope`, each with their own `type`). `ConfigField.Items` is typed as `any` (`types.go:121`), so nested schemas unmarshal into untyped `map[string]interface{}` rather than `ConfigField` values. This means config schemas cannot be validated or traversed without runtime type assertions, undermining the purpose of having typed Go structs. `Items` should be `map[string]ConfigField` (or similar recursive type) to model the spec's nested config schemas.

- [ ] **`ConfigField.Items` as `map[string]ConfigField` breaks the spec's leaf-list config.** The previous finding was fixed by making `Items` a `map[string]ConfigField`, but the spec's component YAML uses `items` in two structurally different ways: (1) as a map of named sub-fields for `expose.items` (works), and (2) as a **single ConfigField** for leaf-list elements like `operations: { type: list, items: { enum: [create, read, update, delete, list] } }`. In case (2), `items` is a single ConfigField with `Enum` set — not a map of named fields. Unmarshaling the spec's exact component YAML produces: `yaml: unmarshal errors: line 11: cannot unmarshal !!seq into ConfigField`. Verified by running the spec's component example through `yaml.Unmarshal`. The `Items` field needs to support both forms (e.g. a union type, or a custom `UnmarshalYAML` that detects which shape it is).

- [ ] **`TestComponentUnmarshalWithConfig` omits the nested `operations.items` from the spec, masking the above failure.** The test at `types_test.go:329` parses a component config with `operations: { type: list }` but drops the `items: { enum: [...] }` sub-field that the spec shows. This means the test passes despite the underlying unmarshal failure. The test should use the spec's full component YAML to catch this.

- [ ] **Type is named `SlotBinding`, task requires `SlotDeclaration`.** The task description explicitly lists `SlotDeclaration` as the type to define for "slot name, entity, gate/chain/fan-out bindings, short_circuit". The code defines `SlotBinding` (`types.go:186`) instead. While functionally equivalent, the naming contradicts the task specification.
