# Review: Task 004 — Registry Structure & Loading

## Findings

- [ ] **Duplicate artifact names are silently overwritten.** `loadArchetypes` (`registry.go:84-95`), `loadComponents` (`registry.go:98-114`), and `loadMixins` (`registry.go:117-133`) index artifacts by the `Name` field from the YAML file. If two directories contain artifacts with the same `name:` value, the second silently overwrites the first in the map. The spec treats names as unique lookup keys ("Lookup by name works for each kind" — acceptance criteria). Silent overwriting is a data integrity risk: a registry author adds a duplicate name and one artifact vanishes with no error.

- [ ] **No validation that directory name matches artifact name.** The task description defines the registry layout as `registry/archetypes/<name>/archetype.yaml`, implying the directory name IS the artifact name. The code indexes by the YAML `name:` field and ignores the directory name entirely. A directory named `foo/` containing `archetype.yaml` with `name: bar` is silently indexed as `bar`. This breaks the contract between filesystem layout and artifact identity — a user browsing the registry directory sees `foo` but must look up `bar`.

- [ ] **Registry layout does not account for `slots/*.proto` files.** The task description explicitly lists `registry/components/<name>/slots/*.proto` as part of the registry directory layout to define. The implementation does not load, index, or verify the existence of proto files. The fixture component `rest-api` references proto paths (`stego.components.rest_api.slots.BeforeCreate`, `stego.components.rest_api.slots.Validate`) in its slot definitions, but no corresponding `slots/` directory or `.proto` files exist in the test fixture. Even if protobuf compilation is deferred, the registry loader should at minimum verify that slot proto files exist on disk for each slot definition, or the fixture should include stub proto files to validate the directory layout contract.

- [ ] **`LoadConfig` performs no validation of parsed config.** An empty YAML file, or a file with `registry: []` and no `pins:`, parses successfully as a valid `RegistryConfig`. The spec requires at least one registry source with both `url` and `ref` fields (`registry: [{url: ..., ref: ...}]`). A `RegistrySource` with empty `URL` or empty `Ref` is accepted without error. Basic validation — at least one registry source, non-empty URL and Ref — would catch malformed configs early rather than failing downstream.
