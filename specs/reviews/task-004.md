# Review: Task 004 — Registry Structure & Loading

## Findings

- [x] [process-revision-complete] **Duplicate artifact names are silently overwritten.** `loadArchetypes` (`registry.go:84-95`), `loadComponents` (`registry.go:98-114`), and `loadMixins` (`registry.go:117-133`) index artifacts by the `Name` field from the YAML file. If two directories contain artifacts with the same `name:` value, the second silently overwrites the first in the map. The spec treats names as unique lookup keys ("Lookup by name works for each kind" — acceptance criteria). Silent overwriting is a data integrity risk: a registry author adds a duplicate name and one artifact vanishes with no error.

- [x] [process-revision-complete] **No validation that directory name matches artifact name.** The task description defines the registry layout as `registry/archetypes/<name>/archetype.yaml`, implying the directory name IS the artifact name. The code indexes by the YAML `name:` field and ignores the directory name entirely. A directory named `foo/` containing `archetype.yaml` with `name: bar` is silently indexed as `bar`. This breaks the contract between filesystem layout and artifact identity — a user browsing the registry directory sees `foo` but must look up `bar`.

- [x] [process-revision-complete] **Registry layout does not account for `slots/*.proto` files.** The task description explicitly lists `registry/components/<name>/slots/*.proto` as part of the registry directory layout to define. The implementation does not load, index, or verify the existence of proto files. The fixture component `rest-api` references proto paths (`stego.components.rest_api.slots.BeforeCreate`, `stego.components.rest_api.slots.Validate`) in its slot definitions, but no corresponding `slots/` directory or `.proto` files exist in the test fixture. Even if protobuf compilation is deferred, the registry loader should at minimum verify that slot proto files exist on disk for each slot definition, or the fixture should include stub proto files to validate the directory layout contract.

- [x] [process-revision-complete] **`LoadConfig` performs no validation of parsed config.** An empty YAML file, or a file with `registry: []` and no `pins:`, parses successfully as a valid `RegistryConfig`. The spec requires at least one registry source with both `url` and `ref` fields (`registry: [{url: ..., ref: ...}]`). A `RegistrySource` with empty `URL` or empty `Ref` is accepted without error. Basic validation — at least one registry source, non-empty URL and Ref — would catch malformed configs early rather than failing downstream.

- [x] [process-revision-complete] **Mixin `adds_slots` proto files are not validated, unlike component slots.** `loadComponents` (`registry.go:128-133`) verifies that each slot's proto file exists on disk at `<component-dir>/slots/<slot-name>.proto`. `loadMixins` (`registry.go:139-163`) performs no equivalent check for `adds_slots`. The fixture mixin `event-publisher` declares `adds_slots` with proto reference `stego.mixins.event_publisher.slots.OnEntityChanged`, but no corresponding `slots/` directory or proto file exists under `testdata/registry/mixins/event-publisher/`. The spec defines mixin `adds_slots` with the same `proto` field structure as component slots. Either mixin slot protos should be validated the same way component slot protos are, or the design decision to skip mixin proto validation should be documented.

- [x] [process-revision-complete] **`Load` succeeds silently when the registry directory does not exist.** `Load("/nonexistent/path")` returns an empty `Registry` with zero archetypes, components, and mixins — no error. This is because `readDirIfExists` (`registry.go:167-172`) returns `nil, nil` for non-existent directories, and all three subdirectory loads (`archetypes/`, `components/`, `mixins/`) use it. A typo in the registry path (e.g. `/opt/regstry` instead of `/opt/registry`) silently produces an empty registry indistinguishable from a valid but empty one. The spec says "The registry is a git repo" — a non-existent directory is not a registry. `Load` should verify that `dir` itself exists before proceeding.

- [x] [process-revision-complete] **Duplicate-name detection code is unreachable and tests are mislabeled.** Lines 98-100, 124-126, 157-159 check for duplicate names in the maps. However, since (a) two directories within the same parent cannot share a name on any filesystem, and (b) the name-mismatch check enforces `YAML name == directory name`, it is structurally impossible for two artifacts of the same kind to have the same name. The duplicate check can never trigger. The tests `TestLoadDuplicateArchetypeName` (line 306), `TestLoadDuplicateComponentName` (line 331), and `TestLoadDuplicateMixinName` (line 349) acknowledge this in comments — they only exercise the name-mismatch path, not duplicate detection. The test names claim to test "duplicate name" but actually test "name mismatch." Either remove the unreachable code or rename the tests to reflect what they actually verify.
