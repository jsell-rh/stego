# Task 013 Review — Compiler Core: Plan/Apply Reconciler

## Round 1

- [process-revision-complete] **Finding 1: RegistrySHA never populated in state.** The spec states "State tracked in `.stego/state.yaml`, pinned to exact registry SHAs for full auditability" and shows `registry_sha: a1b2c3d4e5f6` in the example state.yaml. The `AppliedState` struct declares a `RegistrySHA` field (`state.go:25`), but `computePlan` (`reconciler.go:484-498`) never sets it. After every apply, `RegistrySHA` is always empty, violating the auditability requirement.

- [process-revision-complete] **Finding 2: Component SHA never populated in state.** The spec's state.yaml example shows per-component `sha: a1b2c3d4e5f6`, and `ComponentState` has a `SHA` field (`state.go:41`). However, `computePlan` (`reconciler.go:478-482`) only sets `Version` — `SHA` is always empty. This breaks the "pinned to exact registry SHAs" invariant for components.

- [process-revision-complete] **Finding 3: Deleted entities not detected by plan.** `computeEntityChanges` (`reconciler.go:503-559`) iterates only over current entities (`for _, e := range entities`). If an entire entity present in the previous state is removed from service.yaml, the removal is never reported. For example, removing `Widget` entirely between applies produces zero `EntityChange` entries — the plan silently omits the deletion.

- [process-revision-complete] **Finding 4: TOCTOU race on service.yaml.** `Reconcile` reads `serviceYAMLPath` via `os.ReadFile` (`reconciler.go:93`) to obtain raw bytes for hashing, then calls `parser.ParseServiceDeclaration(serviceYAMLPath)` (`reconciler.go:94`) which reads the same file independently. If the file is modified between the two reads, the hash stored in state will not correspond to the declaration that was actually compiled. The parser should accept the already-read bytes, or the hash should be computed from the same read.

- [process-revision-complete] **Finding 5: Stale comment on `hasEntityChanges`.** The doc comment on `hasEntityChanges()` (`reconciler.go:561`) reads `// HasChanges returns true if the plan includes any generate, update, or entity changes.` — this is the doc for `HasChanges`, not `hasEntityChanges`. The comment was copy-pasted and not updated to reflect the actual function.

- [process-revision-complete] **Finding 6: `runApply` and `FormatPlan` disagree on what constitutes "no changes".** `runApply` (`cmd/stego/main.go:126`) gates apply on `plan.HasChanges()` only, which checks file actions. `FormatPlan` (`reconciler.go:568`) checks both `plan.HasChanges()` and `plan.hasEntityChanges()`. If entity changes exist but all files happen to hash the same (e.g. cosmetic service.yaml edits that don't affect output), `runApply` prints "No changes" and skips, while `FormatPlan` would report entity changes. The two code paths should use the same predicate.

- [process-revision-complete] **Finding 7: go.mod header bypasses `gen.File.Bytes()` mechanism.** `generateGoMod` (`assembler.go:83`) embeds the `// Code generated by stego. DO NOT EDIT.` header directly in `Content`. All other generated files rely on `gen.File.Bytes()` to prepend the header for `.go` files. This means the go.mod's header is part of `Content` and will be included in the file hash computed by `computePlan`, while for `.go` files the header is added by `Bytes()` after hashing via `Content`. This creates an inconsistency: the hash for go.mod includes the header, but the hash for .go files also includes the header (since `computePlan` calls `f.Bytes()`). Functionally correct, but the dual mechanism is fragile — any change to the header text requires updating both `gen.Header` and the hardcoded string in `generateGoMod`.

## Round 2

- [process-revision-complete] **Finding 8: Orphaned files not detected by plan and not removed by apply.** `computePlan` (`reconciler.go:444-468`) only iterates over `generatedFiles`. Files tracked in the previous state (`existingHashes`) that are no longer produced by any generator are silently dropped from the new state and left on disk. The acceptance criteria state "plan shows accurate changeset" — omitting deletions makes the changeset incomplete. Concretely: if an entity is removed from `service.yaml` between applies, the handler file for that entity (e.g. `internal/api/handler_widget.go`) remains on disk referencing types that no longer exist. The acceptance criteria also state "`go build` works on generated output" — orphaned files with stale imports will break compilation.

- [process-revision-complete] **Finding 9: `computePlan` hardcodes output directory, `Apply` accepts it as parameter.** `computePlan` at `reconciler.go:453` uses `filepath.Join(projectDir, "out", f.Path)` to check existing files on disk. `Apply` at `reconciler.go:231` accepts `outDir` as a separate parameter: `func Apply(plan *Plan, projectDir, outDir string) error`. If a caller passes a non-default `outDir` to `Apply`, the plan's on-disk file checks will have been computed against the wrong directory. The `computePlan` function (or `Reconcile`, which calls it) should accept the output directory as a parameter to match `Apply`'s contract.

- [process-revision-complete] **Finding 10: Non-deterministic ordering of deleted entity changes in plan output.** In `computeEntityChanges` (`reconciler.go:579-586`), deleted entities are detected by iterating over the `oldEntities` map (`for entityName, oldFields := range oldEntities`). Go map iteration order is non-deterministic. When multiple entities are simultaneously deleted between applies, the order of `EntityChange` entries in the plan varies between runs, producing non-reproducible plan output. Added and modified entities are iterated via the `entities` slice (deterministic). The deleted-entity changes should be sorted (e.g. by entity name) before appending to maintain deterministic output consistent with the rest of the plan.

## Round 3

- [process-revision-complete] **Finding 11: Plan output omits field types shown in spec example.** The spec's compiler process section (spec.md:450) shows plan output as `+ field: display_name (string)` — field name followed by type in parentheses. `FormatPlan` at `reconciler.go:641` outputs `fmt.Fprintf(&result, "    + field: %s\n", f)` using only the field name string from `EntityChange.Added`, omitting the type entirely. Root cause: `AppliedState.Entities` is typed as `map[string][]string` (`state.go:33`) — a mapping of entity name to field names only. Since the type information is discarded when building the entity snapshot (`reconciler.go:509-515`), `FormatPlan` cannot include it. The plan output format does not match the spec.

- [process-revision-complete] **Finding 12: Entity field type and constraint changes invisible to plan.** `computeEntityChanges` (`reconciler.go:545-624`) compares entity snapshots stored as `map[string][]string` — field names only. If a field's type changes (e.g., `string` → `int32`), its constraints change (e.g., adding `max_length: 255`), or its attributes change (e.g., toggling `computed: true`), the field name is present in both old and new snapshots, so the change is not detected — `computeEntityChanges` reports zero changes. The task description states "Show entity field changes." A type or constraint modification is an entity field change that goes undetected. Concretely: changing `{ name: score, type: string }` to `{ name: score, type: int32 }` between applies produces a plan with no entity changes, even though the generated models, store queries, migration SQL, and OpenAPI schema all change.

- [process-revision-complete] **Finding 13: HTTP port hardcoded, ignoring component config resolution.** `Reconcile` at `reconciler.go:158` sets `httpPort := 8080` instead of resolving the port from the rest-api component's resolved config. The rest-api component declares `config: port: { type: int, default: 8080 }` in `registry/components/rest-api/component.yaml:7-9`. The reconciler already resolves component config via `resolveComponentConfig` (`reconciler.go:306-324`) and passes the result to generators via `ctx.ComponentConfig`, but bypasses this mechanism when constructing `AssemblerInput.Port` at `reconciler.go:211-212`. A service-level port override (e.g., `overrides: rest-api: port: 9090`) is silently ignored — the generated `main.go` always listens on `:8080`.

## Round 4

- [process-revision-complete] **Finding 14: `Reconcile` does not pass service-level port binding overrides to `ports.Resolve`.** The `ports.ResolveInput` struct declares a `ServiceOverrides` field (`ports/resolve.go:88-89`) for port→component mapping overrides from the service declaration, and `ports.Resolve` merges them with archetype bindings (`ports/resolve.go:101-108`). However, `Reconcile` at `reconciler.go:145-151` constructs `ResolveInput` with only `Components` and `ArchetypeBindings` — `ServiceOverrides` is left nil. The spec's "Port Resolution" section (spec.md:180-181) states: "The service declaration can override any binding via `overrides:`" and shows the example `overrides: auth-provider: api-key-auth` (spec.md:187). String-valued entries in `svcDecl.Overrides` represent port binding overrides (as opposed to map-valued entries which represent component config overrides), but the reconciler never extracts or forwards them. A service declaration that overrides a port binding is silently ignored — ports resolve using only archetype defaults.

- [process-revision-complete] **Finding 15: Namespace validation skipped for components without `output_namespace`.** `Reconcile` at `reconciler.go:214-217` gates namespace validation on `comp.OutputNamespace != ""`: if a component has no declared namespace, its generated files are accepted without path validation. The spec states "Each component declares an `output_namespace` ... and can only write files under it — the compiler rejects any file outside the namespace" (spec.md:268). Two MVP registry components — `health-check` (`registry/components/health-check/component.yaml`) and `otel-tracing` (`registry/components/otel-tracing/component.yaml`) — have no `output_namespace`. Their generators are currently no-op stubs returning nil files, so no violation occurs today. But the reconciler's conditional skip means a future non-stub generator for these components could write files anywhere — including under `fills/` (human-owned, never overwritten per spec.md:278) or `cmd/` (compiler-owned shared files) — without detection.

- [process-revision-complete] **Finding 16: No duplicate file path detection across generators and shared files.** `Reconcile` at `reconciler.go:220-222` appends each generator's files to `allFiles`, then appends slot files (`reconciler.go:230`) and assembler shared files (`reconciler.go:246`). No check verifies that file paths are unique across these sources. If two generators produce a file at the same path — or a generator produces a file that collides with a compiler-owned shared file (e.g., `cmd/main.go` or `go.mod`) — the later entry silently overwrites the earlier one in `Apply` (`reconciler.go:271-280`). The spec states "the archetype validates at authoring time that no two component namespaces overlap" (spec.md:269), which prevents this for well-formed archetypes with declared namespaces. But the reconciler performs no runtime collision check, so misconfigured archetypes or generators from namespace-less components (Finding 15) can silently lose generated content without any error or warning.
