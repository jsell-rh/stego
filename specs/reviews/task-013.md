# Task 013 Review — Compiler Core: Plan/Apply Reconciler

## Round 1

- [ ] **Finding 1: RegistrySHA never populated in state.** The spec states "State tracked in `.stego/state.yaml`, pinned to exact registry SHAs for full auditability" and shows `registry_sha: a1b2c3d4e5f6` in the example state.yaml. The `AppliedState` struct declares a `RegistrySHA` field (`state.go:25`), but `computePlan` (`reconciler.go:484-498`) never sets it. After every apply, `RegistrySHA` is always empty, violating the auditability requirement.

- [ ] **Finding 2: Component SHA never populated in state.** The spec's state.yaml example shows per-component `sha: a1b2c3d4e5f6`, and `ComponentState` has a `SHA` field (`state.go:41`). However, `computePlan` (`reconciler.go:478-482`) only sets `Version` — `SHA` is always empty. This breaks the "pinned to exact registry SHAs" invariant for components.

- [ ] **Finding 3: Deleted entities not detected by plan.** `computeEntityChanges` (`reconciler.go:503-559`) iterates only over current entities (`for _, e := range entities`). If an entire entity present in the previous state is removed from service.yaml, the removal is never reported. For example, removing `Widget` entirely between applies produces zero `EntityChange` entries — the plan silently omits the deletion.

- [ ] **Finding 4: TOCTOU race on service.yaml.** `Reconcile` reads `serviceYAMLPath` via `os.ReadFile` (`reconciler.go:93`) to obtain raw bytes for hashing, then calls `parser.ParseServiceDeclaration(serviceYAMLPath)` (`reconciler.go:94`) which reads the same file independently. If the file is modified between the two reads, the hash stored in state will not correspond to the declaration that was actually compiled. The parser should accept the already-read bytes, or the hash should be computed from the same read.

- [ ] **Finding 5: Stale comment on `hasEntityChanges`.** The doc comment on `hasEntityChanges()` (`reconciler.go:561`) reads `// HasChanges returns true if the plan includes any generate, update, or entity changes.` — this is the doc for `HasChanges`, not `hasEntityChanges`. The comment was copy-pasted and not updated to reflect the actual function.

- [ ] **Finding 6: `runApply` and `FormatPlan` disagree on what constitutes "no changes".** `runApply` (`cmd/stego/main.go:126`) gates apply on `plan.HasChanges()` only, which checks file actions. `FormatPlan` (`reconciler.go:568`) checks both `plan.HasChanges()` and `plan.hasEntityChanges()`. If entity changes exist but all files happen to hash the same (e.g. cosmetic service.yaml edits that don't affect output), `runApply` prints "No changes" and skips, while `FormatPlan` would report entity changes. The two code paths should use the same predicate.

- [ ] **Finding 7: go.mod header bypasses `gen.File.Bytes()` mechanism.** `generateGoMod` (`assembler.go:83`) embeds the `// Code generated by stego. DO NOT EDIT.` header directly in `Content`. All other generated files rely on `gen.File.Bytes()` to prepend the header for `.go` files. This means the go.mod's header is part of `Content` and will be included in the file hash computed by `computePlan`, while for `.go` files the header is added by `Bytes()` after hashing via `Content`. This creates an inconsistency: the hash for go.mod includes the header, but the hash for .go files also includes the header (since `computePlan` calls `f.Bytes()`). Functionally correct, but the dual mechanism is fragile — any change to the header text requires updating both `gen.Header` and the hardcoded string in `generateGoMod`.
