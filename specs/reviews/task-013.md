# Task 013 Review — Compiler Core: Plan/Apply Reconciler

## Round 1

- [process-revision-complete] **Finding 1: RegistrySHA never populated in state.** The spec states "State tracked in `.stego/state.yaml`, pinned to exact registry SHAs for full auditability" and shows `registry_sha: a1b2c3d4e5f6` in the example state.yaml. The `AppliedState` struct declares a `RegistrySHA` field (`state.go:25`), but `computePlan` (`reconciler.go:484-498`) never sets it. After every apply, `RegistrySHA` is always empty, violating the auditability requirement.

- [process-revision-complete] **Finding 2: Component SHA never populated in state.** The spec's state.yaml example shows per-component `sha: a1b2c3d4e5f6`, and `ComponentState` has a `SHA` field (`state.go:41`). However, `computePlan` (`reconciler.go:478-482`) only sets `Version` — `SHA` is always empty. This breaks the "pinned to exact registry SHAs" invariant for components.

- [process-revision-complete] **Finding 3: Deleted entities not detected by plan.** `computeEntityChanges` (`reconciler.go:503-559`) iterates only over current entities (`for _, e := range entities`). If an entire entity present in the previous state is removed from service.yaml, the removal is never reported. For example, removing `Widget` entirely between applies produces zero `EntityChange` entries — the plan silently omits the deletion.

- [process-revision-complete] **Finding 4: TOCTOU race on service.yaml.** `Reconcile` reads `serviceYAMLPath` via `os.ReadFile` (`reconciler.go:93`) to obtain raw bytes for hashing, then calls `parser.ParseServiceDeclaration(serviceYAMLPath)` (`reconciler.go:94`) which reads the same file independently. If the file is modified between the two reads, the hash stored in state will not correspond to the declaration that was actually compiled. The parser should accept the already-read bytes, or the hash should be computed from the same read.

- [process-revision-complete] **Finding 5: Stale comment on `hasEntityChanges`.** The doc comment on `hasEntityChanges()` (`reconciler.go:561`) reads `// HasChanges returns true if the plan includes any generate, update, or entity changes.` — this is the doc for `HasChanges`, not `hasEntityChanges`. The comment was copy-pasted and not updated to reflect the actual function.

- [process-revision-complete] **Finding 6: `runApply` and `FormatPlan` disagree on what constitutes "no changes".** `runApply` (`cmd/stego/main.go:126`) gates apply on `plan.HasChanges()` only, which checks file actions. `FormatPlan` (`reconciler.go:568`) checks both `plan.HasChanges()` and `plan.hasEntityChanges()`. If entity changes exist but all files happen to hash the same (e.g. cosmetic service.yaml edits that don't affect output), `runApply` prints "No changes" and skips, while `FormatPlan` would report entity changes. The two code paths should use the same predicate.

- [process-revision-complete] **Finding 7: go.mod header bypasses `gen.File.Bytes()` mechanism.** `generateGoMod` (`assembler.go:83`) embeds the `// Code generated by stego. DO NOT EDIT.` header directly in `Content`. All other generated files rely on `gen.File.Bytes()` to prepend the header for `.go` files. This means the go.mod's header is part of `Content` and will be included in the file hash computed by `computePlan`, while for `.go` files the header is added by `Bytes()` after hashing via `Content`. This creates an inconsistency: the hash for go.mod includes the header, but the hash for .go files also includes the header (since `computePlan` calls `f.Bytes()`). Functionally correct, but the dual mechanism is fragile — any change to the header text requires updating both `gen.Header` and the hardcoded string in `generateGoMod`.

## Round 2

- [process-revision-complete] **Finding 8: Orphaned files not detected by plan and not removed by apply.** `computePlan` (`reconciler.go:444-468`) only iterates over `generatedFiles`. Files tracked in the previous state (`existingHashes`) that are no longer produced by any generator are silently dropped from the new state and left on disk. The acceptance criteria state "plan shows accurate changeset" — omitting deletions makes the changeset incomplete. Concretely: if an entity is removed from `service.yaml` between applies, the handler file for that entity (e.g. `internal/api/handler_widget.go`) remains on disk referencing types that no longer exist. The acceptance criteria also state "`go build` works on generated output" — orphaned files with stale imports will break compilation.

- [process-revision-complete] **Finding 9: `computePlan` hardcodes output directory, `Apply` accepts it as parameter.** `computePlan` at `reconciler.go:453` uses `filepath.Join(projectDir, "out", f.Path)` to check existing files on disk. `Apply` at `reconciler.go:231` accepts `outDir` as a separate parameter: `func Apply(plan *Plan, projectDir, outDir string) error`. If a caller passes a non-default `outDir` to `Apply`, the plan's on-disk file checks will have been computed against the wrong directory. The `computePlan` function (or `Reconcile`, which calls it) should accept the output directory as a parameter to match `Apply`'s contract.

- [process-revision-complete] **Finding 10: Non-deterministic ordering of deleted entity changes in plan output.** In `computeEntityChanges` (`reconciler.go:579-586`), deleted entities are detected by iterating over the `oldEntities` map (`for entityName, oldFields := range oldEntities`). Go map iteration order is non-deterministic. When multiple entities are simultaneously deleted between applies, the order of `EntityChange` entries in the plan varies between runs, producing non-reproducible plan output. Added and modified entities are iterated via the `entities` slice (deterministic). The deleted-entity changes should be sorted (e.g. by entity name) before appending to maintain deterministic output consistent with the rest of the plan.

## Round 3

- [process-revision-complete] **Finding 11: Plan output omits field types shown in spec example.** The spec's compiler process section (spec.md:450) shows plan output as `+ field: display_name (string)` — field name followed by type in parentheses. `FormatPlan` at `reconciler.go:641` outputs `fmt.Fprintf(&result, "    + field: %s\n", f)` using only the field name string from `EntityChange.Added`, omitting the type entirely. Root cause: `AppliedState.Entities` is typed as `map[string][]string` (`state.go:33`) — a mapping of entity name to field names only. Since the type information is discarded when building the entity snapshot (`reconciler.go:509-515`), `FormatPlan` cannot include it. The plan output format does not match the spec.

- [process-revision-complete] **Finding 12: Entity field type and constraint changes invisible to plan.** `computeEntityChanges` (`reconciler.go:545-624`) compares entity snapshots stored as `map[string][]string` — field names only. If a field's type changes (e.g., `string` → `int32`), its constraints change (e.g., adding `max_length: 255`), or its attributes change (e.g., toggling `computed: true`), the field name is present in both old and new snapshots, so the change is not detected — `computeEntityChanges` reports zero changes. The task description states "Show entity field changes." A type or constraint modification is an entity field change that goes undetected. Concretely: changing `{ name: score, type: string }` to `{ name: score, type: int32 }` between applies produces a plan with no entity changes, even though the generated models, store queries, migration SQL, and OpenAPI schema all change.

- [process-revision-complete] **Finding 13: HTTP port hardcoded, ignoring component config resolution.** `Reconcile` at `reconciler.go:158` sets `httpPort := 8080` instead of resolving the port from the rest-api component's resolved config. The rest-api component declares `config: port: { type: int, default: 8080 }` in `registry/components/rest-api/component.yaml:7-9`. The reconciler already resolves component config via `resolveComponentConfig` (`reconciler.go:306-324`) and passes the result to generators via `ctx.ComponentConfig`, but bypasses this mechanism when constructing `AssemblerInput.Port` at `reconciler.go:211-212`. A service-level port override (e.g., `overrides: rest-api: port: 9090`) is silently ignored — the generated `main.go` always listens on `:8080`.

## Round 4

- [process-revision-complete] **Finding 14: `Reconcile` does not pass service-level port binding overrides to `ports.Resolve`.** The `ports.ResolveInput` struct declares a `ServiceOverrides` field (`ports/resolve.go:88-89`) for port→component mapping overrides from the service declaration, and `ports.Resolve` merges them with archetype bindings (`ports/resolve.go:101-108`). However, `Reconcile` at `reconciler.go:145-151` constructs `ResolveInput` with only `Components` and `ArchetypeBindings` — `ServiceOverrides` is left nil. The spec's "Port Resolution" section (spec.md:180-181) states: "The service declaration can override any binding via `overrides:`" and shows the example `overrides: auth-provider: api-key-auth` (spec.md:187). String-valued entries in `svcDecl.Overrides` represent port binding overrides (as opposed to map-valued entries which represent component config overrides), but the reconciler never extracts or forwards them. A service declaration that overrides a port binding is silently ignored — ports resolve using only archetype defaults.

- [process-revision-complete] **Finding 15: Namespace validation skipped for components without `output_namespace`.** `Reconcile` at `reconciler.go:214-217` gates namespace validation on `comp.OutputNamespace != ""`: if a component has no declared namespace, its generated files are accepted without path validation. The spec states "Each component declares an `output_namespace` ... and can only write files under it — the compiler rejects any file outside the namespace" (spec.md:268). Two MVP registry components — `health-check` (`registry/components/health-check/component.yaml`) and `otel-tracing` (`registry/components/otel-tracing/component.yaml`) — have no `output_namespace`. Their generators are currently no-op stubs returning nil files, so no violation occurs today. But the reconciler's conditional skip means a future non-stub generator for these components could write files anywhere — including under `fills/` (human-owned, never overwritten per spec.md:278) or `cmd/` (compiler-owned shared files) — without detection.

- [process-revision-complete] **Finding 16: No duplicate file path detection across generators and shared files.** `Reconcile` at `reconciler.go:220-222` appends each generator's files to `allFiles`, then appends slot files (`reconciler.go:230`) and assembler shared files (`reconciler.go:246`). No check verifies that file paths are unique across these sources. If two generators produce a file at the same path — or a generator produces a file that collides with a compiler-owned shared file (e.g., `cmd/main.go` or `go.mod`) — the later entry silently overwrites the earlier one in `Apply` (`reconciler.go:271-280`). The spec states "the archetype validates at authoring time that no two component namespaces overlap" (spec.md:269), which prevents this for well-formed archetypes with declared namespaces. But the reconciler performs no runtime collision check, so misconfigured archetypes or generators from namespace-less components (Finding 15) can silently lose generated content without any error or warning.

## Round 5

- [process-revision-complete] **Finding 17: `computePlan` does not verify on-disk file presence when state hash matches generated hash.** When `existingHash == hash` at `reconciler.go:534`, the plan immediately marks the file as `ActionUnchanged` without checking whether the file actually exists on disk. If a generated file is manually deleted from the `out/` directory between applies, and the service.yaml has not changed (so the generated content hash equals the state hash), `stego plan` reports "No changes" and `stego apply` prints "No changes. Infrastructure is up-to-date." (`cmd/stego/main.go:139-141`) and returns without calling `Apply`. The deleted file is never recreated. Contrast with the `!exists` branch at `reconciler.go:521-533`, which correctly calls `os.Stat(diskPath)` to detect whether the file needs to be generated. The acceptance criteria state "plan shows accurate changeset" — a missing file should appear as `generate`, not be silently omitted. Reproduction: first `stego apply` generates `out/internal/api/handler.go`. User runs `rm out/internal/api/handler.go`. Next `stego plan` reports "No changes." Next `stego apply` does nothing. The file remains missing and `go build` fails.

- [process-revision-complete] **Finding 18: Generated `go.mod` placement in `out/` makes fill imports unresolvable.** The assembler generates `go.mod` at file path `go.mod` (`assembler.go:87`), which `Apply` writes to `out/go.mod`, making `out/` the Go module root. The assembler also generates fill imports in `cmd/main.go` as `<module>/fills/<name>` (e.g., `github.com/myorg/svc/fills/admin-creation-policy`) at `assembler.go:292-294`. The Go toolchain resolves imports within the main module relative to the module root directory (where `go.mod` lives). It will look for `out/fills/admin-creation-policy/` — but the spec's project structure (spec.md:279-294) places fills at the project root: `project/fills/`, which is one directory level above `out/`. Go modules cannot redirect sub-paths of the main module via `replace` directives. Therefore, fill import paths are structurally unresolvable: the packages exist at `../fills/<name>/` relative to the module root but the Go toolchain has no mechanism to find them there. The acceptance criteria state "`go build` works on generated output." When slot bindings are configured (which is required by MVP scope — spec.md:523-525 mandate "at least one gate fill" and "at least one fan-out fill"), `go build ./cmd/...` from `out/` fails with package-not-found errors regardless of whether the fill code exists.

## Round 6

- [process-revision-complete] **Finding 19: Assembler middleware wrapping generates `.Middleware(mux)` method call on a function value — compile error.** `writeServerStart` at `assembler.go:691` generates `fmt.Fprintf(buf, "\tlog.Fatal(http.ListenAndServe(addr, %s.Middleware(mux)))\n", middlewareVar)`. The middleware variable is derived from the constructor expression `auth.NewAuthMiddleware()` via `findMiddlewareVar` and `rawConstructorVarName`, producing variable name `authMiddleware`. The generated code is: `log.Fatal(http.ListenAndServe(addr, authMiddleware.Middleware(mux)))`. However, the jwt-auth generator (`jwtauth/generator.go:139`) defines the constructor as `func NewAuthMiddleware() func(http.Handler) http.Handler` — the return type is an anonymous function type `func(http.Handler) http.Handler`. Anonymous function types have no methods. Calling `.Middleware(mux)` on a value of type `func(http.Handler) http.Handler` produces: `authMiddleware.Middleware undefined (type func(http.Handler) http.Handler has no field or method Middleware)`. The correct invocation is `authMiddleware(mux)` — calling the function directly. Any service using the `rest-crud` archetype will include `jwt-auth` via `default_auth` (`registry/archetypes/rest-crud/archetype.yaml:13`), triggering middleware wiring. The `TestAssemble_WithAuthMiddleware` test (`assembler_test.go:357`) verifies the string `authMiddleware.Middleware(mux)` is present but only performs syntax parsing via `parser.ParseFile` — syntax parsing does not catch type errors. The acceptance criteria state "`go build` works on generated output" — the generated `cmd/main.go` does not compile.

- [process-revision-complete] **Finding 20: `Reconcile` does not validate that declared mixins are in the archetype's `compatible_mixins` list.** The spec defines `compatible_mixins` on archetypes (spec.md:126-128): `compatible_mixins: [event-publisher, async-worker]`. The `Archetype` type has a `CompatibleMixins []string` field (`types.go:131`). The reconciler's `collectComponentNames` at `reconciler.go:336-348` loads mixin components from the registry and adds them to the component list, but never checks whether each declared mixin appears in `archetype.CompatibleMixins`. A service declaring `mixins: [incompatible-mixin]` where the archetype's `compatible_mixins` does not include `incompatible-mixin` is silently accepted — no error is returned, and planning proceeds with the incompatible mixin's components. The task description states "Validate service.yaml against registry before planning." The `compatible_mixins` field is an archetype-defined constraint that restricts which mixins may be used with it. Not validating this constraint means the reconciler accepts service declarations that violate the archetype's compatibility contract.

## Round 7

- [process-revision-complete] **Finding 21: `compatible_mixins: []` (explicitly empty list) bypasses mixin compatibility validation.** `collectComponentNames` at `reconciler.go:337` gates the mixin compatibility check on `len(archetype.CompatibleMixins) > 0`. When an archetype declares `compatible_mixins: []` — an explicitly empty list — `len(archetype.CompatibleMixins)` is 0 (Go's `gopkg.in/yaml.v3` unmarshals `[]` to an initialized empty slice `[]string{}`, not nil), so the entire validation block is skipped. A service can then use any mixin despite the archetype explicitly declaring that no mixins are compatible. The semantic contract of `compatible_mixins` is a whitelist: an empty whitelist means "nothing is allowed." The current code conflates "no `compatible_mixins` field declared" (nil slice, meaning no constraint — any mixin is acceptable) with "empty `compatible_mixins` list declared" (empty slice, meaning zero mixins are compatible). The fix is to distinguish nil from empty: skip validation only when `archetype.CompatibleMixins == nil`, not when `len(archetype.CompatibleMixins) == 0`. The test `TestCollectComponentNames_NoCompatibleMixinsFieldAllowsAny` (`reconciler_test.go:1839`) verifies the nil case, but no test covers the `compatible_mixins: []` case.

- [process-revision-complete] **Finding 22: Generated `go.mod` replace directives are ineffective — fill imports are structurally unresolvable.** The assembler generates `replace` directives in `go.mod` (assembler.go:92-99) to redirect fill imports from `<module>/fills/<name>` to `../fills/<name>`. This was the fix for finding 18 (round 5). However, this approach is structurally broken because fill import paths (e.g., `github.com/myorg/svc/fills/admin-creation-policy`) are sub-paths of the main module (`github.com/myorg/svc`). Go's module resolution treats any import matching the main module path prefix as an **intra-module package**, resolving it relative to the module root directory (`out/`). Go does not consult `replace` directives for intra-module packages — `replace` only applies to external module dependencies. Since `out/fills/admin-creation-policy/` does not exist, the import fails with a package-not-found error. The `replace` directive is never consulted. Additionally, even if fills were treated as separate modules (by having their own `go.mod` files — not shown in the spec's project structure at spec.md:279-294), the generated `go.mod` omits `require` directives for the fill modules. Go requires a `require` for every `replace` to add the module to the build graph (per Go module spec: "a replace directive alone does not add a module to the module graph"). Reproduction: configure any slot binding in `service.yaml` (e.g., `gate: [admin-creation-policy]`), run `stego apply`, then attempt `go build ./cmd/...` from the `out/` directory — the build fails regardless of whether the fill code exists. The acceptance criteria state "`go build` works on generated output" and the MVP scope requires "at least one gate fill" and "at least one fan-out fill" (spec.md:523-525).

## Round 8

- [process-revision-complete] **Finding 23: Slot binding with entity not in expose list produces unused variable — compile error.** `writeSlotWiring` at `assembler.go:537-542` builds an `injected` set from ALL entries in `slotVarsByEntity`, which includes every slot binding with a non-empty `Entity` field. This set determines whether a slot operator variable gets `_ =` suppression (line 560, 579, 594): variables in the set do NOT get `_ =` because they are assumed to be injected into a handler constructor. However, `writeConstructors` at `assembler.go:515-518` only injects slot variables into constructors whose `ConstructorEntities` entry matches the entity name. If a slot binding declares an entity that is not in the `expose` list — and therefore has no handler constructor — the slot operator variable is created, `_ =` is suppressed (because the entity is non-empty, placing it in `slotVarsByEntity` and thus `injected`), but no constructor receives it. The variable is declared and never used, which is a Go compile error. Neither `Reconcile` nor `Assemble` validates that slot binding entities correspond to entities in the expose list. Reproduction: declare `slots: [{slot: before_create, entity: Gadget, gate: [my-policy]}]` where "Gadget" is not in the `expose` list. `stego apply` generates `beforeCreateGadgetGate := slots.NewBeforeCreateGate(mypolicy.New())` without `_ =` and without injection into any constructor. `go build` fails: `beforeCreateGadgetGate declared and not used`. The acceptance criteria state "`go build` works on generated output" and the task description states "Validate service.yaml against registry before planning." Root cause: the `injected` set is computed from `slotVarsByEntity` (all entity-associated bindings) rather than from the intersection of `slotVarsByEntity` keys with constructor entity values — the set assumes every entity-associated binding has a matching constructor, which is only true when slot entities are a subset of exposed entities.

## Round 9

- [process-revision-complete] **Finding 24: Constructor ordering produces forward variable reference — compile error.** `writeConstructors` at `assembler.go:485-526` emits constructor variable declarations in wiring order, which directly follows the archetype's `components` list order. The `rest-crud` archetype (`registry/archetypes/rest-crud/archetype.yaml:6-10`) lists `rest-api` before `postgres-adapter`. The rest-api generator (`restapi/generator.go:145-146`) produces constructor expressions like `api.NewUserHandler(store)` that reference the `store` variable — this variable is created by the postgres-adapter's constructor `storage.NewStore(db)` (`postgresadapter/generator.go:115`). Since `rest-api` comes before `postgres-adapter` in the wiring order, the assembler emits `userHandler := api.NewUserHandler(store)` BEFORE `store := storage.NewStore(db)`. In Go, a short variable declaration (`:=`) introduces the variable at the point of declaration — referencing `store` before its declaration is a compile error: `undefined: store`. The generated `cmd/main.go` contains:
    ```go
    userHandler := api.NewUserHandler(store)
    store := storage.NewStore(db)
    ```
    `go build ./cmd/...` fails with `undefined: store`. The assembler tests (`assembler_test.go:78-81`, `assembler_test.go:172-175`) use `parser.ParseFile` to validate the generated code, but `parser.ParseFile` only checks syntax — it does not perform name resolution or type checking. Forward variable references are syntactically valid Go, so the tests pass despite the generated code being uncompilable. The acceptance criteria state "`go build` works on generated output." Reproduction: any service using the `rest-crud` archetype with at least one exposed entity triggers this — the rest-api handler constructor always references `store`, and the archetype always lists `rest-api` before `postgres-adapter`.

- [x] [process-revision-complete] **Finding 28: `computeConsumedConstructors` fallback variable reference detection uses `strings.Contains` instead of word-boundary-aware `containsIdentRef` — false positive dependency marking.** `computeConsumedConstructors` at `assembler.go:556-570` has a fallback for detecting transitive constructor dependencies when `ConstructorDeps` is not set. It extracts the argument portion of a constructor expression and checks whether any other constructor's base variable name appears as a substring: `if strings.Contains(args, depVar)` (line 563). The function `containsIdentRef` (assembler.go:592-607) exists in the same file and correctly checks word boundaries — it matches `name + "."` at identifier boundaries, preventing "store" from matching within "datastore". The fallback does not use word-boundary checking, so constructor `b.NewHandler(dataA)` (args = "(dataA)") would falsely match base variable name "a" (from `x.NewA()`), marking `x.NewA()` as transitively consumed. A false positive in consumption analysis causes the constructor to be emitted when it has no downstream consumer, producing an "unused variable" Go compile error.
