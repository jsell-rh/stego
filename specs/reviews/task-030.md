# Review: Task 030 — End-to-End Example with All rest-crud Spec Features

## Round 1

- [x] [process-revision-complete] **Error code prefix `USER` is wrong; spec requires `USERMANAGEMENT`.** The rest-crud spec states: "The error code prefix is derived from the service name (uppercased, hyphens removed): `hyperfleet-api` -> `HYPERFLEET`." The service name is `user-management`. Applying the rule: remove hyphens → `usermanagement`, uppercase → `USERMANAGEMENT`. All error codes in `out/internal/api/errors.go` use `USER` instead (e.g. `USER-NTF-001`, `USER-VAL-001`). The `errorForStatus` function at line 147 hardcodes `"USER"`. This commit updated the `Type` URIs in `errors.go` but did not correct the prefix.

- [x] [process-revision-complete] **`org-settings` collection URL path derived as `orgsettings`, inconsistent with `org-users` → `users`.** The `org-users` collection (scoped by `org_id: Organization`) generates the path segment `users` — the `org-` prefix is stripped. But `org-settings` (same scope) generates `orgsettings` — concatenated, prefix not stripped. Routes in `out/cmd/main.go:82-83` use `/organizations/{org_id}/orgsettings`. OpenAPI paths in `out/internal/api/openapi.json` also use `orgsettings`. Consistent path derivation would produce `/organizations/{org_id}/settings`.

- [x] [process-revision-complete] **OrgSetting GORM model missing unique composite index on `(org_id, key)`.** The service declaration specifies `upsert_key: [org_id, key]`. The store's Upsert method generates `ON CONFLICT (org_id, key)` via `clause.OnConflict{Columns: ...}`. PostgreSQL requires a unique constraint or unique index on the conflict columns. The OrgSetting model in `out/internal/storage/models.go:47-54` has no `uniqueIndex` tag for this pair, and AutoMigrate will not create one. The upsert will fail at runtime with a PostgreSQL error.

- [x] [process-revision-complete] **Organization API struct uses `string` for `description`; GORM model uses `*string`.** The service declaration marks `description` as `optional: true`. The GORM model correctly uses `*string` (`out/internal/storage/models.go:31`). But the API struct in `out/internal/api/router.go:63` uses `string`. When `description` is omitted from a create request, JSON decoding produces the zero-value `""`. After JSON marshal/unmarshal into the GORM struct, this becomes `*string` pointing to `""` — stored in the database as an empty string instead of NULL. The optional semantics are broken.

- [x] [process-revision-complete] **Patch handler does not fire `on_entity_changed` fan-out.** The `org-users` collection binds `on_entity_changed` with a fan-out to `user-change-notifier` and `audit-logger`. The Create handler (`handler_org_users.go:107-112`), Update handler (`:152-157`), and Delete handler (`:171-176`) all call `h.onEntityChangedFanOut.Evaluate(...)` after mutating a User entity. The Patch handler (`:294-329`) modifies the entity via `store.Replace` but does not invoke the fan-out. A partial update is still an entity change.

- [x] [process-revision-complete] **Upsert response returns a phantom UUID, not the existing record's ID.** In `handler_org_settings.go:161-163`, a new UUID is generated for `orgsetting.ID` when the field is empty. On an upsert that matches an existing row (via `ON CONFLICT (org_id, key)`), the existing row's `id` is preserved in the database — it is not one of the `DoUpdates` columns. But the handler returns `orgsetting.ID` (the freshly generated UUID) in the response body and `href` at line 181. The client receives an ID that does not exist in the database.

## Round 2

- [x] [process-revision-complete] **`Store.Get` does not translate `gorm.ErrRecordNotFound` to `api.ErrNotFound`; all not-found GETs return 500.** The `Storage` interface contract at `router.go:46-47` states: "Get must return ErrNotFound when the entity does not exist." GORM's `First` method returns `gorm.ErrRecordNotFound` when no row matches. The store's `Get` method (e.g. `store.go:70-71` for Organization) returns this raw GORM error without translation. In the handlers, `errors.Is(err, ErrNotFound)` (e.g. `handler_organizations.go:89`) checks against `api.ErrNotFound = errors.New("entity not found")`. Since `gorm.ErrRecordNotFound` is not wrapped with `api.ErrNotFound`, the check never matches. Every GET request for a non-existent entity falls through to the `InternalError` branch and returns HTTP 500 instead of 404. The string `gorm.ErrRecordNotFound` does not appear anywhere in the generated code — no translation is performed. This affects all three entity types (Organization, User, OrgSetting).

- [x] [process-revision-complete] **Patch handler panics at runtime: cross-package type assertion `existing.(User)` fails.** In `handler_org_users.go:309`, `user := existing.(User)` performs an unguarded type assertion. The handler is in package `api`, so `User` resolves to `api.User` (defined at `router.go:67-74`). But `store.Get` returns a value of concrete type `storage.User` (defined at `models.go:36-44`). These are different named types in different packages with different structures (`storage.User` embeds `Meta`, has `OrgIDRef`, uses `datatypes.JSON` for Metadata; `api.User` is a flat struct with `*json.RawMessage`). An unguarded type assertion against a mismatched type panics. Every PATCH request to `/organizations/{org_id}/org-users/{id}` will crash the server with a runtime panic.

- [x] [process-revision-complete] **OpenAPI schema requires scope field `org_id` in request body for scoped collections, causing spurious validation failures.** The `User` schema in `openapi.json:1030-1035` lists `org_id` in the `required` array. The `OrgSetting` schema at lines `958-963` does the same. The validation middleware (`validation.go:43-45`) validates POST/PUT/PATCH request bodies against these schemas before the handler runs. For scoped collections (`org-users`, `org-settings`), the handler derives `org_id` from the URL path parameter (e.g. `handler_org_users.go:64`). If a client omits `org_id` from the request body (relying on the URL path, as scoped routing intends), the validation middleware rejects the request with 400 before the handler ever sees it. Clients are forced to redundantly include `org_id` in the body. Additionally, if the body's `org_id` differs from the URL's `org_id`, the handler silently overwrites with the URL value — no conflict check.

- [x] [process-revision-complete] **Create handlers return 500 for unique constraint violations instead of 409 Conflict.** The spec's error handling table (rest-crud spec, Error Handling section) defines category CNF (Conflict, 409) for conflict errors. The `errors.go` file generates a `Conflict()` constructor for this case. But the Organization create handler (`handler_organizations.go:76-79`) and User create handler (`handler_org_users.go:104-107`) treat all `store.Create` errors as `InternalError(err.Error())`. When inserting a duplicate `name` (Organization, `uniqueIndex`) or duplicate `email` (User, `uniqueIndex`), PostgreSQL returns a unique constraint violation. The store passes this error through to the handler, which returns HTTP 500 with the raw database error message exposed to the client. It should return 409 Conflict with a sanitized message.

## Round 3

- [x] [process-revision-complete] **Upsert update-path response returns phantom data; cross-package type assertion always fails.** In `handler_org_settings.go:176-186`, after an update-type upsert (where `created` is false), the handler attempts to re-fetch the updated record by listing all OrgSettings and scanning for a match. The type assertion at line 178 — `listResult.Items.([]OrgSetting)` — asserts against `api.OrgSetting` (the handler is in package `api`), but `store.List` for "OrgSetting" returns `[]storage.OrgSetting` inside `ListResult.Items` (see `store.go:324-328`). These are different named types in different packages with different struct fields (`storage.OrgSetting` embeds `Meta`, has `OrgIDRef`, uses `datatypes.JSON`; `api.OrgSetting` is a flat struct with `json.RawMessage`). The assertion always fails (`ok` is always `false`), the if-block is never entered, and the response at line 194 returns the original `orgsetting` variable containing: (a) a phantom UUID generated at line 162 that does not exist in the database, and (b) client-submitted field values rather than the actual persisted state. This is the same root bug as Round 1 Finding #6 — the fix introduced a recovery path that can never execute.

- [x] [process-revision-complete] **`Store.Replace` does not translate unique constraint violations to `ErrConflict`.** The `Create` method in `store.go` correctly uses `isUniqueConstraintError()` to detect PostgreSQL unique constraint violations (SQLSTATE 23505) and return `api.ErrConflict` (lines 43-44 for Organization, 59-60 for User, 75-76 for OrgSetting). The `Replace` method (lines 123-161) does not perform this translation — it returns the raw GORM error directly. The Update handler (`handler_org_users.go:153-156`) and Patch handler (`:341-344`) both check `errors.Is(err, ErrConflict)` after calling `Replace`. Since `Replace` never wraps errors with `ErrConflict`, these checks never match. A PUT or PATCH that sets a User's `email` to a value already held by another User triggers a PostgreSQL unique constraint violation, which passes through as a raw GORM error to the `InternalError("internal error")` branch — returning HTTP 500 instead of the spec-defined 409 Conflict (CNF category, rest-crud spec Error Handling table).

- [x] [process-revision-complete] **Update handler returns 200 OK for non-existent entities and fires side effects.** The Update handler at `handler_org_users.go:141-168` calls `store.Replace` without first verifying the target entity exists. The `Replace` method uses `db.Model(&User{}).Where("id = ?", id).Select(fields).Updates(&v)` (`store.go:146`). When no row matches the WHERE clause, GORM's `Updates` returns `nil` error and 0 rows affected — it silently succeeds. The handler then: (1) fires `on_entity_changed` fan-out (lines 161-166) with `Action: "update"` for a mutation that never occurred, sending false-positive notifications to `user-change-notifier` and `audit-logger`; and (2) returns HTTP 200 with the request body presented as a persisted entity (line 168), including fabricated `id`, `kind`, and `href` envelope metadata. Contrast with the Patch handler (lines 303-316), which correctly calls `store.Get` first and returns 404 via `ErrNotFound` when the entity does not exist, before attempting any mutation.

## Round 4

- [-] [process-revision-complete] **Delete handler fires `on_entity_changed` fan-out for non-existent entities.** `Store.Delete` (`store.go:196-207`) calls `s.db.WithContext(ctx).Where("id = ?", id).Delete(&User{}).Error`. When no row matches the WHERE clause, GORM's soft-delete returns `nil` error with 0 rows affected — it does not distinguish "deleted a row" from "nothing matched." The org-users Delete handler (`handler_org_users.go:173-189`) treats a nil error as a successful deletion and unconditionally fires the `on_entity_changed` fan-out at line 182-186 with `Action: "delete"`, sending false-positive notifications to `user-change-notifier` and `audit-logger` for a mutation that never occurred. This is the same root cause as Round 3 Finding #3 (Update handler): the Store method does not check `RowsAffected` and does not return `ErrNotFound` when no row is affected. Round 3's fix added `RowsAffected == 0` → `ErrNotFound` to `Store.Replace`; the identical fix is missing from `Store.Delete`.

- [-] [process-revision-complete] **Slot request `Fields` map contains garbled data for `*json.RawMessage` fields.** The org-users Create handler at `handler_org_users.go:74` populates the slot request with `"metadata": fmt.Sprintf("%v", user.Metadata)`. `user.Metadata` is `*json.RawMessage` (which is `*[]byte`). When non-nil, `fmt.Sprintf("%v", ...)` dereferences the pointer and prints the raw byte-slice representation — e.g. `[123 34 107 101 121 34 58 34 118 97 108 34 125]` — instead of the JSON string `{"key":"val"}`. When nil, it produces the string `"<nil>"`. Any fill that accesses `req.Input.Fields["metadata"]` receives an unusable debug dump rather than the actual JSON value. The same pattern appears in the Organization Create handler (`handler_organizations.go:46`) for `description` (`*string`): when the optional field is omitted, `fmt.Sprintf("%v", organization.Description)` produces `"<nil>"` rather than an empty string or an absent key. The slot contract defines `Fields map[string]string`; the values serialized into this map must be meaningful string representations of the field values, not Go debug output from `%v` on pointer types.
