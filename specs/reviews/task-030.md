## Round 2

- [x] [process-revision-complete] **`Store.Get` does not translate `gorm.ErrRecordNotFound` to `api.ErrNotFound`; all not-found GETs return 500.** The `Storage` interface contract at `router.go:46-47` states: "Get must return ErrNotFound when the entity does not exist." - [x] [process-revision-complete] **Patch handler panics at runtime: cross-package type assertion `existing.(User)` fails.** In `handler_org_users.go:309`, `user := existing.(User)` performs an unguarded type assertion. The handler is in package `api`, so `User` resolves to `api.User` (defined at `router.go:67-74`). But `store.Get` returns a value of concrete type `storage.User` (defined at `models.go:36-44`). - [x] [process-revision-complete] **OpenAPI schema requires scope field `org_id` in request body for scoped collections, causing spurious validation failures.** The `User` schema in `openapi.json:1030-1035` lists `org_id` in the `required` array. - [x] [process-revision-complete] **Create handlers return 500 for unique constraint violations instead of 409 Conflict.** The spec's error handling table (rest-crud spec, Error Handling section) defines category CNF (Conflict, 409) for conflict errors. The `errors.go` file generates a `Conflict()` constructor for this case. The store passes this error through to the handler, which returns HTTP 500 with the raw database error message exposed to the client. It should return 409 Conflict with a sanitized message.

- [x] [process-revision-complete] **`Store.Replace` does not translate unique constraint violations to `ErrConflict`.** The `Create` method in `store.go` correctly uses `isUniqueConstraintError()` to detect PostgreSQL unique constraint violations (SQLSTATE 23505) and return `api.ErrConflict` (lines 43-44 for Organization, 59-60 for User, 75-76 for OrgSetting). - [x] [process-revision-complete] **Update handler returns 200 OK for non-existent entities and fires side effects.** The Update handler at `handler_org_users.go:141-168` calls `store.Replace` without first verifying the target entity exists. - [x] [process-revision-complete] **Slot request `Fields` map contains garbled data for `*json.RawMessage` fields.** The org-users Create handler at `handler_org_users.go:74` populates the slot request with `"metadata": fmt.Sprintf("%v", user.Metadata)`. `user.Metadata` is `*json.RawMessage` (which is `*[]byte`). The slot contract defines `Fields map[string]string`; the values serialized into this map must be meaningful string representations of the field values, not Go debug output from `%v` on pointer types.

## Round 7

- [x] [process-revision-complete] **Scoped collection Read/Update/Patch/Delete handlers don't enforce scope, allowing cross-scope data access.** The spec defines a collection as "a scoped, operation-constrained access pattern over an entity" (spec.md:17). The `org-users` collection has `scope: { org_id: Organization }`. The List handler correctly enforces scope at `handler_org_users.go:265`: `store.List(r.Context(), "User", "org_id", scopeValue, opts)` â€” only users belonging to the URL's organization are returned. However, all individual-resource operations bypass scope entirely:
  - **Read** (`handler_org_users.go:131`): `store.Get(r.Context(), "User", id)` queries by ID only. `GET /organizations/{orgB}/org-users/{user_in_orgA}` returns the user regardless of which org it belongs to â€” data leaks across scope boundaries.
  - **Update** (`handler_org_users.go:155-156`): sets `user.OrgID = r.PathValue("org_id")` then calls `store.Replace`. `PUT /organizations/{orgB}/org-users/{user_in_orgA}` silently reassigns the user from Org A to Org B â€” a destructive cross-scope mutation with no error, no check, no audit trail.
  - **Patch** (`handler_org_users.go:317`): `store.Get(r.Context(), "User", id)` reads the entity without verifying it belongs to the URL's org. The patch is applied to a user from any org. Unlike Update, Patch preserves the original `org_id` (doesn't overwrite from URL), creating an inconsistency: PATCH and PUT behave differently regarding scope assignment for the same entity through the same scoped collection endpoint.
  - **Delete** (`handler_org_users.go:181`): `store.Delete(r.Context(), "User", id)` deletes by ID only. `DELETE /organizations/{orgB}/org-users/{user_in_orgA}` deletes the user and fires `on_entity_changed` fan-out â€” regardless of org membership.
  The root cause is structural: the `Storage` interface (`router.go:49-51`) defines `Get(entity, id)`, `Replace(entity, id, value)`, and `Delete(entity, id)` with no scope parameters, while `List` has `scopeField` and `scopeValue`. The handlers cannot enforce scope even if they wanted to, because the storage contract doesn't support scoped individual-resource operations. The `checkAncestors` method verifies the parent Organization exists but does not verify the child User belongs to that parent's scope.

- [x] [process-revision-complete] **Read and List responses include undeclared `created_time` and `updated_time` fields; Create/Update/Patch responses do not â€” response shape is inconsistent across operations.** The spec's response format (rest-crud spec "Response Format" section) defines single resource responses as entity fields plus envelope metadata (`id`, `kind`, `href`). - [x] [process-revision-complete] **OpenAPI schema types `jsonb` fields as `"type": "object"`, causing the validation middleware to reject valid JSON arrays, strings, numbers, and booleans.** The entity type system (spec.md "Entity Type System" section) defines `jsonb` as a stego-specific type for arbitrary JSON data.
