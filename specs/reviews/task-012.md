# Review: Task 012 — Fill Wiring & main.go Assembly

## Round 1

- [x] [process-revision-complete] **1. Slot operators are created but never wired into handler constructors.** The assembler generates operator variables (e.g. `beforeCreateGate := slots.NewBeforeCreateGate(...)`) but immediately discards them with `_ = varName`. The spec states: "An auditor reads `main.go` and sees every fill and every connection" and shows fills passed directly to handler constructors (`api.NewUserHandler(userService, adminpolicy.New(), ...)`). The acceptance criteria requires "Generated main.go shows all fills and connections." The `_ =` pattern shows fills but no connections — the operators are orphaned. (`internal/compiler/assembler.go:242,258,269`)

- [x] [process-revision-complete] **2. `fmt` is always imported regardless of whether it is used.** `writeMainImports` unconditionally adds `"fmt"` to standard library imports (line 142). `fmt` is only used in `writeServerStart` (via `fmt.Sprintf`), which is only called when `hasRoutes` is true. When a service has no routes, the generated `main.go` contains an unused import, which is a Go compile error. The test `TestAssemble_NoRoutes` does not catch this because `go/parser.ParseFile` performs only syntax checking, not semantic analysis. (`internal/compiler/assembler.go:142`, `internal/compiler/assembler_test.go:370-380`)

- [x] [process-revision-complete] **3. `constructorVarName` can produce duplicate variable names.** If two component wirings contain constructors with the same derived name (e.g. `a.NewStore(db)` and `b.NewStore(db)` both produce `store`), the generated `main.go` will contain two `:=` assignments to the same variable, which is a Go compile error. There is no collision detection or disambiguation. (`internal/compiler/assembler.go:348-369`)

- [x] [process-revision-complete] **4. `fillImportAlias` can produce collisions for distinct fill names.** The alias is derived by stripping `-` and `_` and lowercasing: fill names `"a-b"` and `"ab"` both produce alias `"ab"`. Two fills with colliding aliases would generate duplicate import aliases in the same import block, which is a Go compile error. (`internal/compiler/assembler.go:373-377`)

- [x] [process-revision-complete] **5. `needsDB` uses fragile string matching against opaque constructor expressions.** It checks for the literal substrings `(db)` and `(db,` (lines 321-322). A constructor expression with whitespace (e.g. `NewStore( db )`) or a parameter named `dbs` would not match, silently omitting the required `database/sql` setup. The convention that component generators must emit exactly `(db)` or `(db,` is undocumented. (`internal/compiler/assembler.go:315-327`)

- [x] [process-revision-complete] **6. Component import alias collisions.** `writeMainImports` derives the import alias from `path.Base(imp)` (line 168). Two components with different import paths sharing the same base (e.g. `internal/api/models` and `internal/storage/models`) would produce duplicate `models` aliases, which is a Go compile error. (`internal/compiler/assembler.go:167-168`)

## Round 2

- [x] [process-revision-complete] **7. Import alias namespace collision between component, fill, and slots imports.** Component imports, fill imports, and the slots package import all share the same Go import block but use separate disambiguation maps (`aliases`/`aliasUsed` for components at lines 168–169, `fillAliases`/`fillAliasUsed` for fills at lines 195–196). A fill named `"api"` produces alias `api` via `rawFillImportAlias`, which collides with a component import `internal/api` (also alias `api`). Similarly, the slots package uses a hardcoded alias `slots` (line 190) without checking either disambiguation map. Concrete example: a service with component import `internal/storage` and a fill named `storage` produces two `storage` aliases in the same import block — Go compile error. (`internal/compiler/assembler.go:166–203`)

- [x] [process-revision-complete] **8. Constructor variable name disambiguation breaks route variable references.** Routes are opaque string expressions (e.g. `mux.HandleFunc("POST /users", userHandler.Create)`) that reference variables derived from constructor expressions by `rawConstructorVarName`. `writeConstructors` disambiguates colliding variable names (e.g. `userHandler` → `userHandler2`), but routes are emitted verbatim by `writeRouteRegistration` and still reference the original name. If two handler constructors from different component wirings produce the same base name, routes from the second component reference the first component's handler variable. `TestAssemble_ConstructorVarNameCollision` only tests colliding `NewStore` constructors whose variables are not referenced by any routes — it does not cover handler constructor collisions where routes bind to the wrong variable. (`internal/compiler/assembler.go:227–253, 329–339`)

- [x] [process-revision-complete] **9. `log` is unconditionally imported but only used when `hasDB` or `hasRoutes`.** `writeMainImports` always adds `"log"` (line 155), but `log` is only used in `writeDBSetup` (`log.Fatal`) and `writeServerStart` (`log.Printf`, `log.Fatal`). A service with neither routes nor a database (e.g. slot-only wiring or a stub component) produces an unused `log` import — Go compile error. This is the same class of bug as the `fmt` import issue fixed in round 1 (finding 2), where the fix was to conditionalize the import. (`internal/compiler/assembler.go:155`)

## Round 3

- [x] [process-revision-complete] **10. Duplicate slot operator variable names when the same slot is bound to different entities.** `writeSlotWiring` derives the operator variable name solely from the slot name and operator type (e.g. `snakeToCamel(sb.Slot) + "Gate"`). Two slot bindings with the same slot name but different entities — e.g. `{Slot: "before_create", Entity: "User", Gate: ["user-policy"]}` and `{Slot: "before_create", Entity: "Organization", Gate: ["org-policy"]}` — both emit `beforeCreateGate := ...` in the same function scope. This is a Go compile error (duplicate short variable declaration). `buildSlotVarsByEntity` compounds the problem: it maps both entities to the same variable name, so even if the duplicate declaration were somehow resolved, both handlers would reference the same operator variable. The spec's service declaration example (lines 345–349) binds `before_create` to a single entity, but nothing in the spec prohibits per-entity bindings for the same slot — and the `SlotDeclaration` type accepts it. (`internal/compiler/assembler.go:275–346, 537–557`)

- [x] [process-revision-complete] **11. Slot operator variable names are invisible to constructor variable disambiguation.** Slot operators are emitted by `writeSlotWiring` before constructors, producing variables like `beforeCreateGate`, `onEntityChangedFanOut`, etc. `writeConstructors` disambiguates constructor variable names using its own `varNames`/`varUsed` maps (line 240–241), but these maps are not seeded with the slot operator variable names. A component whose constructor derives the same variable name as a slot operator (e.g. a constructor `pkg.NewBeforeCreateGate(...)` produces `rawConstructorVarName` → `beforeCreateGate`) would emit a duplicate `:=` assignment — a Go compile error. The two namespaces (slot operator vars and constructor vars) share the same Go function scope but use independent collision-tracking maps. (`internal/compiler/assembler.go:239–272, 275–346`)

- [x] [process-revision-complete] **12. Entity-to-constructor matching relies on undocumented `New<Entity>Handler` naming convention.** `extractEntityFromConstructor` (line 492–507) hardcodes the pattern `New<Entity>Handler` to determine which constructors receive slot operator injection. If a component generator names its handler constructors differently (e.g. `NewUserController`, `NewUserResource`, `NewUserAPI`), slot operators are silently orphaned with `_ = varName` and the generated `main.go` shows fills but no connections — violating the spec requirement that "An auditor reads `main.go` and sees every fill and every connection." This is the same class of fragile convention-based matching as finding 5 from round 1 (`needsDB` matching `(db)` in constructor strings), which was resolved by adding structured metadata (`NeedsDB bool`) to the `Wiring` struct. The entity-to-constructor relationship should similarly be expressed as structured metadata (e.g. a map from entity name to constructor index) rather than inferred from naming conventions. (`internal/compiler/assembler.go:492–507, internal/gen/gen.go:71–87`)

## Round 4

- [-] [process-revision-complete] **13. Assembler-internal variable names (`mux`, `addr`, `db`, `dsn`, `err`) are not reserved in the constructor variable disambiguation namespace.** `writeConstructors` seeds its disambiguation maps with slot operator variable names (via `allSlotVarNames`, fixing finding 11), but does NOT seed them with the variable names that `writeDBSetup`, `writeRouteRegistration`, and `writeServerStart` introduce into the same function scope. These assembler-internal variables are: `dsn` and `db` and `err` (from `writeDBSetup`, when `hasDB`), `mux` (from `writeRouteRegistration`, when `hasRoutes`), and `addr` (from `writeServerStart`, when `hasRoutes`). A component constructor like `muxutil.NewMux()` produces `rawConstructorVarName` → `mux`, which is emitted as `mux := muxutil.NewMux()`. Then `writeRouteRegistration` emits `mux := http.NewServeMux()` — a duplicate short variable declaration in the same scope, which is a Go compile error. This is the exact same class of bug as finding 11 (slot vars invisible to constructor disambiguation) and requires the same fix: seed the constructor disambiguation maps with the assembler-internal variable names that will be declared later in the function. Reproduction: any `AssemblerInput` with `hasRoutes=true` and a constructor whose `rawConstructorVarName` is `"mux"`. (`internal/compiler/assembler.go:265–276`)

## Round 5

- [-] [process-revision-complete] **14. Disambiguated component import aliases are not propagated to constructor and route expressions.** When two component imports share the same `path.Base` (e.g. `internal/api/models` and `internal/storage/models`), the second import gets a disambiguated alias (`models2`). However, the constructor expressions from that component still reference the original package name. In `writeConstructors` (line 327), the constructor expression is emitted verbatim: `fmt.Fprintf(buf, "\t%s := %s\n", varName, expr)` — `expr` retains the original package reference (e.g. `models.NewBar()`), but the import alias for that component's package is now `models2`. The generated code calls `models.NewBar()` which resolves to component A's package, not component B's. This produces either a compile error (if `NewBar` doesn't exist in component A's package) or silently calls the wrong function. The same issue affects the `TestAssemble_SlotsAliasCollisionWithComponent` scenario: a component with import `internal/slots` gets alias `slots2`, but its constructor `slots.NewProcessor()` still references the `slots` alias (which is the slot operators package). `TestAssemble_ComponentImportAliasCollision` passes because `parser.ParseFile` only validates syntax, not import resolution. Reproduction: any `AssemblerInput` with two component imports sharing the same `path.Base`. (`internal/compiler/assembler.go:190–203, 303–329`)

- [-] [process-revision-complete] **15. `applyVarRenames` uses substring matching, which corrupts route expressions when a renamed variable name is a suffix of another variable name.** `applyVarRenames` (line 430) replaces `oldName + "."` with `newName + "."` via `strings.ReplaceAll`, which matches anywhere in the string, not at word boundaries. Within a single wiring, if constructor A produces variable `store` (renamed to `store2` due to collision with another wiring) and constructor B produces variable `datastore` (no rename), a route referencing `datastore.Get` is corrupted to `datastore2.Get` because the substring `store.` appears within `datastore.`. The variable `datastore2` does not exist — the generated code fails to compile. Reproduction: wiring 0 has constructor `a.NewStore(db)` → var `store`; wiring 1 has constructors `b.NewStore(db)` → var `store` → renamed `store2`, and `b.NewDatastore()` → var `datastore`, with route `mux.HandleFunc("GET /data", datastore.Get)`. After `applyVarRenames`, the route becomes `datastore2.Get`. (`internal/compiler/assembler.go:429–437`)

- [-] [process-revision-complete] **16. Standard library import aliases (`log`, `fmt`, `http`, `os`, `sql`) are not reserved in the constructor variable disambiguation namespace.** `assemblerInternalVars` (line 264) reserves assembler-emitted variable names (`mux`, `addr`, `db`, `dsn`, `err`) but does not reserve standard library import aliases that are used in the generated function body. A constructor like `logger.NewLog()` produces `rawConstructorVarName` → `log`, which is emitted as `log := logger.NewLog()`. In Go, this variable shadows the `log` package import from the point of declaration onward. `writeServerStart` emits `log.Printf(...)` and `log.Fatal(...)` AFTER the constructors — these now reference the constructor variable (a `*logging.Log` or similar), not the `log` package, producing a type error. `writeDBSetup` emits `log.Fatal(err)` BEFORE constructors, so it is unaffected. This is the same class of bug as finding 13 (assembler-internal vars not reserved), extended to import aliases. (`internal/compiler/assembler.go:264–276, 449–458`)

## Round 6

- [x] [process-revision-complete] **17. Intra-wiring constructor rename map overwrites misdirect route and middleware references.** `writeConstructors` stores disambiguation renames in `wiringRenames[i]` as `map[string]string`, keyed by the base variable name (line 365: `wiringRenames[i][baseVar] = varName`). When a single wiring contains multiple constructors deriving the same base name — e.g. `pkg.NewHandler(storeA)` and `pkg.NewHandler(storeB)` both derive `handler` — the first constructor keeps `handler` (no rename entry), the second is renamed to `handler2` (entry: `"handler" → "handler2"`). `applyVarRenames` then replaces ALL `handler.` references in that wiring's routes with `handler2.`, including routes that intended to reference the first constructor's variable `handler`. Concrete reproduction: a wiring with constructors `["pkg.NewHandler(a)", "pkg.NewHandler(b)"]` and routes `["mux.HandleFunc(\"GET /a\", handler.GetA)"]`. The generated route becomes `handler2.GetA` — bound to the wrong handler. The same issue affects `writeServerStart`: if `MiddlewareConstructor` points to the first constructor and a later constructor in the same wiring produces the same base name, `findMiddlewareVar` returns the raw name, and the rename lookup redirects `middlewareVar` to the wrong variable. (`internal/compiler/assembler.go:360-366, 488-493, 532-540`)

- [x] [process-revision-complete] **18. Duplicate (slot, entity, operator) bindings emit duplicate `:=` declarations — Go compile error.** If `SlotBindings` contains two entries with identical `Slot`, `Entity`, and operator type (e.g., both have non-empty `Gate`), `writeSlotWiring` emits two `:=` declarations for the same variable name (e.g., `beforeCreateUserGate := slots.NewBeforeCreateGate(...)` twice in the same scope). This is a Go compile error (duplicate short variable declaration). `go/parser.ParseFile` does not catch this because it performs syntax checking only, not semantic analysis. `buildSlotVarsByEntity` compounds the problem by appending the same variable name to the entity's injection list, producing constructor calls with duplicate arguments (e.g., `api.NewUserHandler(store, beforeCreateUserGate, beforeCreateUserGate)`). Neither `writeSlotWiring`, `buildSlotVarsByEntity`, nor `collectAllSlotVarNames` validates uniqueness of the (slot, entity, operator) triple. The assembler handles many other collision classes (constructor vars, import aliases, assembler-internal vars) but does not handle this one. (`internal/compiler/assembler.go:384-456, 669-689`)

## Round 7

- [-] [process-revision-complete] **19. Non-stdlib import alias disambiguation does not reserve stdlib import names — component/fill aliases can shadow stdlib imports.** `writeMainImports` writes stdlib imports (e.g., `"database/sql"`, `"log"`, `"fmt"`, `"net/http"`, `"os"`) as bare strings at lines 165–185 without registering their implicit package-name aliases in the non-stdlib disambiguation maps (`aliases`/`aliasUsed` at lines 191–192). These maps start empty. If a component has import path base name matching a stdlib import — e.g., `internal/sql` (base "sql") — then `disambiguateAlias("sql", aliases, aliasUsed)` returns `"sql"` (first use in the non-stdlib map). The generated import block becomes `"database/sql"` (implicit alias `sql`) alongside `sql "github.com/myorg/svc/internal/sql"` (explicit alias `sql`). The explicit alias shadows the implicit one: all `sql.Open(...)` references in `writeDBSetup` now resolve to the component package, not `database/sql`. This is a semantic error (wrong package reference + unused import) that `parser.ParseFile` does not catch — it only validates syntax. The same issue applies to fills: a fill named `"log"` produces `rawFillImportAlias` → `"log"`, which shadows the stdlib `"log"` import when `hasDB || hasRoutes`. The `assemblerInternalVars` function (lines 294–323) correctly reserves these names for CONSTRUCTOR variable disambiguation (finding 16 fix), but the IMPORT ALIAS disambiguation maps are a completely separate namespace that lacks this reservation. Fix: seed the import alias disambiguation maps with the conditionally-imported stdlib package names before processing slots, component, and fill imports. Reproduction: any `AssemblerInput` with `hasDB=true` and a component import whose `path.Base` is `"sql"` or `"os"`, or `hasRoutes=true` with base `"fmt"`, `"http"`, or `"log"`. (`internal/compiler/assembler.go:161–251`)

- [-] [process-revision-complete] **20. Slot variable name collision from distinct slot names that normalize to the same camelCase identifier.** `slotVarName` (line 875) derives operator variable names via `snakeToCamel(slot) + entity + operatorSuffix`. The `snakeToCamel` function splits on `"_"` and skips empty segments (line 796: `if p == "" { continue }`). Two slot names that differ only in underscore structure — e.g., `"before_create"` and `"before__create"`, or `"before_create"` and `"beforeCreate"` (if the parser accepts camelCase slot names) — produce the same `snakeToCamel` output `"beforeCreate"`, and therefore the same operator variable name `"beforeCreateUserGate"`. `validateSlotBindingUniqueness` (line 838) checks uniqueness on the raw `(slot, entity, operator)` composite key, which treats these as distinct bindings. `writeSlotWiring` then emits two `:=` declarations for the same variable name in the same function scope — a Go compile error (duplicate short variable declaration). `parser.ParseFile` does not catch this because redeclaration is a semantic error. This is the same collision class as finding 10 (same slot, different entities) but on a different axis: different slots, same entity, normalized to the same variable name. The validator should check derived variable name uniqueness, not just raw composite key uniqueness. (`internal/compiler/assembler.go:838–869, 875–881, 790–810`)
