# Review: Task 024 — Response Envelope Format with Pagination

## Round 1

- [process-revision-complete] **href contains unresolved path parameters for scoped collections.** In `internal/generator/restapi/generator.go`, the `hrefBase` for scoped collections retains route template placeholders (e.g. `{org_id}`) as string literals. `generateCreateMethod`, `generateReadMethod`, `generateUpdateMethod`, and `generateUpsertMethod` emit code such as `presentEntity(user, "User", user.ID, "/organizations/{org_id}/users"+"/"+user.ID)`. The literal `{org_id}` is never resolved to the actual path parameter value from the request. This produces invalid hrefs like `"/organizations/{org_id}/users/abc123"` instead of `"/organizations/actual-org-id/users/abc123"`. Visible in generated example output at `examples/user-management/out/internal/api/handler_org_users.go` lines 113, 131, 156. Spec reference (Response Format): `href -- base_path + collection path + id` with example showing a fully resolved URL `/api/hyperfleet/v1/clusters/abc123`.

- [process-revision-complete] **List response `items` are not presented entities.** In `internal/generator/restapi/generator.go`, `generateListMethod` (around line 792) returns `listResult.Items` directly in the envelope without running each item through `presentEntity()`: `"items": listResult.Items`. Spec reference (Response Format, List response fields): `items -- array of presented entities`. Each item should include `id`, `kind`, and `href` metadata per the presenter function, matching the presentation applied to single-resource responses.

- [process-revision-complete] **OpenAPI spec does not document `orderBy` and `fields` query parameters.** In `internal/generator/restapi/generator.go`, `generateOpenAPI` (lines 1299-1311) adds only `page` and `size` query parameters to list operations. `orderBy` and `fields` are omitted despite being parsed and validated in the generated handler code and listed in the spec (Response Format section: "List query parameters").

- [process-revision-complete] **OpenAPI list response schema does not reflect envelope format.** In `internal/generator/restapi/generator.go`, `generateOpenAPI` (line 1329) describes the list response as `type: array`, but when `response_format: envelope` is enabled, the actual response is an object with `kind`, `page`, `size`, `total`, `items` fields. The OpenAPI spec does not reflect the envelope structure.

## Round 2

- [x] [process-revision-complete] **`fields` query parameter does not ensure `id` is always included.** In `internal/generator/restapi/generator.go`, `generateListMethod` (lines 746–754) parses the `fields` query parameter into a slice from user input and passes it directly to `ListOptions` without ensuring `id` is present. The code comment at line 745 states "id is always included even if not listed", matching the spec requirement, but no code enforces it. Spec reference (Response Format, List query parameters): "fields -- sparse fieldset selection, comma-separated field names (id is always included even if not listed)". The generated handler in `examples/user-management/out/internal/api/handler_all_users.go` lines 63–70 confirms: `fields` is built from user input with no `id` injection.

- [x] [process-revision-complete] **Update handler does not set entity ID from the path parameter before passing to storage.** In `internal/generator/restapi/generator.go`, `generateUpdateMethod` (lines 626–665) decodes the request body into the entity struct and passes it to `h.store.Replace()` without setting `entity.ID = id` from the path parameter. Create (line 562) sets `entity.ID = uuid.New().String()`. Upsert (lines 831–834) sets `entity.ID = uuid.New().String()` when empty. Update does neither. The entity passed to storage has an `ID` field that is whatever the client sends in the request body (possibly empty or mismatched). Visible in generated example at `examples/user-management/out/internal/api/handler_org_users.go` lines 134–157: `user.ID` is never set from the path `id` before `h.store.Replace()`. The `presentEntity()` response uses the path `id` so the response is correct, but the stored entity has an inconsistent ID field.

- [x] [process-revision-complete] **OpenAPI single-resource response schemas do not reflect envelope format.** In `internal/generator/restapi/generator.go`, `generateOpenAPI` (lines 1382–1467), create, read, update, and upsert operations reference the bare entity schema (`$ref: #/components/schemas/Entity`) for their response content, but when `response_format: envelope` is enabled, the actual response includes `id`, `kind`, and `href` fields added by `presentEntity()`. The list response schema was correctly updated to reflect the envelope format (object with `kind`, `page`, `size`, `total`, `items` properties at lines 1352–1367), creating an inconsistency: list operations accurately describe the envelope response while single-resource operations do not. Visible in `examples/user-management/out/internal/api/openapi.json`: the `201` response for `POST /organizations` references `#/components/schemas/Organization` (lines 20–37) but the actual response includes `id`, `kind`, `href` metadata not present in that schema.

## Round 3

- [process-revision-complete] **`fields` query parameter values are not validated against entity fields.** In `internal/generator/restapi/generator.go`, `generateListMethod` (lines 749–764), the `fields` parameter parses user input into a string slice and passes it to `ListOptions.Fields` without validating that each field name exists on the entity. The `orderBy` parameter validates field names against the `validFields` map (line 739: `if !validFields[fieldName]`), but `fields` bypasses this check entirely. AC 4 requires "List query parameters parsed and validated: page, size, orderBy, fields." The task description states under "Pagination mechanics": "SQL injection prevented by field name validation + hardcoded direction strings" — but this protection only applies to `orderBy`, not `fields`. Unvalidated field names in `ListOptions.Fields` reach the storage layer, where they may be used in SQL SELECT clauses without further sanitization. Visible in generated handler at `examples/user-management/out/internal/api/handler_all_users.go` lines 63–78: field names from user input are collected into a slice with no `validFields` check.

- [process-revision-complete] **OpenAPI list response `items` array schema references the bare entity, not the presented entity.** In `internal/generator/restapi/generator.go`, `generateOpenAPI` (line 1370), the envelope list response's `items` array uses `&openAPISchema{Ref: ref}` (the bare entity schema `#/components/schemas/Entity`), but the actual generated handler code runs each list item through `presentEntity()`, adding `id`, `kind`, and `href` metadata. The spec (Response Format, List response fields) states "items -- array of presented entities." Single-resource response schemas were correctly updated in round 2 to use `envelopeResourceSchema(ref)` (allOf composition with envelope metadata), but the same fix was not applied to list item elements within the envelope. Visible in generated `openapi.json`: list `items` references `#/components/schemas/User` (lines 170–174) while the handler at `handler_all_users.go` lines 89–94 produces items with `id`, `kind`, `href` metadata via `presentEntity()`.

## Round 4

- [x] [process-revision-complete] **`fields` sparse fieldset does not produce sparse responses — non-selected fields appear with zero values.** In `internal/generator/restapi/generator.go`, `generateListMethod` (lines 747–771) parses and validates the `fields` query parameter and passes it to the storage layer via `ListOptions.Fields`. However, the response construction (lines 803–825) runs each list item through `presentEntity()` which marshals the entire entity struct via `json.Marshal(entity)` and unmarshals into a `map[string]any`. Entity struct fields use `json:"field_name"` tags without `omitempty` (except `ID`), so non-selected fields appear in the response with their Go zero values (empty strings, 0, false, null). The spec (Response Format, List query parameters) describes `fields` as "sparse fieldset selection" — the standard meaning of sparse fieldset (e.g. JSON:API) is that the response contains only the requested fields. The handler parses and validates the selection but does not filter the response map to remove non-selected fields. Visible in generated `handler_all_users.go`: `?fields=email` produces items containing `{"id":"...","kind":"User","href":"...","email":"user@test.com","role":"","org_id":""}` — the `role` and `org_id` fields are present with empty-string zero values instead of being absent from the response. The generated entity struct at `router.go` lines 53–59 confirms: `Role string \`json:"role"\`` and `OrgID string \`json:"org_id"\`` lack `omitempty`, and `presentEntity` (lines 63–71) does not filter by requested fields.
