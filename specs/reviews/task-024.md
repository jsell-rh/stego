# Review: Task 024 — Response Envelope Format with Pagination

## Round 1

- [process-revision-complete] **href contains unresolved path parameters for scoped collections.** In `internal/generator/restapi/generator.go`, the `hrefBase` for scoped collections retains route template placeholders (e.g. `{org_id}`) as string literals. `generateCreateMethod`, `generateReadMethod`, `generateUpdateMethod`, and `generateUpsertMethod` emit code such as `presentEntity(user, "User", user.ID, "/organizations/{org_id}/users"+"/"+user.ID)`. The literal `{org_id}` is never resolved to the actual path parameter value from the request. This produces invalid hrefs like `"/organizations/{org_id}/users/abc123"` instead of `"/organizations/actual-org-id/users/abc123"`. Visible in generated example output at `examples/user-management/out/internal/api/handler_org_users.go` lines 113, 131, 156. Spec reference (Response Format): `href -- base_path + collection path + id` with example showing a fully resolved URL `/api/hyperfleet/v1/clusters/abc123`.

- [process-revision-complete] **List response `items` are not presented entities.** In `internal/generator/restapi/generator.go`, `generateListMethod` (around line 792) returns `listResult.Items` directly in the envelope without running each item through `presentEntity()`: `"items": listResult.Items`. Spec reference (Response Format, List response fields): `items -- array of presented entities`. Each item should include `id`, `kind`, and `href` metadata per the presenter function, matching the presentation applied to single-resource responses.

- [process-revision-complete] **OpenAPI spec does not document `orderBy` and `fields` query parameters.** In `internal/generator/restapi/generator.go`, `generateOpenAPI` (lines 1299-1311) adds only `page` and `size` query parameters to list operations. `orderBy` and `fields` are omitted despite being parsed and validated in the generated handler code and listed in the spec (Response Format section: "List query parameters").

- [process-revision-complete] **OpenAPI list response schema does not reflect envelope format.** In `internal/generator/restapi/generator.go`, `generateOpenAPI` (line 1329) describes the list response as `type: array`, but when `response_format: envelope` is enabled, the actual response is an object with `kind`, `page`, `size`, `total`, `items` fields. The OpenAPI spec does not reflect the envelope structure.

## Round 2

- [ ] **`fields` query parameter does not ensure `id` is always included.** In `internal/generator/restapi/generator.go`, `generateListMethod` (lines 746–754) parses the `fields` query parameter into a slice from user input and passes it directly to `ListOptions` without ensuring `id` is present. The code comment at line 745 states "id is always included even if not listed", matching the spec requirement, but no code enforces it. Spec reference (Response Format, List query parameters): "fields -- sparse fieldset selection, comma-separated field names (id is always included even if not listed)". The generated handler in `examples/user-management/out/internal/api/handler_all_users.go` lines 63–70 confirms: `fields` is built from user input with no `id` injection.

- [ ] **Update handler does not set entity ID from the path parameter before passing to storage.** In `internal/generator/restapi/generator.go`, `generateUpdateMethod` (lines 626–665) decodes the request body into the entity struct and passes it to `h.store.Replace()` without setting `entity.ID = id` from the path parameter. Create (line 562) sets `entity.ID = uuid.New().String()`. Upsert (lines 831–834) sets `entity.ID = uuid.New().String()` when empty. Update does neither. The entity passed to storage has an `ID` field that is whatever the client sends in the request body (possibly empty or mismatched). Visible in generated example at `examples/user-management/out/internal/api/handler_org_users.go` lines 134–157: `user.ID` is never set from the path `id` before `h.store.Replace()`. The `presentEntity()` response uses the path `id` so the response is correct, but the stored entity has an inconsistent ID field.

- [ ] **OpenAPI single-resource response schemas do not reflect envelope format.** In `internal/generator/restapi/generator.go`, `generateOpenAPI` (lines 1382–1467), create, read, update, and upsert operations reference the bare entity schema (`$ref: #/components/schemas/Entity`) for their response content, but when `response_format: envelope` is enabled, the actual response includes `id`, `kind`, and `href` fields added by `presentEntity()`. The list response schema was correctly updated to reflect the envelope format (object with `kind`, `page`, `size`, `total`, `items` properties at lines 1352–1367), creating an inconsistency: list operations accurately describe the envelope response while single-resource operations do not. Visible in `examples/user-management/out/internal/api/openapi.json`: the `201` response for `POST /organizations` references `#/components/schemas/Organization` (lines 20–37) but the actual response includes `id`, `kind`, `href` metadata not present in that schema.
